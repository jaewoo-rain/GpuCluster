## 1. 연구 질문 & 가설

### 핵심 연구 질문
> **"GPU 하나를 fractional로 나눠 두 추론 워크로드를 동시에 돌리는 것이, 순차로 독점 실행하는 것보다 언제·얼마나 이득인가?"**

### 이득의 조작적 정의 (먼저 고정 — 측정 기준이 흔들리지 않게)
- **makespan 이득**: 두 작업이 모두 끝나는 벽시계 시간.
  - 순차(sequential): `makespan_seq = wall(A solo) + wall(B solo)`
  - 동시(concurrent): `makespan_conc = max(t_end_A, t_end_B) − min(t_start_A, t_start_B)`
  - **speedup = makespan_seq / makespan_conc** (≥ 1.2 면 이득 판정, 후술)
- **throughput 이득**:
  - 순차: `iters_A/wall_A + iters_B/wall_B` (시간대가 겹치지 않음)
  - 동시: `(iters_A + iters_B) / makespan_conc`
- **공유 효율 지표**: `E_share = (wall_solo_A + wall_solo_B) / makespan_conc` — >1 이득, <1 손해.
- **경합 패널티**: `contention_penalty = 1 − (iters_conc_A + iters_conc_B)/(iters_solo_A + iters_solo_B)`.

> makespan speedup과 throughput이 서로 다른 이야기를 할 수 있다. SM 경합이 심하면 throughput은 낮아도 makespan speedup은 1.x가 나올 수 있으므로 **둘 다 보고**한다.

### 가설
- **H1 (컴퓨트 바운드 풀로드 두 개)**: 동시 공유해도 SM 경합으로 각 워크로드 iters/sec가 단독 대비 ~0.4~0.5로 떨어지고, makespan speedup은 1.0 근처(이득 없음 또는 음수). → **"공유가 손해/무의미한 지점"**.
- **H2 (메모리 바운드/idle-heavy + 컴퓨트 바운드)**: idle-heavy 쪽이 SM을 간헐적으로만 쓰므로 컴퓨트 쪽이 빈틈을 메워 speedup > 1.2. → **"공유가 이득인 지점"**.
- **H3 (oversubscription, 합>1.0)**: admission 거부 또는 force 우회 시 hook/driver OOM·throughput 붕괴. → **"공유가 명백히 손해인 지점 + admission의 보호 가치"**.
- **H4 (3-layer 직교성)**: 합=1.0(admission 통과)이라도 CUDA context 오버헤드(~700MiB/프로세스)로 물리 OOM 가능 → admission 통과 ≠ 물리 안전.

## 2. 메트릭 정의 (PASS/이득 판정)

| 메트릭 | 정의 | 출처 |
|---|---|---|
| `wall_s` | spawn POST 직전 ~ `exited` 폴링 감지 | 스크립트 타임스탬프(`date +%s%3N`) |
| `makespan_raw_s` | spawn 포함 동시 완료시간 | 스크립트 타임스탬프 |
| `makespan_compute_s` | `[compute-test] entering compute loop` ~ `compute done` | `docker logs --timestamps` 파싱 (headline) |
| `iters`, `elapsed` | 워크로드가 출력하는 `iters=N elapsed=T.TTs` | 컨테이너 로그 grep |
| `throughput_iters_sec` | `iters / elapsed` | 파생 |
| `peak_mem_mib` | PID별 메모리 피크 | `nvidia-smi --query-compute-apps` |
| `gpu_util_avg_pct` | GPU 전체 util 1초 샘플 평균 | `nvidia-smi --query-gpu=utilization.gpu,memory.used` (신규 캡처) |
| `speedup` | `makespan_seq / makespan_conc` | post-processing |
| `E_share` | `(wall_solo_A+wall_solo_B)/makespan_conc` | post-processing |
| `contention_penalty` | `1 − Σiters_conc/Σiters_solo` | post-processing |

### 이득 판정 기준
- **PASS(이득) = speedup ≥ 1.2** (보수적). 이론 최대 2.0과 실측 사이에 spawn 오버헤드(~2~5s)+SM 경합 감속 갭이 있어 보수적으로 설정.
- **FAIL(speedup < 1.0)은 "나쁜 결과"가 아니라 H1을 입증하는 반례 데이터**로 채택 — SM 경합이 공유 이득을 상쇄.
- 두 makespan 정의를 모두 보고: `makespan_compute_s`를 headline speedup으로, `makespan_raw_s`를 주석으로.

## 3. 시나리오 매트릭스 (solo / 순차 / 동시 × ratio 조합)

기본 워크로드 페어로 한 번 실행에 모든 셀을 자동 순차 진행. `SCENARIOS=1,2,5` 식으로 선택 실행, `REPEAT`(기본 1)로 반복.

| # | 이름 | 설명 | ratio A | ratio B | admission |
|---|---|---|---|---|---|
| 1 | `solo_A` | A 단독 | RATIO_A | – | 통과 |
| 2 | `solo_B` | B 단독 | – | RATIO_B | 통과 |
| 3 | `seq_AB` | A 완료(`exited`) 후 B spawn (순차 참조) | RATIO_A | RATIO_B | 통과 |
| 4 | `conc_50_50` | 동시, 0.5/0.5 | 0.5 | 0.5 | **합=1.0 통과** |
| 5 | `conc_40_60` | 동시, 0.4/0.6 | 0.4 | 0.6 | **합=1.0 통과** |
| 6 | `conc_70_30` | 동시, 0.7/0.3 | 0.7 | 0.3 | **합=1.0 경계 통과(`_TOL=1e-9`)** |
| 7 | `conc_oversub` | 동시, 0.7/0.6 (합=1.3) | 0.7 | 0.6 | **거부 → force=true 필수** |

### admission 관계 (scheduler-architect 합의)
- **합=1.0 조합 3개(0.5/0.5, 0.4/0.6, 0.7/0.3)는 전부 force 없이 통과** — admission은 아무것도 안 막는데 성능은 조합마다 다름 → **admission이 성능과 직교함**을 그대로 보여주는 증거.
- 0.7/0.3은 FP 경계(`_TOL=1e-9`)로 안전 통과 — "admission 경계를 정확히 타는 케이스" 한 줄 보고.
- **oversub(0.7/0.6=1.3)은 admission이 실제로 일하는 유일한 셀.** 두 가지를 보여줌:
  - (a) admission ON → 두 번째 spawn 409 → "동시 공유 불가, 순차밖에 없음" 정책 결과.
  - (b) force=true 우회 → hook 합산 quota가 물리 초과 → OOM/throughput 붕괴 = "손해" 정량 증거.
- **운영 모드 분리(교란 방지)**: 본 성능 실험은 force 일괄(합≤1.0 조합은 어차피 통과). oversub 셀에서만 admission ON/거부를 *정책 결과*로 별도 기록. 같은 그래프에 admission 효과와 성능 효과를 섞지 않는다.

### OOM 시연 셀(H4 전용, 별도)
- 합=1.0(예 0.5/0.5)인데 **합 ALLOC을 ~11000 MiB로** 올려 context 오버헤드까지 합치면 물리 초과 → "admission 통과인데 물리 OOM" = 3-layer 직교성 결정적 figure.

## 4. 워크로드 설계 (A/B)

### 4.1 컴퓨트 바운드 vs 메모리 바운드 두 극단 (기존 파일 재사용)
`test_compute.py`(matmul+relu 루프)를 **파라미터만 바꿔** 두 극단 생성 — 새 파일 불필요(1단계).

| 프로파일 | COMPUTE_SIZE | ALLOC_MIB | HOLD_SEC | 특성 |
|---|---|---|---|---|
| 컴퓨트 풀로드 | 2048 | 512 | 20 | SM 상시 포화, 메모리 낮음 |
| 메모리/idle-heavy | 256 | 3000 | 20 | SM 간헐, 메모리 크게 점유 |

페어 조합:
- **PAIR-CC**: 컴퓨트 × 컴퓨트 → H1 검증(손해 예상).
- **PAIR-MC**: 메모리heavy × 컴퓨트 → H2 검증(이득 예상).

### 4.2 mem ratio vs compute ratio 분리 (이미 구현됨)
- `FGPU_RATIO`(메모리 quota, Stage 1-6)와 `FGPU_COMPUTE_RATIO`(duty-cycle throttle, Stage 12)는 **이미 독립 env**. admission은 메모리 ratio만 합산.
- **메인 실험**: mem=compute 동일 ratio (한 노브 단순 모델).
- **진단 실험 2개**(이득이 어느 자원 축에서 오는지 분해):
  - 메모리 바운드 진단: mem ratio만 0.4/0.6, compute throttle off → 이득이 메모리 분배에서 옴.
  - 컴퓨트 바운드 진단: mem 넉넉(0.8/0.8, force), compute ratio만 0.3/0.7 → 이득이 duty-cycle에서 옴.

### 4.3 throttle ON/OFF (둘 다 측정, 역할 분리)
- **Throttle OFF** = SM 경합 특성화의 주 측정. 컴퓨트×컴퓨트 동시 시 각 iters/sec가 단독 대비 ~0.4~0.5로 떨어지는지 직접 확인.
- **Throttle ON (ratio 0.5/0.5)** = 공정성 메커니즘 검증. 각 iters/sec가 단독의 ~0.5배로 수렴하는지(이득 측정이 아니라 "의도대로 동작" 검증).
- 주의: nanosleep ~50μs 정밀도 + window 100ms 한계 → idle-heavy + throttle ON 조합은 ±0.05 ratio 오차 가능. 해석 시 오차 범위 명시. throttle OFF가 경합 순수 측정에 더 신뢰도 높음.

### 4.4 LLM 추론 워크로드 — 권장안 (docker-runtime-engineer 합의)
모호했던 "llm추론으로 검사"의 **워크로드 해석**(섹션 6에서 분석 해석도 다룸).

- **(a) 실제 소형 LLM**: HF transformers + 소형 모델. 12GB에 두 모델 동시 적재 예산:
  - 가용 ≈ 12288 − reserve(~500) − context(~700×2) ≈ ~10300 MiB.
  - 권장 모델 **Qwen2-0.5B**(bf16 ~1GB) 또는 TinyLlama-1.1B(~2.2GB). GPT-2/DistilGPT-2는 너무 작아 실험 드라마 없음.
  - 함정: `PYTORCH_NO_CUDA_MEMORY_CACHING=1`로 caching off → KV cache 매 스텝 재할당 → generate() 10~30x 저하 + "실배포 조건과 다름". 논문에 **명시적 한계**로 기술 필요.
  - 모델 가중치는 **빌드타임 baking 금지** — 호스트 `~/.cache/huggingface` 를 `-v ...:/root/.cache/huggingface:ro` 마운트(Stage 10 workspace bind-mount 패턴 재사용, `HF_HOME` 주입). 첫 실행만 다운로드.
  - 이미지 변경: `pip install transformers accelerate` 한 레이어(+2~3분 빌드), 첫 풀 1~5분.
- **(b) transformer-like 프록시**: `test_compute.py`를 attention 패턴(`Q@K.T/√d → softmax → @V`, seq-len 파라미터화로 KV cache 흉내)으로 강화. 완전 재현성, 외부 의존성·caching 함정 없음. 단 "실제 LLM" 설득력 약함.

- **권장 (단계적)**:
  1. **1단계(즉시)**: 프록시(b)는 이미 있는 `test_compute.py` 파라미터로 즉시 가능 → 시나리오 매트릭스 전체를 **프록시로 먼저 완주**(재현성·빠른 반복).
  2. **2단계(옵션 마일스톤)**: 실제 LLM(a, Qwen2-0.5B) `test_llm_infer.py` 추가 → 동일 매트릭스 일부(대표 셀) 재측정 + "프록시 vs 실제 LLM hook 카운터 패턴 비교" 테이블로 두 워크로드 사용 정당화.
  - 이유: 캡스톤+단일 12GB+재현성 우선 → 프록시가 메인, 실제 LLM은 설득력 보강 옵션.

### 4.5 안전 ALLOC 예산 (12GB)
- 측정 셀: **A+B 합 ALLOC ≤ 9000 MiB** (ratio 비례 배분: 0.5/0.5→4500/4500, 0.4/0.6→3600/5400, 0.7/0.3→6300/2700).
- OOM 시연 셀: 합 ~11000 MiB(별도, 의도적).
- oversub+force 셀: 각 quota 가까이(예 8200/7000) → hook/driver DENY 유도.

## 5. 측정·자동화 설계

### 5.1 신규 스크립트 위치/이름 (eval 패턴 재사용)
- `scripts/eval/run_sharing_benefit.sh` — 오케스트레이터(bash). run_correlation.sh 골격 재사용(preflight, nvidia-smi 백그라운드 캡처, POST /sessions + parse_field, docker top PID, docker logs --timestamps, DELETE, post-process 호출).
- `scripts/eval/_sharing_benefit.py` — post-processing(stdlib only). `runs.csv`+`gpu_util.csv`만 다룸. `_correlate.py`는 **import하지 않고**, 각 동시 시나리오 후 `_correlate.py`를 추가 호출해 `correlation_<scenario>.csv`(launch↔memory trace) 재사용.

### 5.2 캡처 (기존 + 신규)
- 기존: `nvidia-smi --query-compute-apps=timestamp,pid,process_name,used_memory -l 1` (PID별 메모리), `docker top` PID 조인, `docker logs --timestamps`.
- **신규**: `nvidia-smi --query-gpu=utilization.gpu,memory.used,timestamp --format=csv,noheader -l 1` (GPU 전체 util) 병행 캡처.
- 타이밍: bash가 `date +%s%3N`(ms)로 `t_spawn_ms`(POST 직전), `t_exited_ms`(폴링 exited 감지) 직접 기록 — DB `created_at`은 GPU 작업 시작과 수초 차이 날 수 있어 신뢰 안 함.

### 5.3 CSV 스키마
`runs.csv` (한 행 = 한 런):
```
scenario, job, ratio_a, ratio_b, throttle, pair, t_spawn_ms, t_exited_ms,
wall_s, makespan_compute_s, iters, throughput_iters_sec, peak_mem_mib,
gpu_util_avg_pct, makespan_s, speedup_vs_seq, e_share, contention_penalty, notes
```
`gpu_util.csv`:
```
timestamp, pid, process_name, used_memory_mib, gpu_util_pct, scenario
```
`summary.txt`: solo/seq 기준선 → 각 conc 셀 speedup/E_share/util/VERDICT → "이득 조건 speedup≥1.2" → PROVED/NOT PROVED 섹션 → 최종 VERDICT.

### 5.4 측정 함정 처리
- **spawn 오버헤드(~2~5s)**: `makespan_raw_s`(포함)와 `makespan_compute_s`(로그 타임스탬프 기준, headline) 둘 다 산출. preflight에서 `docker pull` 명시로 이미지 캐시 보장.
- **caching off**: Dockerfile ENV에 이미 `PYTORCH_NO_CUDA_MEMORY_CACHING=1`. summary에 "없으면 hook 우회" 명기.
- **nvidia-smi 1초 해상도**: `gpu_util_avg_pct`에 "N samples, resolution=1s" 명기, ±1s 오차.
- **반복 통계**: 기본 REPEAT=1, 논문용은 REPEAT=3 평균/stddev(GPU 시간 길어짐을 사용자에 고지).

## 6. LLM 분석/리포트 단계 (결과를 Claude로 해석)

> 모호했던 "llm추론으로 검사"의 **두 번째 해석**: 산출 CSV/로그를 LLM(claude-*)에게 분석시켜 판정·해석 리포트를 자동 생성.

- **자동 리포트 단계** `scripts/eval/_llm_report.py`(옵션, 외부 API 의존은 OFF가 기본):
  - 입력: `runs.csv`, `summary.txt`, `correlation_*.csv` 요약 통계.
  - 출력: `report.md` — 연구질문에 대한 자연어 결론("어떤 조합에서 speedup>1.2였고 왜", H1~H4 입증 여부, 한계 반영).
  - 동작 모드:
    - `--mode=local`(기본, 외부 의존 없음): 규칙 기반 템플릿으로 speedup/E_share/penalty를 문장화. API 키 불필요·재현성 보장.
    - `--mode=claude`(옵션): Anthropic Claude API(프로젝트가 Claude 생태계)로 CSV 요약을 보내 해석/논문 문단 초안 생성. `ANTHROPIC_API_KEY` 환경변수, 네트워크 필요. 기본 비활성 — 외부 API 의존은 옵션.
  - 안전장치: claude 모드는 raw 대용량 CSV가 아니라 **사전 집계된 요약 수치**만 전송(토큰/프라이버시).

이로써 "LLM 추론" 두 해석을 모두 반영: **(i) 워크로드로서 LLM 추론**(섹션 4.4) + **(ii) 결과를 LLM이 분석/요약**(이 섹션).

## 8. 리스크 · 의도된 한계의 영향

| 한계 | 실험 결과 해석에 주는 영향 |
|---|---|
| **SM 격리 없음(cooperative wall-clock throttle만)** | 컴퓨트×컴퓨트 동시 = SM 자유경쟁 → speedup 거의 없음(H1). "이득"은 idle-heavy 빈틈을 메울 때만(H2). throttle은 work-conserving 아님 — 한쪽 sleep 중 다른쪽 안 쓰면 GPU idle. |
| **caching allocator(`PYTORCH_NO_CUDA_MEMORY_CACHING=1` 필요)** | 끄면 hook 우회·정상, 켜면 LLM generate() 10~30x 저하·실배포 조건과 괴리. 논문 명시 한계. |
| **단일 RTX 12GB, MIG 없음** | 두 모델 동시 적재 예산 빠듯, context ~1.4GB 오버헤드. multi-GPU aggregation 범위 밖. |
| **CUDA context 오버헤드 미회계** | admission 합=1.0이라도 물리 OOM 가능(H4). admission 통과 ≠ 물리 안전. |
| **launch count ≠ device time** | iters/sec는 GPU 실사용의 proxy. 무거운 커널 1회 vs 가벼운 1000회 구분 못함. |
| **nanosleep ~50μs, window 100ms** | idle-heavy + throttle ON 조합 ±0.05 ratio 오차. throttle OFF가 더 신뢰. |
| **cooperative threat model** | 정적 링크 바이너리는 hook 우회 → quota 미적용. NOT PROVED에 명기. |
| **1회 실행(REPEAT=1 기본)** | 논문 통계 신뢰도엔 REPEAT=3 필요(GPU 시간 비용). |

**"공유가 이득" 성립 조건**: 적어도 한쪽이 SM을 간헐적으로만 쓰는(메모리 바운드/idle-heavy) 워크로드일 때. **불성립 조건**: 둘 다 SM 풀로드 컴퓨트 바운드 — 이땐 순차 독점이 동등하거나 낫다. 이 정직한 경계 자체가 핵심 발견.

## 10. 테스트/검증 계획 (실행 단계)

- **eval 검증(eval-benchmark-engineer)**:
  - `runs.csv` 행 수 = 시나리오 수 × REPEAT × (solo/conc job 분해). 빈 `iters` 없음.
  - solo_A/solo_B 먼저 실행되어 speedup의 sequential 기준값 존재.
  - 각 conc 셀 `correlation_<scenario>.csv` 생성, PID 조인 비어있지 않음.
  - `summary.txt`가 `VERDICT:` 라인으로 끝남.
- **정확성 회귀(test-qa-engineer)**:
  - `_sharing_benefit.py` 파싱 함수 단위 테스트(stdlib pytest): iters/elapsed grep, makespan_conc 계산(max-end − min-start), speedup 산식.
  - admission 경계: 0.7/0.3 합=1.0 통과, 0.7/0.6 합=1.3 거부(기존 `test_admission.py` 재사용/확장).
  - `run_all_tests.sh`에 새 eval 스크립트의 dry-run(또는 짧은 HOLD_SEC smoke) 등록 검토.
- **PASS 조건(논문)**:
  - H2 셀(PAIR-MC)에서 speedup ≥ 1.2 적어도 1개.
  - H1 셀(PAIR-CC)에서 speedup < 1.2(또는 contention_penalty 유의) — 손해 경계 입증.
  - H4 OOM 셀에서 합=1.0인데 물리 OOM 관측.
- **GPU/docker 실행 안전**: 실제 실행 전 사용자 확인(이미지 rebuild, GPU 점유, 첫 LLM 풀 다운로드 시간).
