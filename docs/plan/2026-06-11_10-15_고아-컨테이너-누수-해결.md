## 1. 문제 요약

`SessionManager._create_locked` 는 컨테이너를 **먼저 spawn** 하고(`docker.create_container` →
`containers.run(detach=True, remove=False)`), **그 한참 뒤에** DB record 를 `store.insert` 로 기록한다.
spawn 과 insert 사이 구간(host_port 폴링, jupyter_url 조립, Session pydantic 구성, insert 자체)을
감싸는 try/except·롤백이 전혀 없다.

이 구간에서 예외가 발생하면:
- 이미 떠 있는 컨테이너(`remove=False`)는 계속 running → 고아(orphan).
- DB row 가 없으므로 `store.list_all()` 이 절대 반환하지 않음.
- reconcile 은 **DB record → docker** 단방향이라(`list_all` 이 record 만 순회) orphan 을 영원히 발견 못 함.
- `admission.sum_used_ratio` 는 DB 세션만 순회 → orphan 의 ratio 를 영구히 카운트 안 함 →
  **GPU quota 회계가 실제 점유량과 어긋남**(admission 이 oversubscription 을 허용).
- 수동 `docker rm -f` 전까지 GPU/메모리/포트 누수.

**현실적으로 예외가 나는 경로**
1. `store.insert` — SQLite `OperationalError: database is locked`(WAL 미사용, timeout=5.0s, 동시
   `get/list_all` 의 `update_status` 쓰기와 경합), 디스크 풀, `IntegrityError`(id 충돌).
2. jupyter `host_port` 폴링/`asyncio.sleep`/`to_thread(insert)` 도중 클라이언트 disconnect →
   `asyncio.CancelledError`. (일반 `except Exception` 으로 못 잡음 — `BaseException` 계열.)
3. `Session(...)` pydantic 구성 실패.
4. jupyter workspace `mkdir` 이후 spawn 실패 시 빈 워크스페이스 디렉토리도 누수(#10).

이는 의도된 한계(MIG/SM 격리/cooperative threat model/allocator caching/단일호스트)
어디에도 해당하지 않는 **실제 에러 전파·자원 누수 버그**.

## 3. 수정 전략 (단계별)

### Step A — 최소 수정안: 보상정리(try/except + best-effort rollback) [필수]

`_create_locked` 에서 `create_container` 가 `c` 를 반환한 **직후**부터 `store.insert` 성공까지를
`try` 로 감싸고, 예외 발생 시 best-effort 로 컨테이너를 제거한 뒤 **re-raise** 한다.

설계 포인트:
- **`except BaseException` 으로 잡는다** — `asyncio.CancelledError` 가 Python 3.8+ 에서
  `BaseException` 의 직속 서브클래스이므로 `except Exception` 으로는 못 잡는다(감사 #1의 disconnect 경로).
  잡은 뒤 정리하고 반드시 `raise` 로 원래 예외를 그대로 전파(취소 의미 보존).
- **보상정리 자체의 실패는 swallow** — `remove_container` 가 `docker.errors.NotFound`(이미 사라짐) /
  daemon 다운 등으로 또 던질 수 있다. 안쪽 `try/except`(또는 `NotFound` 외 광범위 catch + 로그)로
  감싸 **원래 예외를 덮지 않게** 한다. 정리 실패는 stderr 경고 로그만 남기고 원래 예외 re-raise.
- **CancelledError 정리 중 await**: 취소된 코루틴 안에서 `await asyncio.to_thread(remove_container, ...)`
  를 호출하면 그 await 도 즉시 취소될 위험이 있다. 정리 await 를 `asyncio.shield(...)` 로 감싸거나,
  취소 컨텍스트에서도 정리가 완주하도록 보장한다. (실행 시 backend-api-engineer 가 shield vs
  별도 to_thread 직접 호출 중 안전한 방식 결정 — 권장: shield + 짧은 타임아웃.)
- **jupyter workspace 정리(#10 연결)**: spawn 전에 만든 `workspace_dir` 도 보상정리에서 같이 지운다.
  단, **막 우리가 만든 빈 디렉토리일 때만** 삭제(사용자 노트북 보호). 안전책:
  `mkdir(exist_ok=True)` 가 새로 만들었는지 여부를 플래그로 추적해, 새로 만든 경우에만
  `shutil.rmtree(ws, ignore_errors=True)`. (`exist_ok=True` 면 이미 있던 dir 을 지워선 안 됨.)
- **반환 직전까지만 try 범위**: `return rec` 는 try 밖이거나 try 끝에 두되, insert 성공 이후에는
  롤백하지 않는다(이미 정상 record 존재).

의사 구조(개념 — 실제 구현은 위임):
```
# (jupyter면) workspace mkdir, created_ws = (디렉토리를 새로 만들었으면 True)
c = await to_thread(self.docker.create_container, ...)   # 성공 시점부터 누수 위험
try:
    ... host_port 폴링 / jupyter_url 조립 / Session(...) ...
    await to_thread(self.store.insert, rec)
    return rec
except BaseException:
    await self._rollback_spawn(c.id, workspace_dir if created_ws else None)
    raise
```
`_rollback_spawn(container_id, ws_dir)` 헬퍼:
```
try:
    await asyncio.shield(to_thread(self.docker.remove_container, container_id, True))
except docker.errors.NotFound:
    pass
except Exception as e:
    log "[fgpu] rollback: remove_container 실패(무시): {e}"
if ws_dir:
    try: await to_thread(lambda: shutil.rmtree(ws_dir, ignore_errors=True))
    except Exception: pass
```

### Step B — 보강안: docker → DB orphan 스윕 [권장, race 안전망]

Step A 는 _이 프로세스가 아는_ `c.id` 만 정리한다. 그러나 다음은 여전히 샐 수 있다:
- 보상정리 await 자체가 프로세스 강제 종료(SIGKILL)로 중단된 경우.
- 과거(수정 전) 버전이 남긴 기존 orphan.
- shield 에도 불구하고 daemon 일시 장애로 정리가 누락된 경우.

따라서 docker daemon 의 `fgpu-` prefix 컨테이너를 enumerate 해 **DB record 가 없는 것**을
탐지/회수하는 **docker → DB 역방향 스윕**을 추가한다.

구현 포인트:
- `DockerManager.list_fgpu_containers()` 추가 — `client.containers.list(all=True, filters={"name":"fgpu-"})`
  로 컨테이너 id/name/status 목록 반환(label 기반 필터가 더 견고하나, 현 코드가 label 을 안 붙이므로
  우선 name prefix; 차후 spawn 시 `labels={"fgpu.managed":"1"}` 추가를 별도 개선으로 권장).
- `SessionManager.reconcile_orphans()` 추가 — daemon 의 `fgpu-` 컨테이너 중 `store.get` 에 없는 것을
  orphan 으로 간주. **회수 정책은 보수적으로**:
  - 기본: orphan 을 `remove_container(id, force=True)` 로 회수(누수 방지 우선).
  - 안전 가드: backend 가 막 spawn 중인(아직 insert 안 끝난) 컨테이너를 잘못 죽이지 않도록,
    `_create_lock` 보유 중이거나 "spawn 진행 중 id 집합"을 추적해 제외. 또는 생성 후 N초 유예
    (`State.StartedAt` 가 grace_period 보다 오래된 것만 회수).
- **호출 시점**:
  - (a) 앱 startup 시 1회(`main.py` lifespan) — 과거 orphan 청소.
  - (b) `list_all()` 경로에 옵트인으로 합치거나 별도 주기 태스크. **`list_all` 마다 무조건 스윕은
    비용/위험** 때문에 권장하지 않음 — startup 1회 + (선택) 저빈도 백그라운드 태스크.
- admission 정합성 회복: 스윕이 orphan 을 제거하면 다음 `sum_used_ratio` 가 자동 정상화.
  (orphan 을 DB 에 "흡수"하는 대안도 있으나, ratio/gpu_index 메타데이터가 컨테이너 env 에만
  있어 복원이 지저분 → **회수(제거)** 가 단순·안전.)

### Step C — 보조 정합성 (선택, 별도 발견과 묶을 수 있음)
- 감사 #7(host_port 폴링이 락 안에서 블로킹)과 함께 처리하면, 폴링을 락 밖으로 빼면서
  보상정리 try 범위도 자연히 재정리된다. **단 본 PLAN 의 1차 범위는 #1 누수 차단**이고
  #7 은 동시성 최적화라 분리 가능. 실행 시 충돌 영역(같은 함수)을 한 PR 로 묶을지 결정.

## 5. 리스크 · 트레이드오프

### Step A
| 항목 | 리스크/트레이드오프 |
|---|---|
| `except BaseException` | `KeyboardInterrupt`/`SystemExit` 도 잡힌다 → 반드시 정리 후 `raise` 로 재전파해 셧다운 의미 보존. |
| CancelledError 중 await | 취소 컨텍스트에서 `to_thread(remove)` await 가 또 취소될 수 있음 → `asyncio.shield` 필수. shield 가 영원히 안 끝나면 셧다운 지연 → 짧은 타임아웃 권장. |
| remove 실패 swallow | 정리 실패를 삼키면 원래 예외는 보존되나 **여전히 orphan 잔존** → Step B 스윕이 안전망. 로그는 남겨 관측 가능하게. |
| workspace 삭제 오삭제 | 사용자 노트북을 지울 위험 → `created_ws` 플래그로 "우리가 새로 만든 빈 dir" 만 삭제. `exist_ok` 가 기존 dir 을 만났으면 절대 삭제 금지. |
| 성능 | 정상 경로엔 try 진입 오버헤드뿐 — 무시 가능. |

### Step B
| 항목 | 리스크/트레이드오프 |
|---|---|
| spawn 중 컨테이너 오살(race) | 막 떠서 아직 insert 안 끝난 컨테이너를 orphan 으로 오인해 죽일 수 있음 → grace_period(StartedAt 기준) + "진행 중 id 집합" 제외 필수. |
| name prefix 신뢰성 | 운영자가 수동으로 `fgpu-` 이름 컨테이너를 만들면 오회수 → label 필터(`fgpu.managed`)가 더 안전. 1차는 prefix, 후속으로 label. |
| 호출 빈도 | `list_all` 마다 스윕은 daemon enumerate 비용 + race 노출 ↑ → **startup 1회 + 저빈도 백그라운드**만 권장. |
| daemon 다운 | 스윕 중 daemon 예외 → 광범위 catch + 로그, list_all 정상 경로를 깨지 않게. |

### 미적용 시(do-nothing) 리스크
- GPU quota 회계가 영구히 어긋나 admission oversubscription → 물리 OOM(Layer C)까지 도달 가능.

   **우리가 새로 만든** workspace dir 이 삭제되는지(#10) 검증.
7. `test_rollback_preserves_existing_workspace` — workspace dir 이 이미 존재(노트북 있음)했다면
   롤백이 그 dir 을 **삭제하지 않는지** 검증(데이터 보호 회귀).
8. `test_success_path_no_rollback` — 정상 생성 시 `remove_container` 가 **호출되지 않는지**(회귀 방지).
9. `test_admission_consistent_after_failed_create` — 실패한 create 이후 `admission_snapshot()` 에
   해당 ratio 가 합산되지 않는지(orphan 미발생 ⇒ 회계 정합) 검증.

**Step B 단위 테스트**
10. `test_orphan_sweep_removes_unknown_fgpu_container` — daemon 에 `fgpu-XXX` 가 있고 DB 에 없으면
    스윕이 `remove_container` 호출하는지.
11. `test_orphan_sweep_keeps_known_container` — DB record 가 있는 컨테이너는 스윕이 건드리지 않는지.
12. `test_orphan_sweep_respects_grace_period` — `StartedAt` 가 grace 이내인(막 뜬) 컨테이너는
    회수 대상에서 제외되는지(spawn race 보호).
13. `test_orphan_sweep_skips_in_flight_ids` — 진행 중 id 집합에 포함된 컨테이너 제외 검증.

**회귀 방지 / 통합**
14. 기존 `backend/tests/test_session_store.py`, `test_admission.py` 가 그대로 통과(no regression).
15. `pytest backend/` 전체 green.
16. (GPU 호스트, 선택) `scripts/run_all_tests.sh` 의 backend pytest 스텝 PASS — 실 docker 경로에서
    정상 create/delete 가 깨지지 않는지.
17. (수동 재현, 선택) `data/` 를 read-only 로 만들어 insert 강제 실패 → `docker ps | grep fgpu-` 로
    orphan 이 남지 않는지 육안 확인.
