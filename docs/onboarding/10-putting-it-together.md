# 10장. 전체를 하나로 — 요청 하나가 시스템을 관통하는 여정

> 여기까지 오셨다면 부품 하나하나(C 훅, LD_PRELOAD, Docker, FastAPI 백엔드)는 이제 낯설지 않을 거예요. 이 마지막 장에서는 그 부품들을 **하나의 이야기로** 엮습니다. "사용자가 버튼 하나 눌렀을 때, 그 요청이 어떤 부품들을 거쳐 GPU 위에서 실제로 강제되는가?"를 끝까지 따라가 봅니다.

---

> 📘 **이 장을 읽고 나면**
>
> - 세션 생성 요청 하나가 **백엔드 → Docker → 컨테이너 → 훅 → GPU**를 어떻게 관통하는지 그림으로 그릴 수 있습니다.
> - 앞에서 배운 조각들(어드미션, LD_PRELOAD, 쿼터, mutex)이 **정확히 어느 순간** 등장하는지 압니다.
> - 프로젝트를 처음 열었을 때 **어느 파일부터 열어야 하는지** 실전 팁을 얻습니다.
> - 이 프로젝트의 **한계와 다음 발전 방향**을 이해합니다.

---

## 1. 하나의 요청, 끝까지 따라가기

시나리오: 어떤 사용자가 **"GPU의 40%를 쓰는 세션을 하나 만들어줘"** 라고 요청합니다. 웹 UI에서 ratio=0.4로 만들기 버튼을 누른 상황이에요.

아래 순서를 따라가며, 각 단계가 **어느 챕터에서 배운 내용인지** 괄호로 표시했습니다.

```
사용자 ──▶ POST /sessions {"ratio": 0.4}
              │
              ▼
┌─────────────────────────────────────────────────────────────┐
│ [백엔드: FastAPI]  (7장)                                       │
│  1. _require_auth 로 토큰 검사 (Bearer)                        │
│  2. sessions.py 의 create_session 라우터가 요청을 받음         │
│     → SessionManager.create(ratio=0.4) 호출                    │
└─────────────────────────────────────────────────────────────┘
              │
              ▼
┌─────────────────────────────────────────────────────────────┐
│ [백엔드: 문지기 ① 어드미션]  (8장)                             │
│  3. asyncio.Lock 을 잡는다 (한 번에 하나씩만 검사 — 동시성)    │
│  4. 지금 살아있는 세션들의 ratio 합을 구한다                   │
│  5. 합 + 0.4 > 1.0 이면 → HTTP 409 거절 (force 로 무시 가능)   │
│     통과하면 계속                                             │
└─────────────────────────────────────────────────────────────┘
              │
              ▼
┌─────────────────────────────────────────────────────────────┐
│ [백엔드 → Docker: 컨테이너 spawn]  (3장 + 8장)                 │
│  6. DockerManager 가 docker run 옵션을 조립:                   │
│       --gpus all            (GPU 노출)                         │
│       -v .../libfgpu.so:/opt/fgpu/libfgpu.so:ro  (훅 마운트)   │
│       -e LD_PRELOAD=/opt/fgpu/libfgpu.so         (가로채기 켜기)│
│       -e FGPU_RATIO=0.4     (한도를 환경변수로 주입!)          │
│  7. asyncio.to_thread 로 blocking 한 docker 호출을 감싼다       │
│  8. 컨테이너가 뜬다                                            │
└─────────────────────────────────────────────────────────────┘
              │
              ▼
┌─────────────────────────────────────────────────────────────┐
│ [백엔드: SQLite 기록]  (8장)                                   │
│  9. 세션 정보(id, container_id, ratio, status...)를           │
│     data/sessions.db 에 저장 → 백엔드 재시작해도 안 사라짐     │
│  10. 201 응답을 사용자에게 반환                                │
└─────────────────────────────────────────────────────────────┘

    ─────────── 여기서부터는 컨테이너 안에서 벌어지는 일 ───────────

┌─────────────────────────────────────────────────────────────┐
│ [컨테이너 안: GPU 프로그램 실행]  (4장 + 5장)                  │
│  11. 프로그램(예: PyTorch)이 시작될 때 LD_PRELOAD 때문에       │
│      libfgpu.so 가 먼저 로드된다                               │
│  12. 프로그램이 cudaMalloc(3GB) 를 호출 → 진짜가 아니라        │
│      우리 훅의 cudaMalloc 이 먼저 불린다 (가로채기 성공!)      │
└─────────────────────────────────────────────────────────────┘
              │
              ▼
┌─────────────────────────────────────────────────────────────┐
│ [훅: 문지기 ② 쿼터 강제]  (5장 + 6장)                         │
│  13. 처음이면 cudaMemGetInfo 로 GPU 총량 확인 → 쿼터 계산      │
│      (8GB × 0.4 = 3.2GB)  ← lazy 계산                          │
│  14. mutex 자물쇠를 잡는다 (동시 접근 안전)                    │
│  15. g_used + 요청크기 > g_quota ?                             │
│        아니오 → 진짜 cudaMalloc 위임 + g_used 증가 → ALLOW     │
│        예     → cudaErrorMemoryAllocation 반환   → DENY        │
│  16. [fgpu] ALLOW/DENY 로그를 stderr 에 남긴다                 │
└─────────────────────────────────────────────────────────────┘
              │
              ▼
┌─────────────────────────────────────────────────────────────┐
│ [NVIDIA 드라이버: 문지기 ③ 물리 한계]                         │
│  17. 훅이 통과시켜도, 진짜 물리 메모리가 없으면 여기서 OOM     │
└─────────────────────────────────────────────────────────────┘
```

이 한 장의 그림이 **프로젝트 전체**입니다. 각 상자를 짚으며 "아, 이건 몇 장에서 배운 거지" 하고 떠올릴 수 있다면 이미 이해도 80%를 넘긴 거예요.

### 핵심 통찰 3가지

1. **한도는 "환경변수"로만 전달됩니다.** 백엔드는 훅에게 함수 호출이나 네트워크로 명령하지 않아요. 그저 컨테이너를 띄울 때 `FGPU_RATIO=0.4`를 꽂아줄 뿐입니다. 훅은 그걸 읽어서 혼자 판단해요. 이 **느슨한 결합** 덕분에 나중에 백엔드를 통째로 바꿔도(예: 쿠버네티스) 훅은 그대로 씁니다.

2. **훅은 컨테이너마다 독립입니다.** LD_PRELOAD는 각 컨테이너 안에서 `libfgpu.so`를 따로 로드해요. 그래서 `g_used`, `g_quota` 같은 상태는 컨테이너별로 완전히 분리됩니다. A의 사용량과 B의 사용량은 서로 모릅니다 — 알 필요도 없고요.

3. **세 문지기는 서로 다른 실패를 잡습니다.** 어드미션은 "만들기 전에 초과 예약"을, 훅은 "실행 중 한도 초과"를, 드라이버는 "진짜 물리 부족"을 잡아요. 하나가 놓쳐도 다음이 받쳐줍니다.

## 2. 프로젝트를 처음 열었을 때 — 어느 파일부터?

실전 팁입니다. 이 순서로 파일을 열어보세요.

1. **[CLAUDE.md](../../CLAUDE.md)** — 프로젝트 전체 지도. 파일별 역할과 스테이지별 성공 기준이 다 적혀 있어요. **가장 먼저 훑을 파일.**
2. **[fgpu_hook.c의 맨 위 주석](../../hook/src/fgpu_hook.c#L1-L63)** — 63줄짜리 한글 주석이 프로젝트 심장을 요약합니다.
3. **[hook/tests/test_alloc.cu](../../hook/tests/test_alloc.cu)** — 훅을 검증하는 가장 단순한 테스트. "256MB는 통과, 6GB는 거절"이 눈에 보여요.
4. **[backend/app/api/sessions.py](../../backend/app/api/sessions.py)** — REST API의 입구. 어떤 엔드포인트가 있는지.
5. **[backend/app/services/session_manager.py](../../backend/app/services/session_manager.py)** — 세션이 태어나는 곳. 위 그림의 3~10단계가 여기 있어요.
6. **[description.md](../../description.md)** — 설계의 "왜"가 궁금할 때 읽는 긴 한글 문서.

## 3. 이 프로젝트가 **안 하는** 것 (한계)

교과서의 정직한 마무리로, 이 프로토타입이 의도적으로 포기한 것들을 짚습니다. 면접이나 발표에서 "한계가 뭐냐"는 질문의 모범답안이에요.

| 한계 | 설명 |
|------|------|
| **협조적(cooperative) 위협 모델** | 훅은 LD_PRELOAD로 동작하므로, **정적 링크된 악의적 프로그램**은 훅을 우회할 수 있어요. 착한 사용자를 전제합니다. |
| **SM(연산 코어) 격리 불가** | 메모리는 나누지만, 연산 코어 자체를 하드웨어로 쪼개지는 못해요. 그건 데이터센터 GPU의 MIG 기능이 필요한데 RTX 4060엔 없어요. Stage 12의 스로틀은 "시간 나눠쓰기"라는 협조적 흉내일 뿐입니다. |
| **PyTorch 캐싱 앨로케이터** | PyTorch는 메모리를 한 번에 크게 잡아두고 재활용해서, per-call 쿼터가 가려질 수 있어요. 그래서 테스트할 땐 `PYTORCH_NO_CUDA_MEMORY_CACHING=1`로 캐싱을 끕니다. |
| **단일 호스트** | GPU 한 대, 서버 한 대 기준이에요. 여러 서버에 걸친 스케줄링(멀티노드)은 미래 과제(Stage 9 full)입니다. |

이 한계들은 **버그가 아니라 설계 결정**입니다. 프로토타입의 목적(메커니즘 학습·논문화)에 맞게 범위를 좁힌 거예요.

## 4. 다음 스텝 — 이제 뭘 하면 되나요?

교과서를 다 읽었다면, 손을 움직여 볼 차례입니다.

1. **9장의 실행 순서**를 따라 (GPU가 달린 Ubuntu 서버에서) `build_hook.sh` → `build_image.sh` → `run_in_container.sh`를 직접 돌려보고, `[fgpu] ALLOW`와 `[fgpu] DENY`가 눈앞에 뜨는 걸 확인하세요. 이 "아하!" 순간이 모든 걸 연결해 줍니다.
2. **작은 실험**: `FGPU_RATIO` 값을 0.4 → 0.6으로 바꿔가며 6GB 할당이 DENY에서 ALLOW로 바뀌는 걸 관찰하세요.
3. **코드 한 줄 고쳐보기**: 훅의 로그 메시지 하나를 바꿔 다시 빌드해 보세요. C 컴파일과 `.so` 마운트의 전체 사이클을 몸으로 익히게 됩니다.
4. **더 깊은 문서**: 이 온보딩 교과서를 뗐다면, 더 전문적인 [docs/study/](../study/) 챕터들과 [description.md](../../description.md), [ARCHITECTURE.md](../../ARCHITECTURE.md)로 넘어가세요.

## ✍️ 스스로 점검

1. `FGPU_RATIO=0.4`라는 환경변수는 어디서 만들어져서(누가 설정) 어디서 읽히나요(누가 사용)? 위 그림에서 두 지점을 짚어보세요.
2. 어드미션(문지기 ①)이 통과시켰는데도 훅(문지기 ②)이 DENY할 수 있는 상황을 하나 상상해 보세요. (힌트: 어드미션은 "예약"만 봅니다.)
3. 이 프로젝트가 "협조적 위협 모델"이라는 게 무슨 뜻이고, 왜 그런 한계를 받아들였는지 설명해 보세요.
