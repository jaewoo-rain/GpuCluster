# 4장. LD_PRELOAD 와 dlsym — 이 프로젝트가 작동하는 마법의 원리

> 📘 **이 장을 읽고 나면**
>
> - 프로그램이 실행될 때 `libcudart.so` 같은 라이브러리를 "어떻게 찾아서 연결하는지"(동적 링킹)를 설명할 수 있어요.
> - `LD_PRELOAD` 환경변수 한 줄로 어떻게 남의 함수를 가로챌 수 있는지 이해하게 돼요.
> - 가로챈 다음 "진짜 함수"를 다시 부르는 `dlsym(RTLD_NEXT, ...)` 기법과 함수 포인터를 이해하게 돼요.
> - 이 프로젝트가 **왜** 이 방식을 골랐는지, 그리고 어떤 한계(정적 링크 우회)가 있는지 말할 수 있어요.

이 프로젝트(fGPU) 전체가 사실 딱 하나의 아이디어 위에 서 있어요. 바로 **"남이 부르는 `cudaMalloc` 을 몰래 내 함수로 바꿔치기한다"** 는 거예요. 이 장에서는 그 바꿔치기가 어떻게 가능한지를 아주 천천히 풀어봅니다. 이 장을 이해하면 나머지 챕터가 전부 쉬워져요.

---

## 4.1 먼저: 프로그램은 함수를 어떻게 "찾을까"? (동적 링킹)

### (1) 왜 필요한가

여러분이 C 로 이런 코드를 짰다고 해봅시다.

```c
#include <stdio.h>
int main(void) {
    printf("Hello\n");
    return 0;
}
```

`printf` 라는 함수, 여러분이 직접 만든 적 없죠? 이건 C 표준 라이브러리(`libc`) 안에 들어 있어요. 그런데 신기하게도 여러분의 프로그램 실행 파일 안에는 `printf` 의 **실제 코드가 들어있지 않아요.** 대신 "실행할 때 `libc` 라는 파일 안에서 `printf` 를 찾아서 연결해줘" 라는 **메모(약속)** 만 들어 있습니다.

이렇게 **실행하는 순간에** 필요한 라이브러리 파일(`.so`)을 찾아 연결하는 것을 **동적 링킹(dynamic linking)** 이라고 불러요. 리눅스에서 `.so` 는 "shared object", 즉 여러 프로그램이 공유해서 쓰는 라이브러리 파일이에요. (윈도우의 `.dll` 과 같은 개념이라고 보면 돼요.)

CUDA 도 똑같아요. 여러분이 `cudaMalloc` 을 부르면, 그 실제 코드는 `libcudart.so`(CUDA Runtime 라이브러리) 안에 있고, 프로그램은 실행 시점에 그걸 찾아 연결합니다.

### (2) 일상 비유

동적 링킹은 **"전화번호부"** 같아요.

여러분 휴대폰에 "엄마" 라고 저장돼 있죠. 실제 전화번호(010-xxxx-xxxx)를 외우고 다니는 게 아니라, "엄마" 라는 **이름표** 만 기억하고, 전화 거는 순간에 전화번호부에서 진짜 번호를 찾아 연결해요. 프로그램도 `printf` 라는 **이름표** 만 들고 다니다가, 실행 순간에 `libc` 라는 전화번호부에서 진짜 주소를 찾습니다.

### (3) 심볼(symbol) = 함수의 이름표

이 "이름표" 를 전문 용어로 **심볼(symbol)** 이라고 불러요. `cudaMalloc`, `printf`, `cudaFree` — 전부 심볼이에요. 라이브러리 파일 안에는 "이 심볼(이름)이 이 주소에 있다" 는 표가 들어 있고, 동적 링커(dynamic linker, 리눅스에서는 `ld-linux.so`)가 실행 시점에 그 표를 뒤져서 연결해줍니다.

핵심 문장: **프로그램은 함수 "이름(심볼)" 으로 라이브러리를 찾고, 실행 순간에 진짜 주소로 연결한다.**

### (5) 흔한 함정

- **"실행 파일 안에 다 들어있겠지?"** → 아니에요. 동적 링킹된 프로그램은 라이브러리가 없으면 실행조차 안 됩니다. (`libcudart.so 를 찾을 수 없음` 같은 에러를 본 적 있을 거예요.)
- 반대로 라이브러리를 실행 파일 안에 **통째로 박아버리는** 방식도 있어요. 그걸 **정적 링킹(static linking)** 이라고 하는데, 이건 우리 프로젝트에서 가로챌 수 없는 대상이에요. (뒤에서 다시 나옵니다.)

### (6) 한 줄 요약
> 동적 링킹 = 프로그램이 함수 "이름표(심볼)" 만 들고 다니다가, 실행 순간에 라이브러리 파일에서 진짜 주소를 찾아 연결하는 것.

## 4.2 LD_PRELOAD — "내 라이브러리를 남보다 먼저 로드해줘"

### (1) 왜 중요한가

이게 이 프로젝트의 **심장** 이에요.

방금 배운 것처럼, 프로그램이 `cudaMalloc` 을 부르면 동적 링커가 라이브러리들을 뒤져서 진짜 `cudaMalloc` 을 찾아 연결해요. 그런데 만약 **똑같은 이름의 `cudaMalloc` 이 두 군데** 있으면 어떻게 될까요? 동적 링커는 **먼저 로드된** 것을 선택해요.

`LD_PRELOAD` 는 바로 이 "먼저" 를 우리가 조종하게 해주는 환경변수예요.

```bash
LD_PRELOAD=/path/to/libfgpu.so  ./내프로그램
```

이렇게 실행하면, 동적 링커는 `libcudart.so` 보다 **먼저** `libfgpu.so`(우리 훅)를 로드해요. 그리고 우리 `libfgpu.so` 안에도 `cudaMalloc` 이라는 이름의 함수가 들어 있죠. 그러면 프로그램이 `cudaMalloc` 을 부를 때, 진짜 CUDA 것이 아니라 **우리 것이 먼저 잡혀서** 불립니다. 이게 바로 **후킹(hooking, 가로채기)** 이에요.

### (2) 일상 비유

**착신전환** 을 생각해보세요.

친구가 여러분 집 전화("cudaMalloc") 로 전화를 걸어요. 그런데 여러분이 전화국에 "내 집 전화로 오는 전화는 전부 먼저 내 휴대폰으로 돌려줘" 라고 설정해뒀어요(`LD_PRELOAD`). 이제 친구가 집 전화로 걸면 여러분 휴대폰이 먼저 울려요. 여러분은 전화를 받아서 (1) 통화 내용을 기록하고(=quota 검사), (2) 필요하면 진짜 집 전화로 다시 연결해줄 수 있어요.

친구(사용자 프로그램)는 자기가 착신전환된 줄도 몰라요. **그냥 평소처럼 `cudaMalloc` 을 불렀을 뿐인데** 중간에 우리가 끼어든 거예요. 사용자 코드를 단 한 줄도 안 고쳤다는 게 핵심입니다.

### (3) 작은 예시

우리 훅에서 `cudaMalloc` 함수의 정의를 보면, 이름이 정확히 `cudaMalloc` 이에요. C 언어라서 이름이 그대로 심볼이 되고, 그래서 진짜 것과 이름이 "충돌" 하면서 우리 것이 이깁니다.

- [fgpu_hook.c:441](../../hook/src/fgpu_hook.c#L441) — `cudaError_t cudaMalloc(void **devPtr, size_t size)` : 진짜 CUDA 와 **똑같은 이름/똑같은 시그니처** 로 정의된 우리 함수. LD_PRELOAD 덕분에 이게 진짜 대신 불려요.

### (4) 이 프로젝트가 왜 LD_PRELOAD 를 골랐나

- **사용자 코드 수정 0.** 연구용 프로토타입에서 "여러분 PyTorch 코드에 우리 라이브러리 import 하세요" 라고 요구하면 아무도 안 써요. LD_PRELOAD 는 실행 명령어 앞에 환경변수 하나만 붙이면 됩니다.
- **컨테이너 단위로 깔끔.** 도커 컨테이너를 띄울 때 `-e LD_PRELOAD=...` 로 그 컨테이너 안에서 도는 모든 프로그램에 훅을 자동 주입할 수 있어요. (3장에서 배운 도커 개념이 여기서 이어집니다.) 컨테이너마다 각자의 훅 인스턴스가 생기니, 각 컨테이너가 자기만의 quota 를 가질 수 있어요.

### (5) 흔한 함정

- **LD_PRELOAD 경로가 틀리면 조용히 무시돼요.** 에러 없이 그냥 훅이 안 걸립니다. `[fgpu] init` 로그가 안 보이면 경로부터 의심하세요.
- **정적 링크된 프로그램은 못 가로채요.** `cudaMalloc` 코드가 실행 파일 안에 통째로 박혀 있으면, 라이브러리를 찾을 일이 없어서 우리가 끼어들 틈이 없어요. 이건 이 프로젝트의 **알려진 한계** 이지, 고칠 버그가 아니에요. (우리는 "협조적 위협 모델" 을 가정해요. 뒤에서 설명합니다.)

### (6) 한 줄 요약
> `LD_PRELOAD` = "내 `.so` 를 남보다 먼저 로드해줘" → 같은 이름의 함수가 있으면 내 것이 이긴다 → 이게 후킹.

## 4.3 그런데 진짜 cudaMalloc 도 결국 불러야 한다 — dlsym(RTLD_NEXT)

### (1) 왜 필요한가

우리 훅이 `cudaMalloc` 을 가로챈 건 좋아요. quota 검사도 했어요. 그런데 결국 **진짜로 GPU 메모리를 할당하긴 해야** 하잖아요? 착신전환 비유에서, 통화 내용을 기록한 뒤에는 결국 진짜 집 전화로 연결해줘야 대화가 되죠.

문제는 이거예요. 우리 함수 이름도 `cudaMalloc`, 진짜 함수 이름도 `cudaMalloc`. 우리 안에서 그냥 `cudaMalloc(...)` 이라고 쓰면 **자기 자신을 다시 부르는** 무한 재귀에 빠져요! 진짜 것을 부를 방법이 필요합니다.

### (2) 일상 비유

착신전환을 켜두면, 여러분이 집 전화번호로 전화를 걸어도 다시 여러분 휴대폰으로 돌아와요(무한 루프). 진짜 집 전화기에 연결하려면 착신전환을 **우회하는 특별한 방법** 이 필요해요. `dlsym(RTLD_NEXT, ...)` 이 바로 그 특별한 우회로예요.

### (3) 핵심 도구 두 가지

**① `dlsym` — "이 함수 주소 좀 찾아줘"**

`dlsym` 은 동적 링커에게 "어떤 이름의 함수가 어느 주소에 있는지 찾아줘" 라고 직접 묻는 함수예요.

**② `RTLD_NEXT` — "나 말고, 내 다음에 로드된 데서 찾아줘"**

`dlsym` 의 첫 번째 인자로 `RTLD_NEXT` 를 주면, "지금 이 `.so`(우리 훅) **다음에** 로드된 라이브러리들 중에서 찾아줘" 라는 뜻이 돼요. 우리 다음에 로드된 건 진짜 `libcudart.so` 죠. 그래서:

```c
real_cudaMalloc = dlsym(RTLD_NEXT, "cudaMalloc");
```

이 한 줄이 **진짜 CUDA 의 `cudaMalloc` 주소** 를 찾아서 돌려줍니다. 자기 자신이 아니라요.

### (3-bis) 함수 포인터 — 주소를 변수에 담기

`dlsym` 이 돌려주는 건 함수의 **주소(번지수)** 예요. 이걸 저장할 변수가 필요한데, 그게 바로 **함수 포인터** 예요.

포인터는 이미 배웠죠? `int *p` 는 "int 가 있는 주소를 담는 변수". 함수 포인터는 "함수가 있는 주소를 담는 변수" 예요. 문법이 좀 낯설지만 뜯어보면 간단해요.

```c
static cudaError_t (*real_cudaMalloc)(void **, size_t) = NULL;
```

이걸 조각내서 읽으면:
- `real_cudaMalloc` : 변수 이름
- `(*real_cudaMalloc)` : 이건 "포인터" 라는 뜻 (별표)
- `(void **, size_t)` : 이 함수는 `void**` 와 `size_t` 두 개를 받는다
- `cudaError_t` : 이 함수는 `cudaError_t` 를 돌려준다
- `= NULL` : 처음엔 비어 있음(아직 주소를 안 찾음)

즉 "`void**` 와 `size_t` 를 받아 `cudaError_t` 를 돌려주는 함수의 주소를 담을 변수" 예요. 진짜 `cudaMalloc` 의 모양과 정확히 똑같죠. 나중에 이 변수에 담긴 함수를 부를 때는 그냥 일반 함수처럼 `real_cudaMalloc(devPtr, size)` 라고 쓰면 돼요.

### (4) fgpu_hook.c 실제 코드에서 보기

**진짜 함수 주소를 담을 함수 포인터들** (처음엔 전부 `NULL`):
- [fgpu_hook.c:94](../../hook/src/fgpu_hook.c#L94) — `real_cudaMalloc`, `real_cudaFree` 선언
- [fgpu_hook.c:100](../../hook/src/fgpu_hook.c#L100) — Driver API 용 `real_cuMemAlloc_v2`, `real_cuMemFree_v2`
- [fgpu_hook.c:106](../../hook/src/fgpu_hook.c#L106) — VMM API 용 `real_cuMemCreate`, `real_cuMemRelease`
- [fgpu_hook.c:112](../../hook/src/fgpu_hook.c#L112) — `real_cudaLaunchKernel`

**dlsym 으로 실제 주소를 채우는 부분** (초기화 루틴 안):
- [fgpu_hook.c:307](../../hook/src/fgpu_hook.c#L307) — `if (!real_cudaMalloc) real_cudaMalloc = dlsym(RTLD_NEXT, "cudaMalloc");` 그리고 그 아래로 나머지 심볼들도 같은 패턴으로 해석해요([~313줄까지](../../hook/src/fgpu_hook.c#L313)).

여기서 `if (!real_cudaMalloc)` 에 주목하세요. **아직 안 찾은(NULL 인) 것만** `dlsym` 을 다시 시도해요. 왜 매번 다시 시도할까요? 사용자 프로그램이 `libcuda.so` 를 **나중에** `dlopen` 으로 뒤늦게 로드하는 경우가 있어요. 그 경우 첫 초기화 때는 driver 심볼이 아직 없어서 `NULL` 이지만, 나중에 다시 시도하면 잡혀요. 이걸 **lazy(지연) 심볼 해석** 이라고 부릅니다. 관련 주석은 [fgpu_hook.c:296](../../hook/src/fgpu_hook.c#L296) 근처에 자세히 있어요.

**진짜 함수를 실제로 부르는 부분**:
- [fgpu_hook.c:465](../../hook/src/fgpu_hook.c#L465) — `cudaError_t err = real_cudaMalloc(devPtr, size);` : quota 검사를 통과한 뒤, 함수 포인터에 담긴 **진짜** `cudaMalloc` 을 호출해서 실제 GPU 메모리를 할당해요.

**초기화가 끝나면 찍히는 로그** — GPU 서버에서 이걸 확인하게 될 거예요:
- [fgpu_hook.c:378](../../hook/src/fgpu_hook.c#L378) — `[fgpu] init: ratio=... quota_bytes=...`
- [fgpu_hook.c:381](../../hook/src/fgpu_hook.c#L381) — `[fgpu] init: real cudaMalloc=0x... cudaFree=0x... ...` : 이 로그에서 주소가 `0x0`(NULL) 이 아니라 진짜 숫자로 찍혀 있으면 "심볼 해석 성공" 이라는 뜻이에요.

### (5) 흔한 함정

- **`_GNU_SOURCE` 를 안 정의하면 `RTLD_NEXT` 를 못 써요.** `RTLD_NEXT` 는 GNU 확장 기능이라, `#include` 보다 **먼저** `#define _GNU_SOURCE` 를 해야 해요. 우리 파일 맨 위 [fgpu_hook.c:70](../../hook/src/fgpu_hook.c#L70) 에 그래서 이게 있어요. 빠뜨리면 `RTLD_NEXT undeclared` 컴파일 에러가 나요.
- **함수 이름이 진짜와 한 글자라도 다르면** 후킹이 안 돼요. 예를 들어 driver API 는 `cuMemAlloc` 이 아니라 정확히 `cuMemAlloc_v2` 라는 심볼로 잡아야 해요([fgpu_hook.c:309](../../hook/src/fgpu_hook.c#L309)). (`_v2` 는 CUDA 4.0 에서 ABI 가 바뀐 뒤 유지되는 공식 심볼명이에요.)
- **초기 로그에서 주소가 `0x0` 으로 찍히면** 그 심볼은 아직 해석 안 된 거예요. 사용자가 그 API 를 안 써서일 수도 있고(정상), 로드 순서 문제일 수도 있어요.

### (6) 한 줄 요약
> `dlsym(RTLD_NEXT, "cudaMalloc")` = "내 다음에 로드된 진짜 라이브러리에서 `cudaMalloc` 주소를 찾아줘" → 함수 포인터에 담아뒀다가 quota 검사 후 진짜를 호출.

---

## 4.4 왜 이 방식인가, 그리고 어디까지 못 막나 (협조적 위협 모델)

### 장점 정리
1. **사용자 코드 수정 0** — 명령어 앞에 환경변수만.
2. **컨테이너 단위 격리** — 컨테이너마다 독립된 훅 인스턴스 = 독립된 quota.
3. **여러 API 계층을 동시에 잡을 수 있음** — `cudaMalloc`(Runtime), `cuMemAlloc_v2`(Driver), `cuMemCreate`(VMM) 를 전부 후킹. (자세한 건 5장에서.)

### 한계: 협조적 위협 모델 (cooperative threat model)

우리 방식은 **"사용자가 딱히 우리를 피하려 하지 않는다"** 를 가정해요. 이걸 "협조적(cooperative) 위협 모델" 이라고 불러요. 왜냐면 마음먹고 피하려는 사람은 피할 수 있거든요:

- **정적 링크된 바이너리** — `cudaMalloc` 코드가 실행 파일 안에 박혀 있으면 라이브러리를 안 찾으니 우리가 못 낀다.
- **직접 `dlopen`** — 사용자가 진짜 `libcuda.so` 를 직접 열어서 함수 주소를 뽑아 쓰면 우리 심볼을 우회할 수 있다.

이건 **버그가 아니라 설계상 명시된 한계** 예요. 이 프로젝트는 악의적 공격자를 막는 보안 도구가 아니라, "정상적으로 CUDA API 를 쓰는 컨테이너들끼리 GPU 메모리를 공평하게 나눠 쓰자" 는 협력적 자원 관리 도구예요. 논문/문서에도 이걸 **한계로 미리 적어둡니다.**

### 한 줄 요약
> 이 방식은 코드 수정 없이 컨테이너 단위로 GPU quota 를 강제하지만, 마음먹고 우회하는(정적 링크/직접 dlopen) 사용자까지 막지는 않는 "협조적" 모델이다.
