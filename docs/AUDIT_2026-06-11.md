### 1. create partial failure leaks orphan container

- **심각도**: 🔴 HIGH  ·  **분류**: error-propagation  ·  **검증 confidence**: high
- **위치**: `session_manager.py:147`  ·  도메인: backend-async-persist

**문제**: Container started at line 147, recorded in DB only at line 199, no rollback; if store.insert raises the container keeps running, remove=False, no DB row, admission never counts it, GPU quota held forever.

**근거**: lines 147,199; remove=False docker_manager.py:152

**검증**: 코드를 직접 읽고 확인함. session_manager.py:147 에서 `self.docker.create_container` 로 컨테이너가 실제로 spawn 되고(docker_manager.py:157 `containers.run(detach=True)`), DB record 는 199 라인 `self.store.insert(rec)` 에서야 기록됨. 147~199 사이를 try/except 로 감싸는 롤백이 전혀 없음(create/_create_locked 함수 전체에 예외 핸들러 없음).

이 구간에서 예외가 realistically 발생 가능:
1) store.insert → session_store.py:87 `sqlite3.connect(..., timeout=5.0)`, WAL 미사용(파일 주석 line 12에서 명시). _create_lock 은 create 만 직렬화하고, get/list_all/delete 가 호출하는 update_status/delete 는 동시에 다른 worker thread 에서 도므로 5초 초과 시 `OperationalError: database is locked` 가능. 디스크 풀도 raise.
2) jupyter 모드의 host_port 폴링 루프(168-179)나 to_thread insert 도중 클라이언트 disconnect → CancelledError. 컨테이너는 이미 떠있음.
3) Session(...) pydantic 구성(181-198) 검증 실패.

결과 체인 확인:
- 컨테이너는 docker_manager.py:149 `remove=False` 로 생성 → 예외 시에도 계속 running.
- DB row 없음 → store.list_all() 이 절대 반환 안 함.
- admission.py:82-89 `sum_used_ratio` 는 sessions(=DB 레코드) 만 순회 → orphan 의 ratio 를 영원히 카운트 안 함. 또한 admission.py 주석(line 9-11)대로 reconcile 은 DB 레코드 기준으로만 docker 와 맞추므로, docker daemon 을 enumerate 하지 않음 → orphan 은 reconcile 에서도 보이지 않음.
- 즉 GPU ratio quota 가 docker 측에서는 실제 점유되지만 admission 레이어에서는 안 보임 → admission 계산이 실제 사용량과 어긋나고, 수동 `docker rm` 전까지 자원 누수.

이것은 의도된 한계(하드웨어 격리/협력 위협모델/allocator caching/single-host 등) 중 어디에도 해당하지 않는 실제 에러 전파/자원 누수 버그임. 근거 file:line: session_manager.py:147(spawn), :199(insert), 147-200 사이 rollback 부재; docker_manager.py:149(remove=False), :157(run detach); admission.py:82(DB sessions 만 순회); session_store.py:87(timeout=5.0, WAL 없음).

**수정안**: _create_locked 의 컨테이너 spawn(147) 이후 store.insert(199) 까지를 try/except 로 감싸고, 예외 발생 시 생성된 컨테이너를 best-effort 로 제거하여 롤백: 예) `c = await asyncio.to_thread(self.docker.create_container, ...)` 직후 `try:` 블록으로 host_port 폴링 + Session 구성 + store.insert 를 감싸고, `except BaseException:`(CancelledError 포함) 에서 `await asyncio.to_thread(self.docker.remove_container, c.id, True)` 를 NotFound 무시하고 호출한 뒤 re-raise. 보강책으로 admission.list_all reconcile 시 docker daemon 의 fgpu-* 컨테이너를 enumerate 해 DB 에 없는 orphan 을 탐지/회수하는 경로를 두면 부분 실패가 누적돼도 quota 정합성 회복 가능.

### 2. _correlate.py: docker(UTC) vs nvidia-smi(로컬시각) 타임스탬프 혼용으로 시계열 조인이 UTC offset만큼 어긋남

- **심각도**: 🔴 HIGH  ·  **분류**: bug  ·  **검증 confidence**: high
- **위치**: `/home/jaewoo/Desktop/backend_ai/GpuCluster/scripts/eval/_correlate.py:61`  ·  도메인: eval-scripts

**문제**: launch 로그는 `docker logs --timestamps`가 RFC3339 UTC(`...Z`)로 찍은 값을 파싱해 tzinfo=UTC로 정확히 태깅한다(line 55에서 'Z'를 strip하는 것이 그 증거). 반면 nvidia-smi --query-compute-apps의 timestamp는 TZ 표기 없는 호스트 '로컬 시각'인데, parse_nvsmi가 strptime 후 무조건 tzinfo=timezone.utc로 강제 태깅한다. 결과적으로 두 시계열이 호스트의 UTC offset(예: KST면 +9h)만큼 서로 어긋나고, t0 = min(all_ts) 계산과 correlation.csv의 (launch_count, used_memory_mib) 시간정렬/조인이 전부 그만큼 틀어진다. 산출물의 핵심 목적(시점별 launch counter ↔ GPU 메모리 상관)이 무효화된다. nvidia-smi에 `--format=...`로 UTC를 강제하지 않는 한, 로컬 tz를 datetime에 부여(astimezone)하거나 동일 기준으로 맞춰야 한다.

**근거**: line 61: `ts = datetime.fromisoformat(ts_str).replace(tzinfo=timezone.utc)` (docker=UTC, 정확) vs line 83-84: `ts = datetime.strptime(row[0].strip(), "%Y/%m/%d %H:%M:%S.%f"); ts = ts.replace(tzinfo=timezone.utc)` (nvidia-smi 로컬시각을 UTC로 오태깅). run_correlation.sh:79는 `nvidia-smi --query-compute-apps=timestamp,... --format=csv,noheader`로 TZ 없는 로컬 timestamp를 생성.

**검증**: 코드를 직접 읽어 확인한 결과 주장은 사실이다.

증거 (file:line):

### 3. cudaMalloc/cudaFree 메인 경로에서 real 심볼 NULL 가드 누락 → dlsym 실패 시 NULL 역참조 크래시

- **심각도**: 🟠 MEDIUM  ·  **분류**: contract-mismatch  ·  **검증 confidence**: medium
- **위치**: `hook/src/fgpu_hook.c:465`  ·  도메인: hook-memory

**문제**: driver API hook들(cuMemAlloc_v2:550, cuMemFree_v2:597, cuMemCreate:658, cuMemRelease:703)은 fgpu_init_locked() 후 real 심볼이 NULL이면 '[fgpu] FAIL ... not resolved' 로그 + 표준 에러(CUDA_ERROR_NOT_INITIALIZED) 반환으로 방어한다. 그러나 Runtime API의 cudaMalloc(441)과 cudaFree(494)은 메인 경로에서 이 가드가 없다. cudaMalloc은 line 465에서 `real_cudaMalloc(devPtr, size)`를, cudaFree는 line 503에서 `real_cudaFree(devPtr)`를 NULL 체크 없이 곧바로 호출한다. reentrancy 진입 경로(446, 496)는 삼항연산자로 NULL을 막지만, 정작 정상 진입 경로는 막지 않는다. dlsym(RTLD_NEXT,"cudaMalloc")이 NULL을 돌려주는 상황(정적링크 cudart, 심볼 미해결 등)에서 real_cudaMalloc이 NULL인 채 line 465가 실행되어 NULL 함수포인터 역참조로 SIGSEGV. 동형 함수들끼리 에러처리 계약이 불일치하며, runtime 경로만 graceful degradation이 빠져 있다. (정적링크는 의도된 한계지만, 그 경우 '안전하게 거절'이 아니라 '크래시'한다는 점이 회귀.)

**근거**: L465: `cudaError_t err = real_cudaMalloc(devPtr, size);` (선행 NULL 체크 없음) vs L550-558: `if (!real_cuMemAlloc_v2) { ...FAIL... return CUDA_ERROR_NOT_INITIALIZED; }`. cudaFree도 L503: `cudaError_t err = real_cudaFree(devPtr);` 가드 없음.

**검증**: 코드 확인 결과 주장은 정확하다. hook/src/fgpu_hook.c에서:
- L465 `cudaError_t err = real_cudaMalloc(devPtr, size);` — 메인(비-reentrant) 경로에 선행 NULL 가드 없음.
- L503 `cudaError_t err = real_cudaFree(devPtr);` — 동일하게 가드 없음.
- 반면 동형 driver API 훅들은 모두 fgpu_init_locked() 직후 명시적 가드를 둔다: cuMemAlloc_v2 L550-558 `if (!real_cuMemAlloc_v2){ ...FAIL...; return CUDA_ERROR_NOT_INITIALIZED; }`, cuMemFree_v2 L597-603 동일. cuMemCreate/cuMemRelease도 동일 패턴(grep상 655/701 init 후 가드).
- 더구나 cudaMalloc/cudaFree 자신의 reentrancy 경로(L446, L496)는 삼항으로 NULL을 막는다(`real_cudaMalloc ? ... : cudaErrorInitializationError`). 즉 저자는 NULL 가드 패턴을 명백히 의도했고, 정작 정상 경로 2곳에서만 빠뜨렸다 — 의도적 설계가 아니라 누락/비일관이다.

이것이 '의도된 한계(정적 링크 cudart)'인지 검토: 의도된 한계 목록은 정적링크/직접 dlopen 우회를 협력 위협모델로 인정한다. 그러나 정적링크 시에는 앱의 자체 cudart 심볼이 이겨 이 훅 자체가 호출되지 않으므로, 바로 그 시나리오에서는 NULL 역참조가 발생하지 않는다. 실제 NULL deref가 일어나려면 (a) 훅이 동적 interpose로 진입했는데 (b) dlsym(RTLD_NEXT,"cudaMalloc")이 NULL을 반환하는 병적 상태가 필요하다 — 좁다. 따라서 reviewer가 붙인 medium 심각도와 'SIGSEGV 회귀' 서사는 도달성 측면에서 과장이다.

그럼에도 '진짜 문제'로 판정하는 이유: 이는 의도된 한계가 아니라 방어적 계약(error-propagation contract)의 내부 비일관이다. 동일 파일 내 6개 동형 함수 중 2개만 가드가 없고, 그 2개조차 reentrancy 분기에선 가드를 갖는다. 수정 비용은 한 줄짜리 `if (!real_X){log; return cudaErrorInitializationError;}` 추가로 사소하며, 미래에 다른 동적링크 구성/심볼 shadowing에서 graceful refuse 대신 크래시할 잠재성이 남는다. 따라서 isReal=true, 단 도달성이 좁아 confidence=medium, 실 심각도는 low에 가깝다.

**수정안**: cudaMalloc 메인 경로(L451 fgpu_init_locked 직후, compute_quota 이전 또는 직후)와 cudaFree 메인 경로(L501 직후)에 driver API 훅과 동일한 NULL 가드를 추가해 계약을 통일하라:

cudaMalloc (L452 부근):
  if (!real_cudaMalloc) {
      fprintf(stderr, "[fgpu] FAIL  cudaMalloc: real symbol not resolved\n");
      pthread_mutex_unlock(&g_lock);
      g_in_hook = 0;
      return cudaErrorInitializationError;
  }

cudaFree (L502 부근, real_cudaFree 호출 전):
  if (!real_cudaFree) {
      fprintf(stderr, "[fgpu] FAIL  cudaFree: real symbol not resolved\n");
      pthread_mutex_unlock(&g_lock);
      g_in_hook = 0;
      return cudaErrorInitializationError;
  }

이렇게 하면 reentrancy 분기(이미 삼항 가드 있음) 및 4개 driver API 훅과 에러처리 계약이 일치하고, dlsym 미해결 시 크래시 대신 표준 에러로 안전 거절한다.

### 4. active 상태 필터가 paused/restarting 컨테이너의 quota 를 잘못 해제함

- **심각도**: 🟠 MEDIUM  ·  **분류**: bug  ·  **검증 confidence**: low
- **위치**: `backend/app/services/admission.py:83`  ·  도메인: admission-scheduler

**문제**: sum_used_ratio 와 usage_snapshot 모두 'active' 를 status in {created, running} 으로만 정의한다(admission.py:83, admission.py:117). 그러나 status 는 docker SDK 의 c.status 가 그대로 reconcile 되어 기록된다(session_manager.py:225 ← docker_manager.py:163, get_status 가 c.status 반환). Docker container status 에는 'paused' 와 'restarting' 이 존재하며, 두 상태 모두 컨테이너/프로세스가 살아 있어 GPU 메모리 할당을 그대로 점유한다(exited/removed 와 달리 quota 가 풀리지 않음). 누군가 docker pause 를 걸거나(외부 운영), 향후 RestartPolicy 가 추가되어 restart 루프에 빠지면, 해당 세션의 ratio 가 admission 합산에서 빠져 같은 device 에 sum(ratios) > 1.0 oversubscription 을 허용하게 된다. 모듈 docstring(admission.py:9-11)의 'stale 세션이 quota 를 잡아두는 것 방지' 의도와도 반대로, 살아있는 세션의 quota 를 놓아버리는 방향의 오류다.

**근거**: admission.py:83 `if r.status not in ("created", "running"): continue` 와 admission.py:117 동일 필터. 한편 docker_manager.py:163 `return c.status, ...` 가 paused/restarting 을 포함한 raw docker status 를 반환하고, session_manager.py:221-225 가 그것을 그대로 store/rec.status 에 기록(정규화 없음 — 코드베이스 전체 grep 결과 status 매핑 레이어 부재).

**검증**: 사실관계는 정확하다. admission.py:83,117 은 활성 필터를 `status in {"created","running"}` 의 화이트리스트로 정의하고, 그 외 모든 status(paused/restarting 포함)를 quota 합산에서 제외한다. docker_manager.py:163 `return c.status, ...` 는 raw docker status 를 그대로 반환하고, session_manager.py:221-225 가 정규화 없이 store/rec.status 에 기록한다(grep 결과 status 매핑 레이어 부재 확인, schema session.py:74 주석도 "docker container status" 라 명시). 따라서 paused/restarting 컨테이너는 GPU 메모리(CUDA context+allocation)를 그대로 점유하지만 ratio 가 합산에서 빠져 같은 device 에서 sum(ratios)>1.0 oversubscription 을 admission 이 허용할 수 있다. 이는 모듈 docstring(admission.py:4,9-11)의 명시적 불변식("sum(ratios)≤1.0", "stale 세션 방지")과 정반대로 *살아있는* 세션의 quota 를 놓아버리는 로직 오류다.

다만 회의적으로 트리거 가능성을 따지면: (1) restarting 은 RestartPolicy 가 있어야 발생하는데 run_kwargs(docker_manager.py:144-153)에 restart_policy 가 없고 코드베이스 전체에 RestartPolicy 부재 — 이 경로는 "향후 추가되면" 의 순수 가설이라 현재 버그 아님. (2) paused 는 시스템이 스스로 만들지 않고 외부 `docker pause` 수동 개입이 있어야 발생. 즉 시스템이 자기 lifecycle 에서 실제로 산출하는 status(created/running/exited/removed)에 대해서는 현재 quota 회계가 정확하다. 그래서 이건 현재 트리거되는 버그라기보다 latent/방어적 결함이다.

그럼에도 isIntendedLimitation=false 로 판정하는 이유: 이 한계는 의도된-한계 목록(협력 위협모델, 하드웨어 격리 없음, 프로토타입 범위 등) 어디에도 없고, 오히려 reconcile(session_manager.py:207-227)가 raw docker 상태를 일부러 읽는 *바로 그 메커니즘*의 회계 정확성을 깨는 방향이다. 올바른 필터는 종료 상태(exited/dead/removed)만 제외하는 블랙리스트여야 하는데, 비종료 상태 2개만 허용하는 화이트리스트라 paused/restarting 같은 살아있는 상태를 잘못 떨어뜨린다. confidence=low 인 것은 현재 시스템 자체 동작으로는 트리거되지 않는 latent 결함이기 때문이지, 오탐이어서가 아니다.

**수정안**: 활성 판정을 화이트리스트가 아니라 종료-상태 블랙리스트로 뒤집는다. admission.py:83,117 의 `if r.status not in ("created", "running"): continue` 를 `if r.status in ("exited", "dead", "removed"): continue` 로 변경하면 paused/restarting 처럼 컨테이너가 살아있어 GPU 메모리를 점유하는 모든 상태가 quota 합산에 포함된다. 더 견고하게는 session_manager.reconcile(또는 docker_manager.get_status) 에 status 정규화 레이어를 추가해 raw docker status 를 {created, running(=running/paused/restarting), terminal(=exited/dead/removed)} 같은 도메인 상태로 매핑하고, admission 은 그 도메인 상태만 신뢰하게 한다. 두 곳(admission.py:77 docstring 포함)을 함께 갱신해 의미를 일치시킨다.

### 5. usage_snapshot 의 per-device ratio_available 이 None('all') 세션을 빼먹어 check() 와 docstring 에 불일치

- **심각도**: 🟠 MEDIUM  ·  **분류**: contract-mismatch  ·  **검증 confidence**: high
- **위치**: `backend/app/services/admission.py:119`  ·  도메인: admission-scheduler

**문제**: gpu_overlaps 규칙상 gpu_index=None 세션은 모든 device 와 overlap 하므로 check() 는 device N 요청 시 None 세션의 ratio 까지 합산한다(admission.py:67-69, 85-87; test_sum_treats_none_as_overlap_with_all 가 0.5(None)+0.3(gpu0)=0.8 로 확인). 그러나 usage_snapshot 은 r.gpu_index 를 그대로 dict 키로 써서 None 세션을 별도 'all' 버킷에만 넣고, per-device 버킷(ratio_used/ratio_available)에는 한 번도 더하지 않는다(admission.py:119). 결과적으로 함수가 리턴하는 device N 의 ratio_available 은 None 세션이 이미 잡아먹은 capacity 를 반영하지 못해 실제 admission 이 허용하는 양보다 크게 표시된다. 예: a(0.4,None)+b(0.3,gpu0)+c(0.2,gpu0) 일 때 snapshot 의 '0' 버킷은 ratio_used=0.5/ratio_available=0.5 로 보고하지만(test_snapshot_groups_by_gpu_with_all_label:155), check(gpu_index=0) 의 실제 used 는 0.4+0.5=0.9 라 0.1 만 남는다. UI/운영자가 이 snapshot 의 available 을 믿고 세션을 만들면 AdmissionDenied 로 거부당해 표시값과 enforcement 가 어긋난다. 게다가 docstring(admission.py:112)은 'None(전체 GPU)은 별도 키로 분리, per-device 합과 합쳐서 보여줌' 이라고 명시하는데, 코드는 '합쳐서' 보여주는 부분을 구현하지 않아 docstring 과도 직접 충돌한다(doc-drift 동반).

**근거**: admission.py:119 `per_gpu[r.gpu_index] = per_gpu.get(r.gpu_index, 0.0) + r.ratio` — None 키를 per-device 키와 분리만 하고 합치지 않음. 대조: admission.py:85 `if not gpu_overlaps(r.gpu_index, gpu_index): continue` 는 None 을 모든 device 합산에 포함. docstring admission.py:112 `None ... per-device 합과 합쳐서 보여줌` 은 미구현.

**검증**: 코드와 테스트로 주장 전부 확인됨. (1) check()는 gpu_overlaps(admission.py:67-69, 85-87)로 인해 device N 검사 시 None 세션 ratio를 합산함 — test_sum_treats_none_as_overlap_with_all(test:80-91)이 0.5(None)+0.3(gpu0)=0.8로 확정. (2) usage_snapshot(admission.py:119) `per_gpu[r.gpu_index] = per_gpu.get(r.gpu_index,0.0)+r.ratio`는 raw gpu_index로 키잉하여 None 세션을 'all' 버킷에만 넣고 per-device 버킷에는 절대 더하지 않음 — test_snapshot_groups_by_gpu_with_all_label(test:143-158)이 a(0.4,None)+b(0.3,gpu0)+c(0.2,gpu0)에서 '0' 버킷 ratio_used=0.5/ratio_available=0.5를 단언. 반면 check(gpu_index=0)의 실제 used=0.4+0.5=0.9라 0.1만 남음 → snapshot의 available이 실제보다 과대 표시. (3) docstring admission.py:112 'None은 별도 키로 분리, per-device 합과 합쳐서 보여줌'에서 '합쳐서 보여줌' 부분 미구현 — 코드는 분리만 함. snapshot은 GET /sessions/admission(sessions.py:98-101)로 UI/운영자에게 노출됨. 이는 실제 계약 불일치+doc-drift이며 의도된 한계 목록(MIG/협력모델/allocator/단일호스트 등)에 해당 안 됨. 단 enforcement(check) 자체는 안전(under-count 없음)하고, 단일 GPU 일반 케이스(모두 None→'all')에서는 자기일관적이라 medium 심각도가 타당 — 위험은 운영자가 과대 available을 믿고 예상치 못한 AdmissionDenied를 받는 표시/enforcement 괴리.

**수정안**: usage_snapshot에서 None 세션 ratio를 per-device 버킷에도 반영하여 check()와 일관성을 맞춤. 두 방법 중 하나: (a) docstring대로 실제로 합치기 — 각 device 버킷 계산 시 None('all') 세션의 ratio를 더해서 ratio_used/ratio_available을 표시(예: device k의 effective_used = per_gpu[k] + per_gpu[None]). 단 device 키가 하나도 없고 None만 있을 때는 device 버킷이 생성되지 않는 점 고려 필요. 또는 (b) sum_used_ratio/check를 진실 소스로 재사용 — 활성 세션에 존재하는 각 distinct gpu_index(None 포함)에 대해 admission.sum_used_ratio(sessions, k)를 호출해 used를 계산하면 overlap 규칙이 자동 일관 적용됨. 어느 쪽이든 docstring(admission.py:112)의 '합쳐서 보여줌'과 코드를 일치시키고, test_snapshot_groups_by_gpu_with_all_label의 기대값을 합산 반영(예: '0' 버킷 ratio_used=0.4+0.5)으로 갱신.

### 6. DB insert 실패 시 이미 기동된 컨테이너가 고아로 누수됨 (롤백 없음)

- **심각도**: 🟠 MEDIUM  ·  **분류**: resource-leak  ·  **검증 confidence**: high
- **위치**: `backend/app/services/session_manager.py:199`  ·  도메인: docker-runtime

**문제**: _create_locked는 create_container로 컨테이너를 실제 기동(147-160)한 뒤 store.insert(199)로 record를 저장한다. insert가 예외를 던지면(SQLite locked/disk full/제약 위반 등) 이미 detach로 떠 있는 GPU 컨테이너를 정리하는 try/except·롤백이 전혀 없다. orphan 정리는 list_all→get이 'DB record→docker' 방향으로만 reconcile하므로(session_manager.py:229-235, 207-219), record가 없는 컨테이너는 영원히 발견되지 않고 GPU/메모리/포트를 점유한 채 남는다. create_container 실패는 API에서 잡히지만(api/sessions.py:93), insert 실패 경로의 컨테이너 누수는 처리되지 않는다. insert 실패 시 remove_container로 보상 정리가 필요하다.

**근거**: session_manager.py:147 `c = await asyncio.to_thread(self.docker.create_container, ...)` 이후 session_manager.py:199 `await asyncio.to_thread(self.store.insert, rec)` 사이에 예외 시 c를 제거하는 코드 부재. reconcile은 session_manager.py:233 `*(self.get(r.id) for r in recs)`로 record 기준만 순회.

**검증**: 코드 확인 결과 제기된 문제는 실제 자원 누수다.

증거:
- session_manager.py:147-160 `create_container`는 docker_manager.py:148/157 `self.client.containers.run(detach=True)`로 컨테이너를 *실제 기동*한다.
- session_manager.py:199 `await asyncio.to_thread(self.store.insert, rec)`가 그 뒤에 온다. 147~199 사이 어디에도 try/except·보상정리(remove_container)가 없다 — 전체 함수 본문에 try 블록 자체가 없음.
- session_store.py:112 insert는 일반 sqlite3 INSERT로, OperationalError(database locked: 동시 쓰기), disk full, IntegrityError(중복 id 제약) 등으로 충분히 throw 가능. 특히 _create_lock(session_manager.py:73)은 asyncio.Lock이라 create 경로끼리만 직렬화할 뿐, 동시 get/list_all의 update_status 쓰기와 SQLite 레벨 경합을 막지 못해 'database is locked'가 현실적으로 발생할 수 있다.
- orphan reconcile은 단방향(DB record→docker)뿐임을 확인: list_all(229-235)이 store.list_all()로 얻은 record들만 self.get으로 순회하고, get(203-227)은 record가 있을 때만 docker.get_status→NotFound→removed 처리를 한다. docker→DB 역방향 스윕(이름 fgpu- prefix로 daemon의 컨테이너를 훑어 record 없는 것 제거)은 docker_manager/session_manager 어디에도 없음(grep으로 list_containers/orphan/prune 부재 확인).
- 따라서 insert 실패로 record가 안 생기면 그 컨테이너는 영원히 발견되지 않고 GPU 슬롯/메모리/포트를 점유. 게다가 admission.check는 DB record 기반 sessions로만 계산(118)하므로 이 orphan은 admission 회계에서도 안 보임 → 회계 정합성까지 깨진다.
- API(sessions.py:93)는 create의 모든 예외를 500으로 변환하지만, 이미 떠버린 컨테이너 정리는 하지 않음.

의도된 한계 목록(MIG/SM 격리, cooperative threat model, allocator caching, context 오버헤드, 단일호스트/HA 미구현 등)은 전부 하드웨어 격리·위협모델·미구현 기능에 관한 것이고, 이번 건은 기동된 컨테이너의 부기 정합성(자원 누수) 문제라 그 어느 항목에도 해당하지 않는다. 따라서 의도된 한계가 아닌 실제 버그.

추가(확신을 약간 낮추는 게 아니라 수정 범위를 넓히는 사항): jupyter 모드에서 create_container(147) 이후 insert(199) 전의 get_host_port(171-173)/asyncio.sleep도 같은 orphan 결과를 낳으므로, 보상정리는 insert만이 아니라 컨테이너 기동 직후부터 감싸야 한다.

**수정안**: _create_locked에서 create_container(147) 직후부터 insert 성공까지를 try로 감싸고, 예외 발생 시 finally/except에서 보상정리: 이미 기동된 c.id를 docker.remove_container(c.id, force=True)로 제거(이 정리 자체의 예외는 swallow). jupyter workspace 디렉토리도 막 생성했으면 정리 고려. 의사코드:

c = await asyncio.to_thread(self.docker.create_container, ...)
try:
    ... host_port/jupyter_url 조립 ...
    rec = Session(...)
    await asyncio.to_thread(self.store.insert, rec)
    return rec
except Exception:
    try:
        await asyncio.to_thread(self.docker.remove_container, c.id, True)
    except Exception:
        pass  # 보상정리 실패는 원래 예외를 덮지 않게
    raise

근본 강건화로는, 단방향 reconcile 한계를 보완해 docker daemon에서 'fgpu-' prefix 컨테이너를 나열해 DB record 없는 것을 주기적/기동시 1회 정리하는 docker→DB 스윕을 추가하면 race로 새는 케이스까지 회수 가능.

### 7. host_port 폴링이 _create_lock 내부에서 최대 1.5s 블로킹 — 모든 세션 생성 직렬 지연

- **심각도**: 🟠 MEDIUM  ·  **분류**: race  ·  **검증 confidence**: high
- **위치**: `backend/app/services/session_manager.py:95`  ·  도메인: jupyter

**문제**: create()는 admission-check와 spawn을 원자화하려 self._create_lock으로 _create_locked 전체를 감싼다(L95-101). 그런데 jupyter 모드의 host_port 폴링 루프(L168-175)는 0.0+0.1+0.2+0.4+0.8 = 최대 1.5s의 await asyncio.sleep을 이 락을 잡은 채 수행한다. 모든 create(batch 포함)가 단일 락에서 직렬화되므로, 포트 publish가 느린 jupyter 세션 하나가 그 동안 다른 모든 POST /sessions를 1.5s까지 블로킹한다. 락은 'admission check ~ store.insert'의 원자성만 필요하며(L69-73 주석), docker.run 직후 컨테이너 ID는 이미 확정되어 capacity 결정에 폴링 결과가 필요 없다. 모듈 docstring의 '진짜 동시 처리 가능'(session_manager.py L10-12) 의도와 직접 충돌. 폴링을 락 밖으로 빼면 해결된다.

**근거**: async with self._create_lock: return await self._create_locked(...) // 내부에서 for delay in (0.0,0.1,0.2,0.4,0.8): if delay: await asyncio.sleep(delay); host_port = ... get_host_port(...)

**검증**: 코드 확인 결과 제기된 문제는 진짜다.

검증:
1. 락 보유 구간: create()(session_manager.py L95-101)가 _create_locked 전체를 async with self._create_lock로 감싼다. jupyter host_port 폴링 루프(L168-175)는 _create_locked 내부, 즉 락을 잡은 채 실행된다. await asyncio.sleep(delay)는 이벤트 루프는 양보하지만 asyncio.Lock은 해제하지 않으므로, 다른 create() 호출(=POST /sessions)은 L95에서 블로킹된다. delay 누적은 0.0+0.1+0.2+0.4+0.8 = 최대 1.5s.

2. 폴링이 capacity 결정에 불필요: admission.check()(admission.py L92-105)는 requested_ratio, gpu_index, 기존 세션 status만 본다. host_port는 capacity와 무관하며 오직 jupyter_url 조립(L177-178)에만 쓰인다. admission 체크는 L116-118에서 docker.run(L147)·폴링(L168) 이전에 끝나고, 컨테이너 ID는 L147 create_container 반환 직후 확정된다. 따라서 폴링은 락이 보호해야 할 "check~insert 원자성"과 무관.

3. 의도와 충돌: 모듈 docstring L10-12 "진짜 동시 처리 가능", 락 주석 L72 "docker.run 자체는 to_thread라 이벤트 루프는 안 막힘"이라고 명시 — 설계 의도는 create 직렬화로 인한 상호 블로킹을 피하는 것. 포트 publish가 느린 jupyter 세션 하나가 그동안 모든 다른 POST /sessions(batch 포함)를 최대 1.5s 직렬 지연시키는 것은 이 의도와 직접 충돌.

이것은 single-host/SQLite/cooperative-threat 같은 의도된 한계 목록에 없는, 실제 동시성/직렬화 결함이다. 다만 1.5s는 워스트케이스(매 재시도마다 get_host_port가 falsy일 때)이고, 포트가 즉시 publish되면 delay=0.0에서 바로 break하여 지연이 거의 없다 — 제기자도 "최대 1.5s"로 정확히 헤지함.

심각도: medium 타당. 정상 경로에선 빠르지만, 부팅 부하/느린 데몬 시 thundering-herd로 모든 세션 생성이 직렬 지연되는 회귀.

**수정안**: host_port 폴링과 jupyter_url 조립을 _create_lock 밖으로 이동. 락은 admission check ~ store.insert 원자성만 필요하므로:
- 옵션 A: 락 안에서는 host_port=None으로 store.insert까지 마치고 락을 해제한 뒤, 락 밖에서 폴링→get_host_port→store.update로 host_port/jupyter_url을 채운다. (Session 스키마/스토어에 host_port·jupyter_url 갱신 경로 필요)
- 옵션 B: create()에서 락 구간을 admission+create_container+insert로 좁히고, 폴링/URL 조립을 락 해제 후 별도 단계로 수행해 반환 직전 rec를 보강.
어느 쪽이든 await asyncio.sleep 백오프가 _create_lock을 잡지 않게 하는 것이 핵심. 컨테이너 ID는 create_container 반환 시 확정되어 admission 원자성에는 폴링 결과가 불필요하므로 안전하다.

### 8. 멀티스레드/멀티스트림에서 비-CAS 윈도우 리셋으로 throttle 누락(under-throttle)

- **심각도**: 🟡 LOW  ·  **분류**: race  ·  **검증 confidence**: low
- **위치**: `/home/jaewoo/Desktop/backend_ai/GpuCluster/hook/src/fgpu_hook.c:784`  ·  도메인: hook-throttle

**문제**: g_window_start_ns 는 RELAXED load/store 만 쓰고 CAS 가 없다. 두 스레드(또는 두 CUDA 스트림에서 launch 하는 워커 스레드)가 거의 동시에 launch 하면: (1) 둘 다 같은 ws 를 읽고 둘 다 sleep 후 각자 g_window_start_ns 를 post-sleep now 로 덮어쓴다 → 한 스레드의 리셋이 다른 스레드가 막 시작한 윈도우를 무효화한다. 그 결과 일부 스레드는 매 launch 마다 elapsed≈0 인 '새 윈도우'를 보고 active_limit 검사를 통과해 sleep 을 건너뛴다. 집계 throughput 이 compute_ratio 보다 높게 새어나가는 under-throttle 가 발생한다. 코드 주석(184-187행)은 '둘 다 now 로 설정하므로 무해'라고 데이터 손상만 다루지, throttle 정확도 손실은 다루지 않는다. 검증된 타깃(test_throttle.cu 는 단일 호스트 스레드 tight loop)에서는 드러나지 않으나, PyTorch DataLoader/멀티스트림 워크로드에서는 비율 보장이 깨진다. 단일 스트림 협력 모델이 의도라 severity 는 low 로 두되, 주석이 정확도 측면을 보장하는 것처럼 읽히는 점이 미문서화 회귀 소지.

**근거**: line 784: int64_t ws = __atomic_load_n(&g_window_start_ns, __ATOMIC_RELAXED); ... line 792/807: __atomic_store_n(&g_window_start_ns, now_ns, __ATOMIC_RELAXED) — read-modify-write 가 CAS 가 아니라 분리된 load 후 store. 주석 184-187: 'atomic load/store. 두 스레드가 동시에 리셋해도 둘 다 "지금" 으로 설정하므로 무해.'

**검증**: 코드를 직접 확인한 결과 리뷰어의 기술적 진단은 전부 정확하다. fgpu_hook.c:784는 RELAXED load, 786/792/807은 RELAXED store로 read-modify-write가 CAS 없이 분리되어 있다. 결정적으로 재진입 가드 g_in_hook은 line 154에서 __thread(per-thread)로 선언돼 있어 스레드 간 직렬화를 전혀 제공하지 않는다 — 따라서 두 워커 스레드/스트림이 throttle 블록(779-816)에 동시에 진입 가능하다. 이 경우 한 스레드의 post-sleep 윈도우 리셋(807행)이 다른 스레드가 막 시작한 윈도우를 now로 덮어써 elapsed≈0을 만들고, 그 스레드는 active_limit 검사(797)를 통과해 sleep을 건너뛴다 → 집계 throughput이 compute_ratio를 초과하는 under-throttle 누수가 발생한다. 검증 타깃 test_throttle.cu(60-62행)는 단일 호스트 스레드 tight loop라 이 누수가 드러나지 않는다. 주석 184-187은 '둘 다 지금으로 설정하므로 무해'라며 데이터 손상만 다루고 throttle 정확도 측면은 보장하지 않는데, 정확도를 보장하는 것처럼 읽힐 소지가 있다.

이것이 의도된 한계가 아닌 이유: description.md(stage12 한계 목록)와 CLAUDE.md, 그리고 제시된 의도된-한계 목록 어디에도 '멀티스트림/멀티스레드에서 비율 보장이 깨진다'는 항목이 없다. 문서화된 한계는 (a) work-conserving 아님, (b) 커널 실행시간 미고려, (c) nanosleep ~50us 정밀도, (d) cooperative(우회 가능)뿐이다. 멀티스레드 윈도우 경합으로 인한 정확도 손실은 이 중 어디에도 해당하지 않으며, throttle 비례성 정확도 측정이 곧 논문 기여라고 명시(description.md)돼 있어 미문서화 회귀에 해당한다.

다만 severity는 low이고 confidence를 low로 둔다: 명시된 타깃인 PyTorch는 단일 호스트/단일 프로세스이고 기본 스트림 하나로 한 호스트 스레드에서 launch하는 경우가 지배적이라 실제 누수 폭은 작다. 리뷰어 본인도 low로 평가했고, cooperative 단일-스트림 모델이 의도라는 점을 인정했다.

**수정안**: 두 갈래로 처리 권장. (1) 최소 비용/즉시: 주석 184-187을 수정해 'data-corruption은 없으나 멀티스레드/멀티스트림 동시 launch 시 윈도우 리셋 경합으로 throttle 비율이 compute_ratio보다 높게 새어나갈 수 있다(single-stream 협력 모델 가정)'를 명시 → 미문서화 회귀 제거. description.md의 stage12 한계 목록에도 동일 항목 추가. (2) 정확도 강화(선택): window 리셋을 CAS로 전환. 예: ws==0/만료 시 __atomic_compare_exchange_n(&g_window_start_ns, &ws, now_ns, ...)로 한 스레드만 리셋 성공하게 하고, post-sleep 리셋(807)도 'sleep 직전 읽은 ws에서만 갱신'하는 CAS로 바꿔 다른 스레드의 새 윈도우를 덮어쓰지 않게 한다. 더 견고하게는 윈도우 경계를 (now_ns / window_ns)로 양자화(quantize)해 모든 스레드가 동일한 윈도우 epoch를 공유하도록 하면 리셋 경합 자체가 사라진다.

## 부록 — 기각된 발견 (의도된 한계 / 오탐)

아래는 리뷰어가 제기했으나 검증 단계에서 '이 프로젝트의 의도된 한계 또는 오탐'으로 판정한 것들.

**1. pop_alloc의 '못 찾음'과 '실제 size 0' 반환값이 구분 불가 + freed>0 가드로 회계 누수 가능**

- 위치: `hook/src/fgpu_hook.c:269`
- 기각 사유: 코드는 사실관계상 리뷰어 주장과 일치한다. fgpu_hook.c:466-468에서 cudaMalloc 성공 시 g_used += size 후 track_alloc()을 부르는데, track_alloc(238-245)의 malloc이 실패하면 노드를 안 만들고 return한다. 그러면 나중에 pop_alloc(257-270)이 ptr을 못 찾아 0을 반환하고, 세 free 경로 모두 `if (freed > 0 && freed <= g_used)` 가드(506, 608, 714) 때문에 g_used를 차감하지 못해 g_used가 단조 증가한다. 메커니즘 자체는 실재한다.

그러나 '진짜 버그'가 아니라 명시적으로 문서화·수용된 의도된 한계다:
1) 라인 241-244 주석이 이 정확한 시나리오를 글자 그대로 인정한다: "malloc 실패는 사실상 메모리 고갈... 그러면 cudaFree 시 size 를 못 찾아 g_used 가 누적된다. Stage 1 에서는 알고도 받아들이는 한계로 둔다." 이건 미문서화 회귀가 아니라 코드 주석에 박힌 알려진 제약이다.
2) 트리거 조건이 host malloc(~24바이트 alloc_entry_t) 실패, 즉 호스트 메모리 고갈이다. 그 시점이면 프로세스 전체가 이미 붕괴 중이라 실무적 영향이 사실상 없다.
3) 리뷰어가 '미문서화'라고 주장하는 핵심 — driver/VMM free 경로(608, 714)까지 동일 전파 — 은 별개 결함이 아니다. 세 경로가 공유하는 단일 track_alloc/pop_alloc/g_used 회계를 설계상 그대로 쓰기 때문에 같은 근본원인 하나가 그대로 적용될 뿐이다. '여러 경로로 전파'는 새 버그가 아니라 동일 한계의 자연스러운 귀결이다.
4) `freed > 0` 가드와 pop_alloc의 not-found→0 반환은 그 자체로 버그가 아니라 올바른 방어다. hook 늦게 붙어 추적 못 한 ptr(라인 254, 492 문서화)을 free할 때 날조된 size를 빼면 회계가 더 망가진다. not-found와 size-0을 모두 '회수할 기록된 size 없음'으로 묶는 건 의도적이고 타당하다.

리뷰어 본인도 결론을 '의도된 한계에 가깝지만'으로 적어 내 판정과 수렴한다. 따라서 실제 버그가 아니라 의도된(그리고 주석으로 문서화된) Stage 1 한계로 판정한다.
