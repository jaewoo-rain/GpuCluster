- **결론(H1 확정)**: 둘 다 GPU를 포화시키니 공유해도 처리량을 쪼개 가질 뿐, 공짜 이득 없음.

## 2. H2 가설

> 한쪽이 **메모리-heavy / idle-heavy(연산 적음)**, 다른 쪽이 **컴퓨트-heavy**면,
> 한쪽이 GPU를 놀리는 틈을 다른 쪽이 메워 **점유율 합 < 100%** →
> **공유 이득(speedup > 1 또는 compute 잡의 처리량이 solo에 근접)** 이 나온다.

H1과 같은 틀(단일 RTX 4070 12GB, Jupyter 세션 2개, 노트북 내 파이썬 측정)을 유지한 채
**워크로드 비대칭**만 도입해 이를 실측으로 보인다.

## 3. 상보 워크로드 설계 (후보·트레이드오프·권장안)

### 3.1 후보 비교

| 후보 | idle-heavy 잡 구성 | 장점 | 단점 |
|---|---|---|---|
| (a) 컴퓨트-heavy | 기존 LLM generate (Qwen2-0.5B) | H1과 동일, 검증됨 | — (이건 A 쪽 그대로 유지) |
| (b1) LLM 긴컨텍스트 + sleep 섞인 간헐 generate | generate 사이에 sleep | "LLM 두 개" 스토리 일관 | generate 자체가 메모리-bound·가변 길이 → **idle fraction 예측 불가, 재현성 낮음** |
| (b2) **큰 텐서 상주 + 주기적 matmul burst + sleep** | `torch.zeros(~ratio×12GB)` 상주 + iter마다 4096² matmul 1회 + `sleep` | **idle fraction이 실험 파라미터로 고정**, 메모리/compute 분리 명확, gpu_util 타임라인에 burst/sleep 패턴 뚜렷, A의 tps 지표 그대로 유지 | matmul은 LLM과 SM 경합 패턴 다름(한계로 문서화) |

### 3.2 권장안 (확정)

- **A (compute-heavy)**: 기존 LLM generate 그대로. `MODEL_ID=Qwen/Qwen2-0.5B-Instruct`,
  greedy, `MAX_NEW_TOKENS=128`, `N_ITERS=20`, **`ratio≈0.5`**.
- **B (idle-heavy)**: **후보 b2**. 큰 텐서(`IDLE_TENSOR_MIB≈4096`, ~ratio×12GB의 안전 범위) 상주 +
  각 iter `4096×4096 matmul 1회 + synchronize` (= compute burst) + iter **밖**에서
  `time.sleep(IDLE_SLEEP_S)`. **`ratio≈0.5`**, **idle fraction ≈ 80%** 목표.
- 두 ratio 합 ≈ 1.0 (admission 통과). 메모리 quota는 양쪽 모두 정상 강제됨.

**왜 b2인가**: H2의 핵심 주장("B가 GPU를 놀리는 틈")이 측정된 파라미터(`IDLE_SLEEP_S`)와 일치해
재현 가능하고, gpu_util 타임라인에서 burst/sleep가 명확히 대비된다. b1은 generate 길이가 가변이라
idle fraction을 사후 해석하기 어렵다.

### 3.3 시간축 엇갈림 보장 (H1 함정 회피의 열쇠)

- **sleep을 throttle window/커널 시간 척도보다 훨씬 크게**: `IDLE_SLEEP_S = 5~10s`.
  → B의 idle 구간이 충분히 길어 A의 여러 launch/iter를 포함. 두 잡의 위상이 어떻게 맞든
  B sleep 구간이 A의 활성 구간을 덮는다.
- **배리어 동기화는 쓰지 않는다.** 대신 **수동 위상차**: A를 먼저 Run All(warmup 3회 ≈ 30s 동안
  B는 대기), A 본 측정 시작 직후 B를 Run All. → A 본 측정 20회 전체 구간에서 B가 상주.
  makespan은 사후 `iter_start/end_epoch`로 정밀 계산하므로 시작 skew 무관.
  (컨테이너 간 IPC 배리어는 협력적 위협 모델 밖 + workspace 분리로 복잡 → 회피.)
- **사전 검증**: B를 **solo로 먼저 1회** 돌려 `gpu_util_timeline`에서 sleep 구간 util이
  명확히 0 근처로 떨어지는지 확인. 안 떨어지면 `IDLE_SLEEP_S` 늘리거나 burst 줄임.

## 4. throttle 처리 (확정: T1 = OFF)

### 4.1 결정

**H2는 throttle을 끈다.** 세션 생성 시 `compute_ratio` 파라미터를 **생략** →
`docker_manager`가 `FGPU_THROTTLE_ENABLE`를 주입하지 않음 → hook 기본값 0(OFF).
**메모리 quota(`FGPU_RATIO`)만 강제.** (코드 변경 0 — 운영 파라미터로 달성.)

### 4.2 근거 (gpu-throttle-perf-engineer)

- **T2(throttle ON)는 H2 이득을 구조적으로 가린다.** ON이면 compute-heavy 잡 B가 윈도우의
  `compute_ratio`만 launch 통과, 나머지는 nanosleep으로 묶임. A가 SM을 비우는 구간과 B의
  throttle-OFF 구간이 일치한다는 보장이 없어 "SM이 비었는데 아무도 못 쓰는" 시간 발생 →
  speedup ≤ 1.0으로 떨어질 수 있다.
- **T1(OFF)**: throttle 개입 제거 → "CUDA 드라이버 자연 time-slicing이 idle 틈을
  work-conserving하게 메우는가"라는 **순수 하드웨어 공유 관찰**. 이득이 관측되면 전적으로
  time-sharing 효과로 귀속. 메모리 quota는 유지되므로 "메모리 fractional 시행"은 충족.
- **T3(비대칭)는 실험 오염** — 변수 2개(throttle + workload)가 섞여 인과 추론 불가.
- **idle-heavy 잡엔 throttle이 어차피 무의미**: throttle은 윈도우 내 wall-clock elapsed가
  `active_limit` 초과 시에만 발동. idle 잡은 window당 launch가 적어 발동 조건 자체가 거의 성립 안 함.

### 4.3 H1과의 관계 (논문 서술)

> H1은 throttle **ON**(메모리 quota + compute duty-cycle), H2는 throttle **OFF**(메모리 quota만).
> 두 실험은 **서로 다른 축**을 측정한다. H2 speedup은 드라이버 자연 time-slicing의 공유 이득이다.

### 4.4 drain 구간 재해석

T1에서는 B가 먼저 끝나 A가 혼자 도는 drain 구간의 tps 상승이 "throttle 해제"가 아니라
**"SM 경합자 소멸"** = 물리적으로 자연스러운 관찰. 단 **speedup 계산은 overlap 구간 기준**으로만
하고 drain을 제외(포함 시 speedup 과대).

## 5. 메트릭 · 성공 기준

### 5.1 분석 프레임 (H1과 동일 재사용)

`iter_start_epoch`/`iter_end_epoch`(epoch=CLOCK_REALTIME, 컨테이너 간 직접 비교 가능)로
**solo / overlap / drain 3구간** 분리 + **speedup** + **점유율 합**. 직전 overlap 계획
(`2026-06-11_16-34`)의 `_overlap.py`(미구현) 또는 분석 노트북 셀로 처리.

### 5.2 핵심 지표

| 지표 | 정의 | H2 기대 |
|---|---|---|
| `overlap_tps_A / solo_tps_A` | A가 B와 공존 시 단독 대비 처리량 유지율 | ≥ 0.80 (B idle 충분 시 0.90+) |
| `speedup` | 순차 makespan / 동시 makespan (**overlap 구간 기준**) | ≥ 1.1~1.2 |
| `overlap 평균 gpu_util` (둘 합산, B sleep 구간 한정 A util) | nvidia-smi/pynvml 폴링 | < 95% (여유 용량 증거) |
| B `cycle_time` (concurrent vs solo) | idle 잡 처리량 대용(토큰 없음) | 차이 < 20% (A 간섭 미미) |

- **idle-heavy 잡(B)의 처리량 지표**: `tokens_per_s`는 0(생성 없음). 대신 **cycle time**
  (sleep+burst 1사이클 길이)을 기록해 solo 대비 변화로 간섭을 본다.

### 5.3 성공 기준 (보수적 단계화 — 확정)

- **1순위 (worst-case 보호)**: `overlap_tps_A / solo_tps_A ≥ 0.80`.
  "quota-only 공존이 compute 잡 처리량을 크게 해치지 않는다" — B 행동 분산에 덜 의존해 재현성 높음.
- **2순위 (강한 주장)**: `speedup ≥ 1.2` (overlap 기준). 마진이 얇으니 1순위가 메인.
  현실적 상한: consumer RTX 4070 + MPS 없음 + `PYTORCH_NO_CUDA_MEMORY_CACHING=1`(매 step
  cudaMalloc/Free로 A의 연속 SM 점유가 끊겨 time-slicing 빈틈 메우기에 다소 불리) →
  실측 1.05~1.4 예상. **PASS 컷은 `speedup ≥ 1.1` 권장**(1.2는 stretch goal로 표기).
- **3순위 (여유 용량 직접 증거)**: B sleep 구간의 A `gpu_util`이 A solo util의 90% 이상 유지 +
  overlap 평균 합산 util < 95%.

**판정**: 1순위 충족 필수. 1+2 또는 1+3 충족 시 "H2 공유 이득 관측"으로 보고.
speedup만 단독 판정 금지(B iter 수 설계로 임의 조정 가능하므로).

### 5.4 H1 대비 표/그래프

H1과 동일 포맷으로 (a) overlap_tps_A 유지율 막대, (b) speedup 비교(H1 1.00 vs H2),
(c) gpu_util 타임라인(H1 둘 다 포화 vs H2 B sleep 골 + A 채움), (d) 점유율 합(99% vs <100%).

## 부록 — 정직성 주석 (논문 신뢰성)

**H2가 증명하는 것**: throttle OFF·메모리 quota만인 상태에서 idle-heavy 잡과 compute-heavy 잡을
공존시키면, idle 구간 동안 compute 잡의 SM 경합이 줄어 처리량이 solo에 근접하고 makespan 효율이
개선될 수 있다(드라이버 자연 time-slicing).

**증명하지 못하는 것**: SM 격리(X) / throttle·SM경합 완전 분리(burst가 겹치면 비선형) /
일반화(특정 `IDLE_SLEEP_S·burst`에서만 — burst 길어지면 H1 포화로 수렴) /
협력적 위협 모델 전제(B가 sleep 중 메모리 해제 안 함) /
matmul burst ≠ LLM SM 패턴 / consumer GPU 컨텍스트 스위치 오버헤드로 speedup 상한 제약.
