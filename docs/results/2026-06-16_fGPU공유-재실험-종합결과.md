# fGPU 공유 재실험 — 종합 결과

**측정일**: 2026-06-16 / **환경**: RTX 4070 12GB (WSL2 Ubuntu, Docker), Qwen2-0.5B/1.5B-Instruct, caching ON
**설계 문서**: [docs/plan/2026-06-15_17-15_재실험-오버헤드-latency-정직재측정.md](../plan/2026-06-15_17-15_재실험-오버헤드-latency-정직재측정.md)

---

## 0. TL;DR

> **fGPU 분수 공유의 이득은 "GPU가 얼마나 비었나(occupancy)"가 결정한다.**
> - GPU가 한가하면(작은 모델/작은 배치) 공유가 순차 실행보다 **1.2~1.45배 빠르다**.
> - 워크로드가 무거워질수록 occupancy가 1.0에 가까워지고 이득은 1.0으로 수렴한다(= H1 "포화→이득 0").
> - **코어 시분할(throttle)** 은 4:6 같은 비율 분배를 **안정적으로 보장**하지만, 그 대가로 처리량의 **40~50%를 희생**한다(work-conserving 아님).

H1("둘 다 포화 → speedup 1.00, 이득 없음")과 본 결과는 **모순이 아니라 같은 곡선의 양 끝**이다. H1은 caching-off 인공물로 GPU가 포화돼 보였고, 본 실험은 현실적(caching-on) 추론에서 작은 모델이 GPU를 다 못 채워 이득이 드러난 경우다.

## 1. 지난 H1 대비 바로잡은 것 (방법론)

| 문제 | 수정 |
|---|---|
| 분석이 20-iter 전체 평균 사용 → drain 오염 | **overlap 구간만** 잘라 per-job 지표 산출 (`_overlap.py`, 단위테스트 12개) |
| `speedup≥1.2=PASS` 단일 판정 (throughput 편향) | **2축 보고**: makespan 오버헤드(①) + 요청당 latency(②) |
| caching-off로 latency 5~30배 왜곡 (인공물) | **caching ON** (현실적 추론). 단 이미지에 박힌 `PYTORCH_NO_CUDA_MEMORY_CACHING=1` 을 `import torch` 전에 제거해야 실제로 켜짐 |
| solo 도 throttle 켜서 기준선 왜곡(occupancy 2.66 등 비물리적) | **solo=throttle OFF(원래 속도) / concurrent=throttle ON(분배)** 의 정석 설계 (`THROTTLE_MODE=conc`) |
| 브라우저 Jupyter 수동 동시 실행(재현 불가) | **헤드리스 측정 스크립트 + 컨테이너 2개 동시 기동** (`test_infer_measure.py`, `run_sharing.sh`) |

## 2. 실험 설계

- **시나리오 3종**: solo A / solo B / concurrent A+B. 각 run = 같은 횟수(NITERS) 반복 측정.
- **정책(기본)**: solo 는 throttle OFF(= 혼자면 GPU 전부 사용 = "원래 작업 시간"), concurrent 만 throttle ON 으로 비율 분배.
- **지표** (검증된 `_overlap.analyze_concurrent`):
  - `speedup = makespan_seq / makespan_conc` (따로 합 vs 같이; >1 = 공유가 빠름)
  - `overhead_seq = makespan_conc − makespan_seq` (초; + = 손해)
  - `occupancy_sum = overlap_tps_A/solo_tps_A + overlap_tps_B/solo_tps_B` (~1=포화, <1=throttle 낭비, >1=여유 회수)
  - `a_share` = overlap 구간 A 컴퓨트 점유율 (목표 = ratio_A)
  - `lat_x` = 요청당 latency 가 solo 대비 느려진 배수
- **반복**: 각 조합 3회 → 평균±편차.

## 3. 결과

### Stage 1 — 포화 스윕 (Qwen2-0.5B, ratio 0.4/0.6, n=3)

| batch | throttle | speedup | overhead(s) | occupancy | a_share(목표0.40) | lat_x A / B |
|---|---|---|---|---|---|---|
| 1 | **off** | **1.445**±.046 | −13.0 | 1.345 | 0.515 | 1.38 / 1.39 |
| 8 | **off** | **1.336**±.007 | −10.6 | 1.236 | 0.470 | 1.50 / 1.49 |
| 16 | **off** | **1.331**±.015 | −10.6 | 1.239 | 0.485 | 1.51 / 1.50 |
| 1 | conc(4:6) | 0.465±.018 | +47.1 | 0.666 | **0.373** | 3.60 / 2.31 |
| 8 | conc(4:6) | 0.478±.033 | +46.3 | 0.733 | **0.381** | 3.43 / 2.12 |
| 16 | conc(4:6) | 0.557±.025 | +34.4 | 0.832 | **0.369** | 2.80 / 1.82 |

- throttle OFF: 배치 1~16 전부 speedup>1 (1.33~1.45). 이득이 견고하나 배치↑ 시 약간 감소(occupancy 1.35→1.24).
- throttle ON: 항상 speedup<1, 단 배치↑ 시 낭비 감소(occupancy 0.67→0.83).
- **4:6 분배는 배치와 무관하게 견고** (a_share 0.37±0.03).

### Stage 2 — 분배 정확도 (Qwen2-0.5B, batch1, throttle conc, n=3)

| 목표 A 비율 | 실측 a_share | 오차 | overhead(s) | occupancy |
|---|---|---|---|---|
| 0.30 | 0.231±.008 | −0.07 | +87.6 | 0.687 |
| 0.40 | 0.373±.032 | −0.03 | +47.1 | 0.666 |
| 0.50 | 0.515±.021 | +0.02 | +16.7 | 0.692 |

- 방향·순서는 정확(단조), **50:50에서 가장 정확**.
- 치우친 비율일수록 약자(A)가 명목보다 덜 받고(−0.07), 총 손해(overhead)도 커짐(+17→+88s).

### Stage 3 — 모델 크기 (Qwen2-1.5B, batch1, ratio 0.4/0.6, n=3)

| 모델 | throttle | speedup | occupancy | a_share | lat_x A/B |
|---|---|---|---|---|---|
| 0.5B | off | 1.445 | 1.345 | 0.515 | 1.38/1.39 |
| **1.5B** | **off** | **1.193**±.066 | **1.094**±.039 | 0.515 | 1.69/1.75 |
| 1.5B | conc(4:6) | 0.533±.018 | 0.732 | 0.375 | 3.61/2.00 |

- 큰 모델 → occupancy 1.35→1.09(GPU 더 참) → 이득 1.45→1.19로 축소. **포화 가설 직접 입증.**

## 4. 종합 해석 — occupancy가 이득을 예측

| 설정 (throttle off) | occupancy | speedup |
|---|---|---|
| 0.5B · batch1 | 1.345 | 1.445 |
| 0.5B · batch16 | 1.239 | 1.331 |
| 1.5B · batch1 | 1.094 | 1.193 |
| (외삽: 7B / 큰 배치) | →1.0 | →1.0 (= H1) |

워크로드가 무거워질수록 occupancy↓ → speedup↓. **"언제 공유가 이득인가" = "GPU에 여유 용량이 있을 때".** 단일/소배치 LLM decode 는 메모리-bound 라 현대 GPU 를 다 못 채우므로 분수 공유에 이득 여지가 있다.

---

## 5. throttle(코어 시분할) 결론 — 공정성 vs 처리량 trade-off

- **항상 처리량 손해**: speedup 0.47~0.56 (순차보다 느림). 배치↑ 시 손해 감소.
- **비율 분배는 안정적**: a_share 가 목표 비율을 추종(모델·배치 무관, 50:50 최정확).
- **원인**: throttle 은 per-process duty-cycle 이라 A·B 의 sleep 구간이 겹치면 GPU 가 idle(non-work-conserving). occupancy<1 이 그 낭비의 직접 증거.
- **시스템 한계**: hook 은 "지금 나 혼자인지" 를 모름 → 경쟁자가 없어도 항상 비율로 자신을 묶음. 더 나은 설계는 "경쟁 시에만 throttle"(work-conserving) 이어야 함.

---

## 6. 정직한 한계

- decode 는 메모리-bound → batch16/1.5B 로도 완전 포화는 미달. **완전 포화(speedup→1.0) 케이스는 외삽** (7B/대배치로 직접 확인 필요).
- 단일 GPU(RTX 4070), 단일 모델군(Qwen2), 단일 프롬프트.
- throttle-solo 의 다운클럭 영향은 분리하지 않음(그래서 solo 기준선은 throttle OFF 로 측정).
- 두 컨테이너 간 동시 시작은 수동 위상차(거의 동시 기동). overlap 구간 분석으로 skew 흡수.

## 7. 재현 방법 (리눅스 GPU 호스트)

```bash
# 이미지: base → pytorch(jupyter) → +transformers 파생
./scripts/build_image.sh
./scripts/build_pytorch_image.sh
docker build -t fgpu-runtime-pytorch:stage4-infer - <<'EOF'
FROM fgpu-runtime-pytorch:stage4
RUN pip install --no-cache-dir numpy transformers accelerate sentencepiece
EOF
# (TODO: 위 transformers 설치를 runtime-image-pytorch/Dockerfile 로 내재화)

# 단일 측정 (solo off / concurrent throttle 4:6)
IMAGE=fgpu-runtime-pytorch:stage4-infer THROTTLE_MODE=conc NITERS=30 \
  bash scripts/eval/run_sharing.sh

# 포화 스윕 (배치 × throttle × 반복) + 평균±편차 집계
IMAGE=fgpu-runtime-pytorch:stage4-infer \
  NITERS=12 BATCHES='1 8 16' THROTTLES='off conc' REPS=3 \
  bash scripts/eval/run_sweep.sh
# → experiments/sweep_<TS>/summary.txt , sweep_summary.csv
```
