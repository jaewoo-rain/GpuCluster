## 1. 변수와 자료형 — 값에도 "종류"가 있어요

### 왜 필요한가

Python에서는 `x = 5` 하면 끝입니다. 파이썬이 알아서 "아, 정수네" 하고 처리해줘요. 그런데 C는 **변수를 만들 때 "이건 무슨 종류의 값이야"라고 미리 알려줘야** 합니다. 이걸 "자료형(type)"이라고 해요.

왜 이렇게 번거롭게 할까요? C는 컴퓨터의 메모리를 아주 가깝게 다루는 언어라서, "이 값이 메모리에서 몇 바이트를 차지하는지" 정확히 알아야 하기 때문입니다. GPU 메모리를 바이트 단위로 세는 이 프로젝트에서는 이게 특히 중요해요.

### 일상 비유

택배 상자를 생각해보세요. "작은 봉투(엽서용)", "중간 상자(책용)", "냉장 박스(음식용)"처럼 **담을 내용물의 종류에 따라 상자 종류가 다르죠.** C의 자료형도 똑같습니다. 담을 값의 종류에 맞는 "상자"를 골라야 합니다.

### 아주 작은 코드 예시

```c
int    age    = 30;        // 정수 (integer). 보통 4바이트.
double ratio  = 0.4;       // 소수(실수). 8바이트. 파이썬의 float.
size_t bytes  = 268435456; // "크기"를 담는 특별한 정수형 (256MB를 바이트로).
void  *ptr;                // 아무 종류나 가리킬 수 있는 "주소". (뒤에서 자세히!)
```

각 자료형을 하나씩 볼게요.

- **`int`**: 그냥 정수예요. `-3`, `0`, `42` 같은. 나이, 개수, 에러 코드 등에 씁니다.
- **`double`**: 소수점이 있는 실수. `0.4`, `3.14` 같은. 이 프로젝트에서 **비율(ratio)**을 담는 데 씁니다. (GPU의 40%를 쓰겠다 → `0.4`)
- **`void`**: "값이 없음" 또는 "타입을 정하지 않음"이라는 특수한 표시. 그 자체로 변수를 만들진 않고, 뒤에 나올 `void*`나 함수 반환값에 씁니다.

### size_t가 왜 메모리 크기에 쓰이나요? (중요)

`size_t`는 **"메모리 크기나 개수를 담기 위해 특별히 만든 정수형"**입니다. 그냥 `int`를 쓰면 안 될까요?

문제는 `int`는 담을 수 있는 최대값이 약 21억(약 2GB)까지라는 점입니다. 그런데 GPU 메모리는 8GB가 넘어가죠. 8GB를 바이트로 세면 약 85억 바이트인데, `int`에는 안 들어갑니다! **그래서 절대 음수가 될 수 없고, 아주 큰 값(그 시스템에서 가능한 최대 메모리 크기)까지 담을 수 있는 `size_t`**를 씁니다.

> 규칙처럼 외우세요: **"메모리 바이트 수를 세면 `size_t`."**

### 이 프로젝트 실제 코드에서

GPU 메모리 사용량과 상한선을 담는 변수들이 전부 `size_t`입니다:

- [fgpu_hook.c:128](../../hook/src/fgpu_hook.c#L128) — `static size_t g_used = 0;` (현재까지 쓴 GPU 메모리, 바이트)
- [fgpu_hook.c:129](../../hook/src/fgpu_hook.c#L129) — `static size_t g_quota = 0;` (쓸 수 있는 상한선, 바이트)
- [fgpu_hook.c:130](../../hook/src/fgpu_hook.c#L130) — `static double g_ratio = 1.0;` (비율은 소수라서 `double`)

`g_used`와 `g_quota`는 "몇 바이트"라서 `size_t`, `g_ratio`는 "0.4 같은 비율"이라서 `double`. 자료형이 왜 이렇게 나뉘었는지 이제 보이시죠?

### 초보가 흔히 하는 오해/함정

- **"`int`면 다 되겠지"** → GPU 메모리처럼 큰 값에서는 `int`가 넘쳐버립니다(overflow). 크기·개수는 `size_t`.
- **"`double`이 더 정밀하니까 다 `double`로 쓰자"** → 아닙니다. 개수를 세는 데 `double`을 쓰면 미세한 오차가 생길 수 있어요. 정수는 정수형, 소수는 소수형.

### 한 줄 요약

> C는 값마다 "상자 종류(자료형)"를 정해야 하고, GPU 메모리 바이트 수는 아주 커서 `size_t`, 비율은 소수라 `double`을 씁니다.

## 2. 포인터 — 값이 아니라 "주소"를 다루기

포인터는 C의 가장 큰 진입장벽입니다. 하지만 하나의 그림만 머리에 넣으면 끝나요. 천천히 갑시다.

### 왜 필요한가

이 프로젝트의 후킹 함수 `cudaMalloc`은 **"GPU에 메모리를 잡아서, 그 위치를 사용자에게 알려주는"** 일을 합니다. 그런데 "위치를 알려준다"는 게 바로 포인터입니다. 포인터를 모르면 이 프로젝트의 핵심 함수 시그니처조차 읽을 수 없어요.

### 일상 비유 (이 그림 하나면 됩니다)

메모리를 **거대한 아파트 단지**라고 생각하세요. 각 집(메모리 칸)에는 **번호(주소)**가 붙어 있고, 그 안에 **물건(값)**이 들어 있습니다.

```
  주소(집 번호)      내용물(값)
  ┌─────────────┐
  │  1004호      │  →  [ 30 ]      ← age 라는 변수가 여기 살아요
  ├─────────────┤
  │  2008호      │  →  [ 1004 ]    ← "1004호를 가리키는 포인터"가 여기 살아요
  └─────────────┘
```

- **일반 변수** `int age = 30;` → 1004호에 숫자 `30`을 넣은 것.
- **포인터** → "값" 대신 **"다른 집의 주소(집 번호)"**를 담는 변수. 위 그림의 2008호는 값으로 `1004`(다른 집의 번호)를 들고 있어요.

즉 포인터는 **"물건을 직접 들고 있는 게 아니라, 물건이 있는 집 주소를 적은 쪽지"**입니다.

### 기호 두 개: `&` 와 `*`

- **`&age`** → "age의 **주소**를 줘." (`&` = address-of, 주소 연산자). 위 예에서 `&age`는 `1004`.
- **`*ptr`** → "ptr이 가리키는 **집에 가서 그 안의 물건을** 꺼내와." (`*` = 역참조, dereference). ptr이 1004를 담고 있으면 `*ptr`은 `30`.

```c
int  age = 30;
int *ptr = &age;   // ptr 은 age 의 주소(1004)를 담는다.

printf("%d\n", age);    // 30  (값 직접)
printf("%d\n", *ptr);   // 30  (주소 따라가서 꺼낸 값)

*ptr = 99;              // "ptr 이 가리키는 집의 물건을 99 로 바꿔라"
printf("%d\n", age);    // 99  ← age 가 바뀌었다! 같은 집이니까.
```

마지막 부분이 포인터의 진짜 힘입니다. `*ptr = 99`를 했더니 `age`가 바뀌었죠? **ptr과 age가 같은 집(1004호)을 가리키고 있으니까요.** 이게 "함수 밖의 변수를 함수 안에서 바꾸는" 트릭의 핵심입니다.

### 변수 선언의 `*`는 "이건 포인터야"라는 표시

초보가 제일 헷갈리는 부분입니다. `*`가 두 가지 상황에서 쓰여요:

1. **선언할 때** `int *ptr;` → "ptr은 int를 가리키는 **포인터다**"라는 **종류 표시**.
2. **사용할 때** `*ptr` → "그 주소로 가서 **값을 꺼내라**"라는 **동작**.

같은 별표지만 위치에 따라 의미가 다릅니다. 선언에서는 "타입", 사용에서는 "따라가기"라고 기억하세요.

### void* = "종류를 모르는 주소"

`void *ptr;`는 **"뭔가의 주소인데, 그게 int인지 double인지 뭔지는 몰라"**라는 포인터입니다. GPU 메모리는 그냥 "바이트 덩어리"라서 종류가 정해지지 않았죠. 그래서 CUDA의 메모리 주소는 `void*`로 표현합니다.

### 배열처럼 쓰기: `arr[i]`는 사실 `*(arr + i)`

포인터를 `arr[0]`, `arr[1]`처럼 **배열처럼** 쓸 수 있어요. 그런데 이건 새로운 문법이 아니라, 방금 배운 `*`(주소 따라가기)의 **편한 표기**일 뿐입니다. **`[]`는 마법이 아니에요:**

```c
arr[0]  ≡  *(arr + 0)  ≡  *arr     // 시작 칸을 엶
arr[1]  ≡  *(arr + 1)              // 한 칸 옆으로 가서 엶
arr[3]  ≡  *(arr + 3)              // 세 칸 옆으로 가서 엶
```

즉 `arr`이 "줄의 시작 **주소**"라면, `arr[i]`는 "거기서 i칸 이동한 곳의 **값**"이에요. 그래서 `arr[0]`은 `*arr`과 완전히 똑같습니다.

한 가지 더 — `arr + 1`은 주소를 **1바이트**가 아니라 **한 칸(= `sizeof(자료형)`)** 만큼 옮깁니다. `arr`이 `int*`(4바이트)면 `arr + 1`은 +4바이트예요. 타입이 "한 칸이 몇 바이트인지"를 알려주기 때문이죠.

```
arr (2000번지) ──► ┌──────┬──────┬──────┬─────┐
                    │ 2000 │ 2004 │ 2008 │ ... │  ← 각 칸의 주소 (+4씩)
                    │ [0]  │ [1]  │ [2]  │ ... │  ← 인덱스(칸 번호)
                    └──────┴──────┴──────┴─────┘
```

> 헷갈릴 때 규칙: **`arr[i]` = `*(arr + i)`**, 그리고 **`arr[0]` = `*arr`**. `[]`는 "그 주소로 가서 i칸 옆 값을 꺼내라"는 뜻입니다.

### 이 프로젝트 실제 코드에서

`cudaMalloc`의 시그니처를 봅시다:

- [fgpu_hook.c:441](../../hook/src/fgpu_hook.c#L441) — `cudaError_t cudaMalloc(void **devPtr, size_t size)`

`void **devPtr`에 별표가 **두 개**나 붙어 있죠? 무섭게 생겼지만 이유가 있습니다.

- 우리가 GPU 메모리를 잡으면 "그게 어디 있는지(주소)"를 사용자에게 **돌려줘야** 합니다.
- 그런데 C 함수는 값을 하나만 `return` 할 수 있어요. (여기서는 에러코드를 리턴에 씀)
- 그래서 "결과를 담을 그릇의 주소"를 미리 받아서, 그 그릇에 결과를 **써넣는** 방식을 씁니다.
- GPU 주소 자체가 `void*`인데, "그 `void*`를 담을 그릇의 주소"라서 `void**` (별표 두 개)가 됩니다.

실제로 후킹 함수 안에서 이렇게 씁니다:

- [fgpu_hook.c:465-468](../../hook/src/fgpu_hook.c#L465) — 진짜 `cudaMalloc`을 부른 뒤 `track_alloc(*devPtr, size);` — 여기서 `*devPtr`은 "그릇에 담긴 실제 GPU 주소를 꺼낸 것"입니다.

또 하나, 사용자가 이 함수를 부를 때는 이렇게 씁니다(테스트 코드 개념):

```c
void *gpu_mem;                 // GPU 주소를 담을 그릇
cudaMalloc(&gpu_mem, 1024);    // "gpu_mem 그릇의 주소를 줄게, 여기에 결과를 채워줘"
// 이제 gpu_mem 안에 실제 GPU 메모리 주소가 들어 있음
```

### 초보가 흔히 하는 오해/함정

- **`&`와 `*`를 헷갈림** → `&`는 "주소 내놔", `*`는 "주소 따라가서 값 꺼내". 반대 방향이에요.
- **초기화 안 한 포인터 사용** → `int *p; *p = 5;`는 "어딘지도 모르는 집에 물건을 던져넣는" 위험한 짓입니다. 프로그램이 죽어요(segfault). 포인터는 반드시 유효한 주소로 초기화하고 써야 합니다.
- **`void*`에서 바로 값을 꺼내려 함** → 종류를 모르니 못 꺼냅니다. 필요하면 `(int*)`처럼 종류를 먼저 알려줘야(캐스팅) 합니다.

### 한 줄 요약

> 포인터는 "값" 대신 "값이 사는 집 주소"를 담는 변수이고, `&`는 주소 얻기, `*`는 주소 따라가기, `void*`는 종류를 모르는 주소입니다.

### 💬 포인터 실전 Q&A — 실제로 막히는 지점들

> 아래는 이 프로젝트의 `cudaMalloc(void **devPtr, size_t size)`를 읽다가 신입이 **실제로 던진 질문들**을 순서대로 정리한 것입니다. 포인터가 헷갈릴 때 딱 이 지점들에서 막히거든요. 위 내용이 아직 흐릿하다면 이 Q&A로 한 번에 꿰어보세요.

??? question "Q1. `malloc`은 주소를 바로 돌려주는데, 왜 `cudaMalloc`은 `void**`로 받아요?"

    사실 그 직관이 맞아요 — C 표준 `malloc`은 진짜로 주소를 리턴합니다: `void* malloc(size_t size);`.

    차이는 딱 하나예요. **`cudaMalloc`은 하나뿐인 리턴 자리를 "에러코드"에 써버렸습니다.** GPU는 실패하는 경우(메모리 부족, 장치 없음 등)가 많아서, CUDA는 **모든 함수가 통일되게 `cudaError_t`를 리턴**하도록 설계했거든요. 리턴 자리가 찼으니 **주소는 매개변수로** 내보낼 수밖에 없고 → 그래서 `void**`입니다.

??? question "Q2. 함수 밖 변수를 바꿀 땐 별 하나(`*p`)로 됐는데, 왜 여긴 별 두 개예요?"

    규칙 하나로 통일됩니다:

    > **매개변수의 별 개수 = 바꾸려는 변수의 타입 + 별 1개**

    - `int`를 바꾸려면 → `int*` (별 1개)
    - `void*`(주소)를 바꾸려면 → `void**` (별 2개)

    함수 **안에서 써넣는 동작**은 둘 다 `*p = ...`(별 하나)로 **똑같아요.** 별이 하나 더 붙는 건 "타입 선언"에서일 뿐이고, 이유는 **바꾸려는 대상이 이미 포인터(주소)라서** 입니다.

??? question "Q3. `*devPtr = 2000`과 `devPtr = 3000`은 뭐가 달라요?"

    동작 자체는 이렇게 이해하면 정확합니다:

    - `*devPtr = 2000;` → devPtr이 **가리키는 곳(집)의 값**을 2000으로 바꿈
    - `devPtr = 3000;` → devPtr **자신**이 다른 곳(3000)을 가리키게 바꿈

    그런데 **함수 밖에 미치는 효과가 완전히 달라요** (C는 인자를 복사해서 넘기니까 — pass-by-value):

    | 코드 | 무엇을 바꾸나 | 함수 밖에서 보이나? |
    |------|----------------|----------------------|
    | `*devPtr = 2000;` | 가리키는 **공유된 집**의 값 | ✅ 보임 (호출자 변수 바뀜) |
    | `devPtr = 3000;` | **복사본 포인터 자신** | ❌ 안 보임 (return하면 증발) |

    바로 이 차이 때문에, 호출자의 변수를 진짜로 채우려는 `cudaMalloc`은 `devPtr = 새주소`(복사본만 바뀜)가 아니라 **`*devPtr = 새주소`**(호출자 변수에 도달)를 써야 하고, 그래서 `void**`가 필요합니다.

    > ⚠️ 참고: `void *devPtr`(별 1개)로는 `*devPtr = ...`이 컴파일 안 돼요. `void*`는 무슨 타입·몇 바이트인지 몰라 역참조를 못 합니다. `int *`처럼 타입이 있어야 하죠. (진짜 `cudaMalloc`은 채울 값이 "주소"라서 `void**`가 딱 맞습니다.)

??? question "Q4. 선언의 `*`랑 사용의 `*`가 같은 거예요?"

    정반대로 동작해요. 같은 별표라 헷갈리는 겁니다:

    | 어디서 | `*`의 뜻 |
    |--------|----------|
    | **선언** `void **devPtr` | 타입 표시 — "주소의 주소를 담는 그릇" (별을 **붙임**) |
    | **사용** `*devPtr` | 역참조 — "상자를 하나 **연다**" (별을 **뗌**) |

    그래서:

    - `devPtr` (그대로) = **주소의 주소** (`void**`)
    - `*devPtr` (별 하나 뗌) = **주소** (`void*`, 실제 GPU 주소)

    별을 하나 뗄 때마다 상자를 한 겹씩 여는 거예요. 실제로 [fgpu_hook.c:468](../../hook/src/fgpu_hook.c#L468)의 `*devPtr`이 "그릇을 열어 꺼낸 실제 GPU 주소"입니다.

??? question "Q5. GPU 주소면, CPU에서 그 값을 직접 못 다뤄요?"

    네 — 이게 모든 것의 뿌리예요. **GPU 메모리(VRAM)는 CPU 램과 물리적으로 다른 칩**이라, CPU 명령어로는 GPU 주소를 **직접 역참조 못 합니다(하면 크래시).**

    그래서 `cudaMalloc`이 준 GPU 주소는 CPU 입장에선 **"손잡이(handle)"** 예요:

    - **CPU 코드**: 그 주소를 직접 열지 못함. 대신 `cudaMemcpy(gpu주소, cpu데이터, 크기, ...)`로 **경계 너머로 복사**해서 넣고 뺌.
    - **GPU 코드(커널)**: 그 주소를 진짜 포인터처럼 **직접 역참조** (자기 동네니까).

    즉 `cudaMalloc`이 "손잡이를 네 변수에 넣어주는 것" 자체가 존재 이유이고, 그래서 `void**`로 생긴 겁니다.

??? question "Q6. 그럼 이 전체 흐름이 맞나요?"

    ```c
    void *gpu_mem;                 // ① 주소 담을 그릇 (gpu_mem 값이 곧 주소)
    cudaMalloc(&gpu_mem, 1024);    // ② 그릇을 '채워야' 하니 &gpu_mem (주소의 주소)
                                   //    → 이제 gpu_mem 안에 GPU 주소가 들어감
    cudaMemcpy(gpu_mem, cpu_data,  // ③ 채워진 주소를 '읽기만' → gpu_mem (값 그대로)
               1024, cudaMemcpyHostToDevice);
    ```

    맞아요. 그리고 ②와 ③의 차이를 **일반 규칙**으로 외우면 앞으로 어떤 C 함수를 봐도 안 헷갈립니다:

    > **함수가 내 변수를 바꿔야 하면 → `&변수`(주소)를 넘긴다.**
    > **함수가 내 변수 값을 읽기만 하면 → `변수`(값)를 넘긴다.**

    `cudaMalloc`은 빈 그릇을 **채워주는** 함수라 `&`, `cudaMemcpy`는 채워진 그릇을 **가져다 쓰는** 함수라 그냥 넘기는 거예요.

    > ⚠️ 참고: `cudaMallocManaged`(Unified Memory) 같은 고급 모드에선 이 CPU/GPU 경계가 흐려져 같은 포인터를 공유하기도 합니다. 하지만 기본 모델은 "둘은 다른 메모리" — 이것부터 확실히 잡으세요.

## 3. 함수와 에러 코드 — C가 "성공/실패"를 알리는 방식

### 왜 필요한가

Python이나 Java는 문제가 생기면 예외(exception)를 던지죠(`raise`, `throw`). 그런데 **C에는 예외라는 게 없습니다.** 대신 함수가 **"성공했는지 실패했는지를 반환값(return)으로 알려주는"** 아주 오래된 관례를 씁니다. CUDA 함수들이 전부 이 방식이라, 꼭 알아야 합니다.

### 일상 비유

식당에서 음식을 주문했는데, 종업원이 "완성!"이라고 접시를 주는 대신 **영수증에 도장을 찍어 돌려주는** 상황을 상상해보세요. 도장이 `0`이면 "정상 처리", `2`면 "재료 소진(메모리 부족)". 음식(실제 결과물)은 따로 전달하고, **영수증(반환값)으로 성공/실패만 알려주는** 겁니다.

### 아주 작은 코드 예시

```c
// 반환값으로 성공(0)/실패(0이 아닌 값)를 알리는 흔한 C 관례
int divide(int a, int b, int *result) {
    if (b == 0) {
        return 1;          // 실패! "0으로 나눌 수 없어" (에러 코드 1)
    }
    *result = a / b;       // 진짜 결과는 포인터로 돌려준다
    return 0;              // 성공! (관례상 0 = OK)
}
```

여기서 두 가지 관례가 보입니다:
1. **반환값 = 에러 코드.** `0`이면 성공, 0이 아니면 무슨 문제인지 나타내는 숫자.
2. **진짜 결과는 포인터 인자로 돌려준다.** (`int *result`에 써넣음)

이게 바로 아까 `cudaMalloc(void **devPtr, ...)`이 왜 그렇게 생겼는지에 대한 답이기도 합니다.

### cudaError_t — CUDA 판(版) 에러 코드

CUDA 함수들은 `cudaError_t`라는 자료형으로 결과를 돌려줍니다. 이건 사실상 "미리 이름 붙여둔 정수들의 집합"이에요:

- `cudaSuccess` = 0 (성공)
- `cudaErrorMemoryAllocation` = 2 (메모리 할당 실패 = GPU OOM)

### 이 프로젝트 실제 코드에서

우리 후킹 함수는 "quota를 넘으면 진짜 CUDA를 부르지도 않고 OOM 에러를 돌려주는" 게 핵심입니다:

- [fgpu_hook.c:455-461](../../hook/src/fgpu_hook.c#L455) — quota를 넘으면 `return cudaErrorMemoryAllocation;` (도장 `2`를 찍어 돌려줌)
- [fgpu_hook.c:466](../../hook/src/fgpu_hook.c#L466) — 진짜 `cudaMalloc`이 `cudaSuccess`를 돌려줬을 때만 사용량을 늘림

이 `cudaErrorMemoryAllocation`(=2)은 PyTorch까지 그대로 전달되어 파이썬에서 `OutOfMemoryError`로 보입니다. 즉 **C의 에러 코드 하나가 여러 층을 타고 올라가는** 겁니다.

### 초보가 흔히 하는 오해/함정

- **"0이 실패 아니야?"** → C의 오랜 관례에서는 **0이 성공**입니다(에러가 0개). 헷갈리기 쉬우니 조심.
- **반환값을 무시함** → `cudaMalloc(...)`을 부르고 결과 코드를 안 보면, GPU가 실패했는데도 모르고 진행하다 나중에 크래시합니다. 항상 반환값을 확인하세요.

### 한 줄 요약

> C에는 예외가 없어서, 함수는 반환값(에러 코드, 0=성공)으로 성공/실패를 알리고 진짜 결과물은 포인터로 돌려줍니다.

## 4. malloc / free — 메모리 대여와 반납 (이 프로젝트의 심장)

여기가 제일 중요합니다. 공들여 읽어주세요. 이 프로젝트 전체가 결국 "GPU 판 malloc을 가로채는" 것이기 때문이에요.

### 왜 필요한가

프로그램은 실행 중에 "지금 데이터를 담을 공간이 더 필요해!"라는 순간이 옵니다. 얼마나 필요할지는 실행해봐야 아는 경우가 많죠(예: 사용자가 올린 이미지 크기). 이렇게 **실행 중에 필요한 만큼 메모리를 빌려 쓰는 것이 `malloc`**이고, 다 쓰고 **돌려주는 게 `free`**입니다.

이 프로젝트의 `cudaMalloc`은 정확히 이것의 **GPU 버전**입니다. CPU 메모리 대신 GPU 메모리를 빌려주죠.

### 일상 비유

**셀프 창고 대여**를 생각하세요.

- `malloc(1024)` = "1024바이트짜리 창고 칸을 빌려주세요." → 창고 회사가 빈 칸을 찾아서 **그 칸의 열쇠(주소)**를 줍니다.
- 그 칸에 물건(데이터)을 넣고 씁니다.
- `free(주소)` = "이 칸 이제 안 써요, 반납합니다." → 다른 사람이 쓸 수 있게 됩니다.

만약 **반납(free)을 안 하면** 창고 칸이 계속 점유된 채로 남습니다. 이걸 "메모리 누수(memory leak)"라고 해요. 창고가 꽉 차버리면 더 이상 빌릴 수 없게 되죠.

### 아주 작은 코드 예시 (CPU 메모리)

```c
// 정수 100개를 담을 공간을 빌린다
int *arr = malloc(100 * sizeof(int));   // sizeof(int)=4, 총 400바이트 대여
if (arr == NULL) {                       // 실패하면 NULL(빈 주소)을 돌려줌
    // 메모리 부족 처리
}

arr[0] = 42;   // 빌린 공간에 값 쓰기
// ... 다 쓰고 나면 ...

free(arr);     // 반납! 이걸 잊으면 메모리 누수.
```

- `malloc(N)` → N바이트를 빌리고 **그 시작 주소를 반환**. 실패 시 `NULL`.
- `free(ptr)` → `ptr`이 가리키는 공간을 반납.

이 힙(heap) 메모리는 "실행 중에 자유롭게 빌리고 반납하는 공간"이라고 이해하면 됩니다.

### cudaMalloc = GPU 판 malloc

이제 핵심입니다. `cudaMalloc`은 `malloc`과 똑같은 개념인데 **CPU가 아니라 GPU 메모리를 빌려줍니다.** 그리고 이 프로젝트는 바로 이 `cudaMalloc`을 **가로채서(hook)** "네가 빌리려는 만큼 더하면 quota(상한선)를 넘니?"를 검사합니다.

넘으면? 진짜 창고 회사(CUDA)에 요청을 **전달하지도 않고** "재료 소진(OOM)" 영수증을 돌려줍니다.

### 이 프로젝트 실제 코드에서 (매우 중요)

**후킹된 cudaMalloc의 전체 흐름**:

- [fgpu_hook.c:441](../../hook/src/fgpu_hook.c#L441) — `cudaMalloc(void **devPtr, size_t size)` 함수 시작
- [fgpu_hook.c:455-462](../../hook/src/fgpu_hook.c#L455) — **quota 검사**: `g_used + size > g_quota`이면 진짜 함수를 안 부르고 OOM 반환
- [fgpu_hook.c:465](../../hook/src/fgpu_hook.c#L465) — 통과하면 `real_cudaMalloc(devPtr, size)`로 **진짜 CUDA에 위임**
- [fgpu_hook.c:467-468](../../hook/src/fgpu_hook.c#L467) — 성공하면 `g_used += size;` (빌린 만큼 사용량 증가) + `track_alloc(...)`으로 기록

**후킹된 cudaFree의 흐름**:

- [fgpu_hook.c:494](../../hook/src/fgpu_hook.c#L494) — `cudaFree(void *devPtr)` 함수 시작
- [fgpu_hook.c:503](../../hook/src/fgpu_hook.c#L503) — `real_cudaFree(devPtr)`로 진짜 반납 위임
- [fgpu_hook.c:505-507](../../hook/src/fgpu_hook.c#L505) — 반납된 크기를 찾아 `g_used -= freed;` (사용량 감소)

그리고 재미있는 점 — 우리 후킹 코드 **내부에서도** 일반 `malloc`/`free`를 씁니다. 포인터↔크기 기록용 노드를 만들 때요:

- [fgpu_hook.c:239](../../hook/src/fgpu_hook.c#L239) — `malloc(sizeof(*e))`로 기록용 노드 하나 대여
- [fgpu_hook.c:264](../../hook/src/fgpu_hook.c#L264) — `free(dead)`로 그 노드 반납

즉 **CPU 메모리는 일반 `malloc`/`free`로, GPU 메모리는 `cudaMalloc`/`cudaFree`로** 다룹니다. 이름이 비슷한 건 우연이 아니에요 — 같은 개념이니까요.

### 초보가 흔히 하는 오해/함정

- **free를 잊음** → 메모리 누수. 프로그램이 오래 돌수록 메모리를 야금야금 먹다가 결국 터집니다.
- **free를 두 번 함(double free)** → 이미 반납한 창고를 또 반납하려 하면 프로그램이 죽습니다.
- **free한 뒤에 그 주소를 계속 씀(use-after-free)** → 반납한 창고에 계속 물건을 넣는 격. 위험합니다.
- **malloc 실패를 확인 안 함** → 메모리가 부족하면 `NULL`이 오는데, 이걸 안 보고 쓰면 크래시. [fgpu_hook.c:240](../../hook/src/fgpu_hook.c#L240)에서 `if (!e)`로 실패를 확인하는 걸 보세요.

### 한 줄 요약

> `malloc`은 메모리를 빌리고 `free`는 반납하며, `cudaMalloc`은 그 GPU 버전 — 이 프로젝트는 바로 이 GPU 대여 요청을 가로채 상한선을 검사합니다.

## 5. 컴파일이란? — .c 파일이 실행되기까지

### 왜 필요한가

Python은 `.py` 파일을 바로 실행하죠(`python foo.py`). 그런데 C는 사람이 쓴 `.c` 파일을 컴퓨터가 바로 이해하지 못합니다. **컴퓨터가 알아듣는 기계어로 번역하는 과정이 필요**한데, 이걸 "컴파일(compile)"이라고 합니다. 이 프로젝트의 `libfgpu.so`도 컴파일의 결과물이에요.

### 일상 비유

한국어로 쓴 소설을 영어권 독자에게 보여주려면 **번역가**가 필요하죠. 컴파일러(`gcc`)가 그 번역가입니다. `.c`(한국어 원고)를 → 기계어(영어 번역본)로 바꿔줍니다. 번역이 끝나면 원고 없이 번역본만 있어도 독자(컴퓨터)가 읽을 수 있어요.

### 흐름과 용어

```
   fgpu_hook.c   ──[ gcc 컴파일러 ]──►   libfgpu.so
   (사람이 쓴 소스)                       (컴퓨터가 실행하는 결과물)
```

- **`gcc`**: 리눅스에서 가장 많이 쓰는 C 컴파일러(번역가).
- **헤더(.h)**: "이런 함수가 있어요"라고 목록만 적어둔 파일. 요리로 치면 **메뉴판**입니다. 실제 요리법(구현)은 다른 곳에 있고, 메뉴판만 먼저 보여줘서 "이 함수 써도 돼"라고 알려줍니다. `#include <stdio.h>`가 "표준 입출력 메뉴판을 가져와"라는 뜻.
- **링크(link)**: 여러 조각(내 코드 + 라이브러리)을 하나로 이어붙이는 마지막 단계. `-lcudart`는 "CUDA 런타임 라이브러리랑 이어붙여줘"라는 뜻.

### 이 프로젝트 실제 코드에서

파일 맨 위의 `#include`들이 바로 "메뉴판 가져오기"입니다:

- [fgpu_hook.c:72](../../hook/src/fgpu_hook.c#L72) — `#include <stdio.h>` (`fprintf`, `stderr` 쓰려고)
- [fgpu_hook.c:73](../../hook/src/fgpu_hook.c#L73) — `#include <stdlib.h>` (`getenv`, `malloc`, `free` 쓰려고)
- [fgpu_hook.c:79](../../hook/src/fgpu_hook.c#L79) — `#include <cuda_runtime_api.h>` (`cudaError_t` 등 CUDA 메뉴판)

실제 컴파일 명령은 빌드 스크립트에 있습니다 (2장에서 이 스크립트를 줄줄이 뜯어봅니다):

- [build_hook.sh:15-19](../../scripts/build_hook.sh#L15) — `gcc`로 `fgpu_hook.c`를 `libfgpu.so`로 번역하는 명령

### 초보가 흔히 하는 오해/함정

- **"컴파일만 하면 실행되겠지"** → 컴파일(번역) 후 링크(이어붙이기)까지 성공해야 실제로 돌아가는 결과물이 나옵니다.
- **헤더(.h)에 실제 코드가 있다고 착각** → 헤더는 대부분 "선언(목록)"만 있고, 실제 구현은 라이브러리(`.so`) 안에 있습니다.

### 한 줄 요약

> `.c`는 사람 언어라 `gcc`가 기계어로 번역(컴파일)하고 조각들을 이어붙여야(링크) 실행 가능한 결과물이 됩니다.

## 6. 공유 라이브러리 .so — libfgpu.so의 정체

### 왜 필요한가

이 프로젝트의 최종 산출물은 `build/libfgpu.so`라는 파일입니다. 이게 도대체 뭘까요? `.so`가 뭔지 알아야 이 프로젝트가 "어떻게 CUDA 함수를 가로채는지"를 이해할 수 있습니다.

### 일상 비유

`.so`(shared object)는 **여러 프로그램이 공유해서 쓰는 "부품 상자"**입니다. 윈도우를 써보셨다면 `.dll` 파일을 본 적 있을 텐데, `.so`는 리눅스판 `.dll`이라고 생각하면 정확합니다(사촌 관계).

- 여러 프로그램이 똑같은 기능(예: CUDA 메모리 할당)을 각자 갖고 있으면 낭비죠.
- 그래서 그 기능을 `.so` 부품 상자 하나에 담아두고, **여러 프로그램이 실행 시점에 꺼내 씁니다.**

### 이게 후킹의 핵심 트릭입니다

리눅스에는 `LD_PRELOAD`라는 특별한 환경변수가 있어요. 여기에 `.so` 파일 경로를 넣으면 **그 라이브러리를 다른 무엇보다 먼저 로드**합니다.

우리 `libfgpu.so` 안에는 진짜 CUDA와 **똑같은 이름**의 `cudaMalloc` 함수가 들어 있습니다. `LD_PRELOAD`로 먼저 로드되면, 프로그램이 `cudaMalloc`을 부를 때 진짜 CUDA 것이 아니라 **우리 것이 먼저 불립니다.** 이게 "가로채기"의 원리예요.

```bash
# 실행할 때 우리 .so 를 먼저 끼워넣는다
LD_PRELOAD=./build/libfgpu.so  FGPU_RATIO=0.4  ./my_gpu_program
```

### 이 프로젝트 실제 코드에서

- 파일 맨 위 주석 [fgpu_hook.c:26-33](../../hook/src/fgpu_hook.c#L26)에 이 `LD_PRELOAD` 원리가 자세히 설명되어 있습니다.
- 그럼 "진짜" `cudaMalloc`은 어떻게 부를까요? [fgpu_hook.c:307](../../hook/src/fgpu_hook.c#L307) — `dlsym(RTLD_NEXT, "cudaMalloc")`로 "나 다음에 로드된 진짜 CUDA의 cudaMalloc 주소를 찾아줘"라고 요청해서 함수 포인터에 저장합니다.
- 빌드 스크립트에서 `.so`를 만드는 핵심 옵션: [build_hook.sh:15](../../scripts/build_hook.sh#L15) — `gcc ... -fPIC -shared`. `-shared`가 "실행파일 말고 공유 라이브러리(.so)로 만들어"라는 뜻입니다.

### 초보가 흔히 하는 오해/함정

- **".so는 그냥 두 번 클릭하면 실행되는 거 아냐?"** → 아닙니다. `.so`는 **혼자 실행되지 않는 부품**입니다. 다른 프로그램이 불러서 써야 합니다.
- **"LD_PRELOAD면 모든 걸 가로챌 수 있다"** → 정적 링크된(자기 안에 CUDA를 통째로 품은) 프로그램은 못 가로챕니다. 이 프로젝트가 "협력적(cooperative) 모델"이라 부르는 한계입니다.

### 한 줄 요약

> `.so`는 리눅스판 `.dll`(공유 부품 상자)이고, `LD_PRELOAD`로 우리 `libfgpu.so`를 먼저 끼워 넣어 진짜 CUDA 함수를 가로챕니다.

## 7. 전역변수, static, struct, 연결 리스트 — g_used와 g_allocs 읽기

마지막입니다. 이 네 개념을 알면 `fgpu_hook.c`의 상태 관리 코드를 읽을 수 있어요.

### 7-1. 전역변수 (global variable)

**왜 필요한가**: 후킹된 `cudaMalloc`과 `cudaFree`는 서로 다른 함수지만, **"현재 얼마나 썼는지(g_used)"라는 정보를 공유**해야 합니다. malloc이 늘리고 free가 줄이니까요. 함수 안에 넣으면 함수가 끝날 때 사라지므로, **함수 밖에 두어 프로그램이 사는 내내 유지되는** 변수가 필요합니다. 그게 전역변수입니다.

**비유**: 여러 직원(함수)이 같이 보는 **사무실 화이트보드**. 한 명이 "현재 재고 30"이라고 적으면 다른 직원도 그 값을 봅니다.

- [fgpu_hook.c:128](../../hook/src/fgpu_hook.c#L128) — `static size_t g_used = 0;` — 여러 후킹 함수가 공유하는 "현재 사용량 화이트보드"

> 관례: 이 프로젝트에서 전역변수는 이름 앞에 `g_`를 붙입니다(`g` = global). `g_used`, `g_quota`, `g_lock`... 보이면 "아, 전역이구나" 하세요.

### 7-2. static

**왜 필요한가**: `static`을 전역변수에 붙이면 **"이 변수는 이 `.c` 파일 안에서만 보여"**라는 뜻입니다. 다른 라이브러리에 우연히 같은 이름 변수가 있어도 충돌하지 않게 막아주는 안전장치예요.

**비유**: 화이트보드를 **우리 사무실 안에만** 두고, 옆 회사에서는 못 보게 하는 것.

- [fgpu_hook.c:94](../../hook/src/fgpu_hook.c#L94) — `static cudaError_t (*real_cudaMalloc)(...) = NULL;` — 이 파일 전용 함수 포인터. 주석 [fgpu_hook.c:91-92](../../hook/src/fgpu_hook.c#L91)에 `static`의 의미가 설명돼 있습니다.

### 7-3. 구조체 struct — 관련 데이터 묶기

**왜 필요한가**: "이 GPU 포인터는 몇 바이트짜리였지?"를 나중에 알아야 `cudaFree`에서 사용량을 정확히 뺄 수 있습니다. 그러려면 **"주소 + 크기"를 한 묶음으로** 저장해야 해요. 서로 다른 자료형(포인터 + 크기)을 하나로 묶는 게 `struct`입니다.

**비유**: 택배 송장. 한 장에 "주소, 무게, 다음 송장 위치"가 같이 적혀 있죠. `struct`는 그런 **여러 칸이 있는 양식**입니다.

**아주 작은 예시**:

```c
struct point {
    int x;   // 칸 1
    int y;   // 칸 2
};

struct point p;
p.x = 3;     // 점(.)으로 각 칸에 접근
p.y = 5;
```

**이 프로젝트 실제 코드에서**:

- [fgpu_hook.c:226-230](../../hook/src/fgpu_hook.c#L226) — `alloc_entry_t` 구조체 정의:
  - `void *ptr;` — GPU 주소 (칸 1)
  - `size_t size;` — 그 할당의 바이트 수 (칸 2)
  - `struct alloc_entry *next;` — **다음 노드를 가리키는 포인터** (칸 3)

세 번째 칸이 바로 다음 개념, 연결 리스트로 이어집니다.

### 7-4. 연결 리스트 (linked list)

**왜 필요한가**: 살아있는 GPU 할당이 몇 개일지 미리 알 수 없습니다(실행 중에 늘었다 줄었다). 배열은 크기를 미리 정해야 하죠. 그래서 **"각 노드가 다음 노드를 가리키며 사슬처럼 이어지는"** 유연한 자료구조, 연결 리스트를 씁니다.

**비유**: **보물찾기 쪽지**. 첫 쪽지에 "다음은 냉장고 뒤" → 냉장고 뒤 쪽지에 "다음은 책상 서랍" → ... 각 쪽지(노드)가 **다음 쪽지의 위치(next 포인터)**를 알려줍니다. 몇 개든 이어붙일 수 있어요.

```
g_allocs ──► [ptr:0xA, size:256, next] ──► [ptr:0xB, size:64, next] ──► NULL
 (head)         첫 번째 노드                  두 번째 노드            (끝 표시)
```

- `g_allocs`는 **첫 노드(head)의 주소**를 담습니다.
- 마지막 노드의 `next`는 `NULL`(끝이라는 표시).

**이 프로젝트 실제 코드에서**:

- [fgpu_hook.c:232](../../hook/src/fgpu_hook.c#L232) — `static alloc_entry_t *g_allocs = NULL;` — 리스트의 머리(head). 처음엔 비어서 `NULL`.
- [fgpu_hook.c:238-250](../../hook/src/fgpu_hook.c#L238) — `track_alloc`: 새 할당을 **리스트 맨 앞에 끼워넣기**. `e->next = g_allocs; g_allocs = e;` (새 쪽지가 기존 첫 쪽지를 가리키게 하고, 자기가 새 첫 쪽지가 됨)
- [fgpu_hook.c:257-270](../../hook/src/fgpu_hook.c#L257) — `pop_alloc`: 리스트를 훑어 특정 주소를 찾아 **제거하고 크기를 돌려줌**. `cudaFree`가 이걸 써서 "얼마를 뺄지" 알아냅니다.

이제 처음의 전체 그림이 맞춰집니다:
1. `cudaMalloc` → 성공하면 `g_used += size`, `track_alloc`으로 리스트에 쪽지 추가.
2. `cudaFree` → `pop_alloc`으로 쪽지 찾아 크기 회수, `g_used -= 그크기`.

### 초보가 흔히 하는 오해/함정

- **struct 접근을 `.`과 `->` 중 뭘 쓸지 헷갈림** → 구조체 변수는 `p.x`, 구조체 **포인터**는 `p->x`. 이 프로젝트는 대부분 포인터라 `->`를 씁니다(예: `e->size`).
- **연결 리스트가 배열보다 항상 빠르다고 오해** → 아닙니다. 특정 원소를 찾으려면 처음부터 하나씩 훑어야 해서 느릴 수 있어요. 주석 [fgpu_hook.c:221-224](../../hook/src/fgpu_hook.c#L221)에도 "할당이 많아지면 hash map으로 바꾸라"고 적혀 있습니다.
- **`next`를 `NULL`로 안 끝냄** → 리스트 끝을 표시 안 하면 무한 루프에 빠집니다.

### 한 줄 요약

> 전역변수(`g_`)는 함수들이 공유하는 화이트보드, `static`은 파일 전용 표시, `struct`는 데이터 묶음, 연결 리스트는 "다음 노드를 가리키며 이어지는" 유연한 목록 — 이 넷으로 GPU 할당 상태를 추적합니다.
