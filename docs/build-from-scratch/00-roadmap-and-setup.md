## 1. 개발의 대원칙: "얇게, 세로로, 검증하며"

밑바닥부터 큰 시스템을 만들 때 초보가 가장 많이 하는 실수는 **"완벽한 부품을 하나씩 완성한 뒤 마지막에 조립"** 하려는 것입니다. 그러면 조립하는 날 모든 게 한꺼번에 터집니다.

이 프로젝트는 정반대 철학으로 지어졌습니다. **CLAUDE.md**에 명시된 규칙이기도 해요:

> "각 스테이지의 결과물은 그 자체로 빌드·검증 가능해야 한다. 한 단계가 끝나면 다음 단계로."

즉 **얇지만 끝에서 끝까지 동작하는 조각(수직 슬라이스)** 을 먼저 만들고, 그게 검증되면 살을 붙입니다. 이걸 3가지 습관으로 기억하세요.

### 습관 ①: 가장 위험한 것부터 (훅 먼저)

이 프로젝트에서 "정말 될까?" 하고 가장 불확실한 건 **"LD_PRELOAD로 cudaMalloc을 진짜 가로챌 수 있나"** 입니다. 이게 안 되면 프로젝트 자체가 성립을 안 해요. 그래서 **Stage 1(최소 훅)** 을 가장 먼저 만듭니다. 백엔드·UI 같은 건 익숙한 기술이라 나중에 해도 안 무섭거든요.

> 💡 **원칙**: 기술적으로 가장 불확실한 부분(=프로젝트를 통째로 무너뜨릴 수 있는 부분)을 맨 앞에 두세요.

### 습관 ②: 스텁 → 검증 → 살 붙이기

절대 한 파일을 처음부터 끝까지 완성하려 하지 마세요. 예를 들어 훅을 만들 때조차:

1. `stderr`에 한 줄 찍는 빈 `cudaMalloc` 래퍼만 만들고 → **로드되는지 확인**
2. `dlsym`으로 진짜 함수 잡아서 그냥 위임만 → **가로채고 되돌려주는지 확인**
3. quota 계산 추가 → **ALLOW 뜨는지 확인**
4. DENY 로직 추가 → **거절되는지 확인**

각 단계 사이에 **실행해서 눈으로 확인**하는 "검증 게이트"가 있습니다. 이 리듬이 이 교과서 전체를 관통합니다.

### 습관 ③: baseline과 hooked를 항상 나란히

이 프로젝트의 모든 검증은 **"훅 없을 때(baseline) vs 훅 있을 때(hooked)"** 를 두 번 돌려 비교합니다. 훅 없을 땐 6GB 할당이 성공하고, 훅 있을 땐 거절되는 걸 눈으로 대조하는 거죠. 개발하면서도 이 습관을 유지하세요.

## 2. 전체 개발 순서 (한눈에)

아래가 밑바닥부터 짓는 **권장 순서**입니다. 이 교과서의 챕터 순서와 같아요.

```
[준비]  개발 환경 + 저장소 골격            ← 이 장(0장)
   │
   ▼
[1] Stage 1   최소 훅 (cudaMalloc/Free + quota)      ← 1장  ⭐가장 먼저·가장 중요
   │          "가로채기가 된다"를 증명
   ▼
[2] Stage 2·4 컨테이너화 + PyTorch 통합              ← 2장
   │          "격리된 상자 안에서도 훅이 먹힌다"
   ▼
[3] Stage 3·8 백엔드 (FastAPI+Docker+SQLite)         ← 3장
   │          "API로 세션을 만들고 관리한다"
   ▼
[4] Stage 5   평가 인프라 + 웹 UI                    ← 4장
   │          "실험 데이터를 뽑고 눈으로 조작한다"
   ▼
[5] Stage 5C·6·7 훅 확장 (Driver/VMM/launch)         ← 5장
   │          "여러 CUDA 계층을 다 덮는다" (재진입 가드 등장)
   ▼
[6] Stage 9·10·11 운영 기능 (인증·멀티GPU·Jupyter·어드미션) ← 6장
   │          "실제로 여럿이 안전하게 쓴다"
   ▼
[7] Stage 12  Duty-cycle 컴퓨트 스로틀                ← 7장
   │          "메모리에 이어 연산 시간도 나눈다"
   ▼
[종합]  전체 의존성·순서 요약·함정 체크리스트         ← 8장
```

### 왜 이 순서인가?

- **1 → 2**: 훅이 호스트에서 되는 걸 확인한 다음에야 "컨테이너 안에서도 되나"로 넓힙니다. 컨테이너부터 하면 훅 버그인지 도커 설정 문제인지 구분이 안 돼요.
- **2 → 3**: 컨테이너를 명령줄(`docker run`)로 손수 띄울 수 있게 된 다음에야, 그걸 **자동화**하는 백엔드를 만듭니다. 백엔드는 결국 `docker run`을 코드로 대신 쳐주는 것뿐이에요.
- **3 → 4**: 세션을 만들 수 있어야 그 세션들로 **실험**(격리·오버헤드)을 할 수 있습니다.
- **4 → 5**: 기본 파이프라인이 다 돌아간 뒤에, 훅을 여러 CUDA 계층으로 **확장**합니다. 이때 처음으로 "이중 카운트" 문제가 터지고 **재진입 가드**가 필요해져요. 일찍 넣으면 왜 필요한지 체감이 안 됩니다.
- **5 → 6 → 7**: 핵심이 튼튼해진 다음 운영 기능(인증·어드미션)과 고급 기능(스로틀)을 얹습니다.

> ⚠️ **함정**: "어드미션(여러 세션 지분 합 관리)"이나 "멀티 GPU" 같은 걸 일찍 하고 싶어질 수 있어요. 참으세요. 그건 세션이 여러 개 안정적으로 뜬 다음에야 의미가 있습니다.

## 3. 개발 환경 준비

이 프로젝트는 **NVIDIA GPU가 달린 Ubuntu 리눅스**에서 개발/실행합니다. (Windows 개발 PC에서는 `.sh`와 LD_PRELOAD가 안 돌아요 — 온보딩 2·3장 참고.) 자세한 설치 런북은 저장소의 [LINUX_SETUP.md](../../LINUX_SETUP.md)에 있고, 여기선 큰 항목만 짚습니다.

준비물 체크리스트:

1. **NVIDIA 드라이버** — `nvidia-smi`가 동작해야 합니다.
2. **CUDA 툴킷** — `/usr/local/cuda`에 설치. 훅을 `gcc`로 컴파일할 때 헤더/라이브러리가 필요해요. `nvcc`(CUDA 컴파일러)도 여기 들어 있습니다.
3. **Docker** + **nvidia-container-toolkit** — `docker run --gpus all ...`이 컨테이너 안에서 GPU를 보이게 해줍니다.
4. **Python 3.10+** — 백엔드용 (`venv` 권장).
5. **git** — 버전 관리.

> 💡 **핵심 제약 하나**: 호스트에서 빌드한 `libfgpu.so`는 컨테이너 안의 `libcudart`에 동적으로 링크됩니다. 그래서 **호스트 CUDA 메이저 버전과 컨테이너 이미지의 CUDA 메이저 버전을 맞춰야** 합니다(예: 둘 다 12.x). 이걸 안 맞추면 나중에 컨테이너 안에서 훅이 조용히 안 먹거나 심볼 에러가 납니다. 개발 초기에 반드시 확인하세요.

## 4. 저장소 골격 세우기

빈 디렉토리에서 시작한다면, 아래 골격을 미리 만들어 두면 각 장을 채워나가기 편합니다. (지금은 빈 폴더/파일만 만들고, 내용은 각 장에서 채웁니다.)

```
GpuCluster/
├── hook/
│   ├── src/fgpu_hook.c          ← 1장에서 채움 (훅의 심장)
│   └── tests/                    ← 각 스테이지 .cu 스모크 테스트
├── runtime-image/
│   ├── Dockerfile                ← 2장
│   └── entrypoint.sh             ← 2장
├── runtime-image-pytorch/        ← 2장 (Stage 4)
├── backend/
│   ├── pyproject.toml            ← 3장
│   ├── app/                      ← 3장 (main/api/services/schemas)
│   └── tests/                    ← 3·6장 (단위테스트)
├── scripts/
│   ├── build_hook.sh             ← 1장
│   ├── build_image.sh            ← 2장
│   ├── run_*.sh                  ← 각 장
│   └── eval/                     ← 4장 (실험 하네스)
├── build/                        ← 컴파일 산출물(libfgpu.so). gitignore
├── data/                         ← SQLite DB, 워크스페이스. gitignore
├── experiments/                  ← 실험 아티팩트. gitignore
└── CLAUDE.md / description.md     ← 문서
```

가장 먼저 만들 파일은 **딱 두 개**입니다: `hook/src/fgpu_hook.c`(빈 파일)와 `scripts/build_hook.sh`. 1장에서 이 둘부터 시작합니다.

> 💡 실제 완성된 저장소의 파일별 역할 지도는 [CLAUDE.md](../../CLAUDE.md)의 "Repository layout" 섹션에 전부 정리돼 있습니다. 막히면 그걸 정답지처럼 참고하세요.

## 5. 이 교과서 사용법

- 각 장은 **"이 장에서 만들 것 → 개발 순서 체크리스트 → 스텝별 코드 → 검증 게이트 → 함정 → 다음 예고"** 구조입니다.
- **검증 게이트를 건너뛰지 마세요.** 각 게이트를 통과해야 다음 스텝의 버그를 "이번에 새로 짠 것" 안에서만 찾을 수 있습니다.
- 완성본이 궁금하면 각 장의 `[파일:줄]` 링크로 실제 코드를 열어 대조하세요. 이 교과서는 **베끼기가 아니라 순서대로 쌓는 법**을 가르칩니다.
