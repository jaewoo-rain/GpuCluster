# 9장. 신청 한 건이 흘러가는 길 🚀

> 이 장에서 배울 것: 버튼을 누르는 순간부터 GPU가 일을 끝낼 때까지, 한 번의 신청이 어떻게 여행하는지 따라가 봐요.

## 🎯 오늘의 목표

1. 신청 하나가 어떤 순서로 흘러가는지 알기
2. 각 부품들이 서로 무엇을 주고받는지 알기
3. 통과되거나 거절될 때 어떤 일이 일어나는지 알기

---

## 편지 배달 이야기

편지를 보내는 과정을 생각해 봐요.

편지를 쓰고 -> 우체통에 넣으면 -> 집배원이 가져가고 -> 분류 센터를 거쳐 -> 받는 사람 집에 도착해요.

여러 단계를 거치지만, 각 단계에서 정해진 일을 해요.

GPU 신청도 똑같이 **여러 단계**를 거쳐요.
한 번 따라가 봐요!

---

## 전체 흐름도 — 위에서 아래로 따라가요

```
        +-----------------------------+
        |   1단계: 사람이 버튼 클릭   |
        |  비율 0.4 입력하고 클릭!   |
        +-----------------------------+
                      |
                      v
        +-----------------------------+
        |   2단계: 서버가 검사        |
        |  비율 합이 1.0 이하?        |
        |  YES -> 3단계로             |
        |  NO  -> "공간 부족" 거절    |
        +-----------------------------+
                      |
                      v
        +-----------------------------+
        |   3단계: Docker가 방 생성   |
        |  libfgpu.so 넣기            |
        |  번호표(세션 id) 발급       |
        +-----------------------------+
                      |
                      v
        +-----------------------------+
        |   4단계: 프로그램 시작      |
        |  "비율 0.4 = 내 몫 3.2GB"  |
        |  계산해서 기억              |
        +-----------------------------+
                      |
                      v
        +-----------------------------+
        |   5단계: GPU 주문           |
        |  libfgpu.so 가 가로챔       |
        |  quota 검사                 |
        |  OK -> GPU에 전달 (ALLOW)   |
        |  NO -> 에러 반환 (DENY)     |
        +-----------------------------+
                      |
                      v
        +-----------------------------+
        |   6단계: 결과 화면에 표시   |
        |  상태: 완료                 |
        |  로그에 ALLOW/DENY 기록     |
        +-----------------------------+
```

## 단계별로 자세히 보기

### 1단계: 사람이 버튼을 클릭해요

웹 화면에서 신청서를 써요.
- "비율은 0.4로 해주세요"
- "이 프로그램을 실행해 주세요"

그리고 버튼을 클릭해요.

```
  웹 화면
  +-----------------------------+
  | 비율: [ 0.4       ]         |
  | 프로그램: [ AI 프로그램 ]   |
  |                             |
  |       [ 신청 버튼! ]        |
  +-----------------------------+
         클릭! |
               v
```

---

### 2단계: 서버가 신청을 받고 검사해요

서버가 신청서를 받아요.
먼저 **비율 합 검사**를 해요.

```
  서버 (접수 창구):
  "지금 0.7짜리 방이 이미 있는데, 0.4를 또 달라고?
   0.7 + 0.4 = 1.1 -> 1.0을 넘어! 거절!"

  또는:

  "지금 아무것도 없네. 0.4는 1.0 이하. 통과!"
```

통과하면 3단계로 가요.
거절되면 웹 화면에 "공간 부족" 메시지가 나와요.

---

### 3단계: Docker가 방을 만들어요

서버가 Docker에게 명령해요.
Docker가 새 컨테이너 방을 만들어요.

방을 만들 때 이것들을 넣어줘요.
- `libfgpu.so` (가로채기 도구)
- "이 방은 0.4만큼만 써"라는 메모 (4장에서 배운 먼저 끼어들기 목록에 같이 적어줘요)
- 실행할 프로그램

```
  Docker가 방을 만드는 순서:
  1. 빈 방 하나 준비
  2. libfgpu.so 넣기
  3. "이 방은 0.4만큼만 써"라고 방 안에 메모 붙이기
  4. 프로그램 준비
  5. 방 문 열기 (프로그램 시작!)
```

방이 생기면 서버는 번호표(세션 id)를 웹 화면에 보내줘요.

---

### 4단계: 방 안에서 프로그램이 시작돼요

방 문이 열리면 프로그램이 시작돼요.
`libfgpu.so`가 "이 방은 0.4만큼만 써" 메모를 읽어요.
그리고 "GPU 전체의 40%가 내 몫이구나!"를 계산해서 기억해 둬요.

```
  libfgpu.so 준비:
  "메모 확인: 비율 0.4
   GPU 전체 8GB x 0.4 = 3.2GB.
   나는 3.2GB까지만 허용할게."
```

---

### 5단계: GPU에 주문이 들어가요

프로그램이 일을 시작하면 GPU에 메모리를 달라고 해요.
바로 이때 `libfgpu.so`가 먼저 받아요.
(04장에서 배운 그 검사예요!)

```
  프로그램 -> "cudaMalloc: 1GB 줘!"
                    |
                    v
           libfgpu.so 가 받음
                    |
                    v
        1단계: 지금 쓰는 양 = 0
        2단계: 이번에 달라는 양 = 1GB
        3단계: 0 + 1GB = 1GB <= 3.2GB (내 몫)
        -> ALLOW! 진짜 GPU에 전달
                    |
                    v
              GPU 가 1GB 줌
              (방금 1GB를 받았으니 쓴 양이 1GB가 된 거예요!)
```

나중에 프로그램이 더 요청하면:

```
  프로그램 -> "cudaMalloc: 3GB 더 줘!"
                    |
                    v
           libfgpu.so 가 받음
                    |
                    v
        1단계: 지금 쓰는 양 = 1GB
        2단계: 이번에 달라는 양 = 3GB
        3단계: 1GB + 3GB = 4GB > 3.2GB (내 몫 초과!)
        -> DENY! 에러 반환
                    |
                    v
         프로그램에 "메모리 부족" 에러
         (GPU 는 아무것도 모름)
```

---

### 6단계: 결과가 화면에 나와요

프로그램이 끝나면 상태가 바뀌어요.
웹 화면에서 새로고침하면 "완료" 상태가 돼요.
로그 버튼을 누르면 `libfgpu.so`가 남긴 기록이 나와요.

```
  웹 화면 - 로그 창:
  [fgpu] ALLOW cudaMalloc size=1GB  used=1/3.2GB
  [fgpu] DENY  cudaMalloc size=3GB  used=1/3.2GB
  [fgpu] FREE  size=1GB             used=0/3.2GB
```

---

## 전체 흐름 한 줄씩 요약

1. 사람이 버튼 클릭 (비율 0.4 입력)
2. 서버가 신청 받고 비율 합 검사 (초과면 "공간 부족" 거절)
3. Docker가 방 만들고 번호표 발급 (libfgpu.so + 비율 메모 포함)
4. 방 안에서 프로그램 시작 (libfgpu.so가 "내 몫 3.2GB" 계산)
5. GPU 주문 -> libfgpu.so 가 가로채서 quota 검사 (ALLOW or DENY)
6. 결과 화면에 표시 (상태 "완료", 로그에 ALLOW/DENY 기록)

---

## 📦 한 줄 정리

버튼 하나를 누르면 여섯 단계를 거쳐요.
각 단계마다 다른 부품이 자기 일을 해요.

---

## ❓ 스스로 물어보기

**Q1. 2단계(서버 검사)에서 거절되는 건 어떤 상황이에요?**
👉 이미 있는 방들의 비율 합 + 새 요청 비율 > 1.0일 때예요.

**Q2. `libfgpu.so`가 DENY를 보낼 때 GPU는 알아요?**
👉 몰라요. libfgpu.so가 GPU에 주문을 보내기 전에 막으니까요.

**Q3. 로그에서 ALLOW/DENY를 어디서 볼 수 있어요?**
👉 웹 화면의 로그 창에서 볼 수 있어요.
