## 0. 교수님 피드백 (원문 → 실험 지시 번역)

| 원문 | 번역 |
|---|---|
| ① A·B 각각 실행 합친 시간 vs 병행 시간 → 오버헤드 도출 | speedup이 아니라 **공유의 시간 오버헤드**를 정량화 |
| ② 더 느려도 유저와 상호작용하면 효용 O. 학습 X, 추론 O | 지표를 throughput → **요청당 latency**로. 추론은 latency에 효용 |
| ③ 동시가 빨라지면 같은 사람=효용 O, 다른 사람=의미 X | makespan 단축은 **단일 소유자**에게만. 다른 유저는 각자 latency |
| ④ 로우데이터 다 수집 후 인사이트 선별 | 매 요청 원시 기록 → 분석에서 의미 있는 지표 선별 |

**철회한 접근**: "유저 대기시간(queue wait)" 비교 — 직접 측정값이 아니라 가정된
baseline이라 부정확. 사용자 지적 수용. 본 설계는 **직접 측정값만** 사용한다.

## 1. 지난 H1 실험 코드 검증 결과 (평결)

`notebooks/fgpu_infer.ipynb` + `notebooks/fgpu_analysis.ipynb` 직접 검토.

### 1.1 잘 된 점 (계측은 정확)
- `fgpu_infer.ipynb` cell[8]: `do_sample=False`(greedy → 출력 길이 고정·재현성),
  워밍업 3회 제외, 매 iter `torch.cuda.synchronize()`로 정확한 타이밍,
  **iter마다 `iter_start_epoch`/`iter_end_epoch` 기록**(epoch=CLOCK_REALTIME, 컨테이너 간 비교 가능).
- → 교수님 ④("로우데이터 다 수집")는 **infer 레벨에서 이미 충족**. CSV에 요청 단위 시작·끝·토큰이 다 남는다.

### 1.2 진짜 결함 2개
1. **분석이 오염된 평균을 씀.** `fgpu_analysis.ipynb` cell[5]는 메타의
   `mean_tokens_per_s`(= 20 iter **전체 평균**)를 그대로 쓴다. concurrent에서 B가 먼저 끝나
   A가 혼자 도는 **drain 구간**이 이 평균에 섞여 "공유 중 A 처리량"을 부풀린다.
   (발표 슬라이드 6의 14.6→7.0 정정이 이 문제. 단 그 정정은 **임시 스크립트**였고
   분석 노트북 파이프라인에는 반영 안 됨.)
2. **판정 프레임이 틀림.** cell[5]·cell[9]의 VERDICT 로직이 `speedup ≥ 1.2 = PASS(이득)`로
   **하드코딩**돼 있다. 교수님 ②가 지적한 "throughput speedup은 추론 효용의 잣대가 아님"과 정면 충돌.

### 1.3 평결
> **계측은 정확. "이득 없음(speedup≈1.0)"은 버그가 아니라 "둘 다 포화 + throttle 시분할"
> 조건에서의 참값.** 고칠 것은 **(a) 분석의 drain 오염**과 **(b) speedup-중심 판정 프레임** 둘 뿐.
> 새 실험은 측정 조건을 거의 그대로 두고(아래 §2), **분석·지표·해석**을 바로잡는다.

### 1.4 직전 권고("고정 시간") 정정
직전 대화에서 "고정 횟수 → 고정 시간"을 제안했으나, 분석 노트북 검토 후 **철회**한다.
이유: 교수님 ①의 오버헤드는 **각 잡의 완료 시간(makespan)** 비교라 **고정 작업량(N_ITERS)**이
있어야 정의된다. 고정 시간으로 바꾸면 완료 시점이 사라져 ① 지표가 무너진다.
**drain은 makespan의 정당한 일부**이고(잡이 실제로 그만큼 더 걸린 것), 오염 문제는
**per-job throughput/latency에 한해 overlap 구간으로 잘라** 해결한다(§4.2). → 고정 작업량 유지.

## 3. 측정·해석 프레임 (핵심 변경)

### 3.1 시나리오 (3 run — 사용자 뼈대 그대로)
1. **A solo** (ratio 0.4, throttle ON, N_ITERS 고정)
2. **B solo** (ratio 0.6, throttle ON, N_ITERS 고정)
3. **A+B concurrent** (수동 동시 시작, 나머지 동일)

모든 run에서 매 iter raw 기록(이미 됨). 추가 계측 불필요.

### 3.2 보고할 2개 축 (직접 측정값만)

**축 A — 시간 오버헤드 (교수님 ①, ③의 "단일 소유자" 관점)**
- `makespan_seq = makespan(A solo) + makespan(B solo)` (따로 이어 돌린 총시간)
- `makespan_conc = max(t_end_A, t_end_B) − min(t_start_A, t_start_B)` (같이 돌린 총시간)
- `overhead = makespan_conc − max(makespan_A_solo, makespan_B_solo)` (이상적 완전병렬 대비 초과)
- `speedup = makespan_seq / makespan_conc` — **단, 판정 잣대로 쓰지 않고 서술값으로만**.
  해석: ≈1.0 = "공유해도 총시간 손해 없음"(두 잡 공짜 동시 수용), <1.0 = 경합 패널티.
- **의미**: 내 일 2개를 한 GPU에서 동시에 돌릴 때의 총시간 비용. 단일 소유자에게 유효.

**축 B — 요청당 latency (교수님 ②, ③의 "다른 사람" 관점)**
- 각 iter의 `latency_s`(한 응답 생성 시간)를 solo vs concurrent로 비교.
- `p50_latency`, `p95_latency`를 **A·B 각각** 보고(다른 유저는 각자 latency만 신경).
- concurrent는 **overlap 구간 iter만** 사용(drain 제외, §4.2).
- **의미**: "같이 쓰면 한 응답이 얼마나 느려지나". 추론(사람이 응답 기다림)에서 진짜 중요한 비용.
  학습은 이 latency가 무의미 → 교수님 ②의 "학습 X, 추론 O"가 이 지표로 정직하게 드러남.

### 3.3 보조 지표
- `overlap_tps_X` = overlap 구간 `sum(gen_tokens)/wall-clock` (산술평균 아님).
- `occupancy_sum` = `overlap_tps_A/solo_tps_A + overlap_tps_B/solo_tps_B`
  (≈1.0이면 GPU 포화 = 여유 없음 = H1 확인).
- `gpu_util_timeline` (pynvml 전체 GPU 합산, 보조 — per-process 분해 불가).

## 4. 분석 변경 (drain 오염 영구 제거)

### 4.1 overlap 구간 정의 (직전 `16-34` 계획 재사용)
- `overlap_start = max(A 첫 iter_start, B 첫 iter_start)` (둘 다 활성 시점)
- `overlap_end   = min(A 마지막 iter_end,  B 마지막 iter_end)` (먼저 끝난 잡 종료)

### 4.2 iter 3분류 (경계 걸친 iter는 제외)
- `overlap`: `iter_start ≥ overlap_start` AND `iter_end ≤ overlap_end`
- `drain`:   `iter_start ≥ overlap_end`
- `boundary`: 경계 걸침 → 분석 제외(각주만). overlap iter < 5면 신뢰도 경고.

### 4.3 N_ITERS 상향
A(0.4)가 B(0.6)보다 느려 overlap 구간이 얇아진다(H1은 A overlap iter 7개로 노이즈).
→ **N_ITERS를 30~40으로** 올려 overlap iter ≥ 10 확보. A·B 동일 횟수(makespan 정의 단순 유지).
makespan_seq는 실제 완료시간 기반이라 N_ITERS 상향에 영향받지 않음(스케일만 커짐).

### 4.4 판정 프레임 교체
- `fgpu_analysis.ipynb`의 `speedup ≥ 1.2 = PASS` 하드코딩 **제거**.
- 대신 **서술형 리포트**: (1) makespan 오버헤드 표, (2) A·B 각 latency solo→conc 표,
  (3) occupancy_sum, (4) "이 워크로드는 포화형(=학습 성격) → 총시간 중립·요청 latency 증가"
  같은 **조건부 해석 문구**. PASS/FAIL 단일 verdict 대신 "어떤 관점에서 효용/비용인가"를 표로.

## 5. 정직한 예상 결과 & 범위 (과대광고 금지)

이 조건(둘 다 포화 + throttle 40/60)에서 **예상**:
- 축 A: `makespan_conc ≈ makespan_seq` → speedup ≈ 1.0, overhead ≈ 0.
  = **"공유해도 총시간 손해 없이 두 잡을 수용"** (정직한 긍정 결과).
- 축 B: concurrent의 요청당 latency가 solo 대비 **대략 1/ratio배 증가**
  (A는 throttle 0.4 + 경합, B는 0.6 + 경합).
- occupancy_sum ≈ 1.0 (GPU 포화, 여유 없음 — H1 재확인).

**이 실험이 증명하는 것**
- fGPU 코어 시분할이 두 잡을 **총시간 중립**으로 공존시킨다(오버헤드 ≈ 0).
- 비용은 **요청당 latency 증가**이며, 그 크기를 A·B 각각 정량화한다.
- 교수님 ②③: 이 latency 비용이 **추론(상호작용)에서는 견딜 만한 trade-off일 수 있으나
  학습(비상호작용)에서는 순수 손해**임을 — 지어낸 수치 없이 — 측정된 latency로 논의.

**이 실험이 증명하지 못하는 것 (정직)**
- "speedup > 1" 같은 시간 이득. 포화형 워크로드에선 구조적으로 안 나온다.
  시간 이득은 **상보/idle 워크로드**(이전 H2 계획)에서만 가능 — 이번 범위 밖(사용자 보류).
- SM 격리 / throttle-경합 완전 분리 / caching on 일반화.

## 부록 — 한 줄 요약
> 지난 실험은 계측 정확, 결함은 **분석의 drain 오염 + speedup-중심 판정**. 새 실험은 같은
> 조건에서 **makespan 오버헤드(①)와 요청당 latency(②)** 를 직접 측정값으로 보고하고,
> drain은 overlap 구간으로 제거한다. 포화형이라 시간 이득은 안 나오지만(정직), "총시간 중립 +
> latency 비용"을 정량화해 **추론에선 trade-off, 학습에선 손해(②③)** 를 지어낸 수치 없이 보인다.
