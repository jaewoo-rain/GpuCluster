---

## 지금까지 만난 부품들

이 책에서 하나씩 만났던 부품들을 기억하나요?

- GPU: 계산을 아주 빨리 하는 특별한 두뇌
- `libfgpu.so`: 주문을 가로채서 검사하는 도구
- 컨테이너: 각자의 칸막이 방
- Docker: 방을 만들어 주는 도구
- 백엔드 서버: 신청을 받는 접수 창구
- 기록 공책(DB): 누가 어떤 방을 받았는지 사물함처럼 적어두는 공책
- 웹 화면: 사람이 보고 클릭하는 화면

이 모든 것이 연결되어 하나의 서비스가 돼요.
지금부터 그 **전체 설계도**를 볼 거예요.

## 큰 그림 — 전체 설계도

손가락으로 위에서 아래로 따라가 보세요.

```
사용하는 사람
      |
      | (버튼 클릭, 신청서 작성)
      v
+------------------------------------------+
|          웹 화면 (브라우저)               |
|  - "GPU 방 주세요" 버튼                   |
|  - 방 목록 보기                           |
|  - 로그(기록) 보기                        |
+------------------------------------------+
      |
      | (신청 전달)
      v
+------------------------------------------+
|        백엔드 서버 (FastAPI)              |
|  - 신청서 받기                            |
|  - 비율 합 검사 (1.0 넘으면 거절)         |
|  - Docker에게 방 만들라고 시킴            |
|  - 번호표(세션 id) 발급                   |
|  - 기록 공책(DB)에 저장                   |
+------------------------------------------+
      |
      | (방 만들기 명령)
      v
+------------------------------------------+
|             Docker (도커)                 |
|  - 컨테이너 방 생성                       |
|  - libfgpu.so 방 안에 넣기               |
|  - 얼마만큼 쓸지 정한 값(비율) 전달      |
+------------------------------------------+
       |                  |
       v                  v
+-------------+    +-------------+
| 컨테이너 A  |    | 컨테이너 B  |
| (비율: 0.4) |    | (비율: 0.6) |
|             |    |             |
| 프로그램 A  |    | 프로그램 B  |
|      |      |    |      |      |
| libfgpu.soA |    | libfgpu.soB |
| (quota 검사)|    | (quota 검사)|
+------|------+    +------|------+
       |                  |
       +--------+---------+
                |
                v
         +-------------+
         |     GPU     |
         | (실제 계산  |
         |  두뇌)      |
         +-------------+
```

## 각 부품의 역할 한 줄씩

| 부품 이름 | 역할 한 줄 |
|---|---|
| 웹 화면 | 사람이 보고 클릭하는 화면 |
| 백엔드 서버 | 신청받고 기록하고 지시하는 접수 창구 |
| 기록 공책(DB) | 누가 어떤 방을 받았는지 적어두는 사물함 같은 공책 |
| Docker | 컨테이너 방을 만들어 주는 도구 |
| 컨테이너 | 각자의 칸막이 방 |
| `libfgpu.so` | 방 안에서 GPU 주문을 가로채서 검사하는 도구 |
| GPU | 실제 계산을 엄청 빨리 하는 두뇌 |

## 부품들 사이에 무슨 말을 주고받아요?

화살표마다 주고받는 내용이 달라요.

```
웹 화면 -> 서버:
  "비율 0.4짜리 방 만들어 주세요!"

서버 -> Docker:
  "비율 0.4인 컨테이너 하나 만들어. libfgpu.so도 넣어."

Docker -> 컨테이너:
  방 생성! (libfgpu.so + "이 방은 0.4만큼만 써"라는 메모 붙이기)

컨테이너 안 프로그램 -> libfgpu.so:
  "cudaMalloc - GPU야, 3GB 줘!"

libfgpu.so -> 프로그램 (검사 후):
  "ALLOW" 또는 "DENY"

libfgpu.so -> GPU (통과한 경우):
  진짜 cudaMalloc 요청 전달

서버 -> 웹 화면:
  "번호표는 abc-123이에요. 상태: 실행중"
```

## 두 겹의 방어막

이 설계에는 **두 겹의 검사**가 있어요.
막는 **때**가 달라서 둘 다 필요해요.

**1겹 - 서버에서 미리 막기:**
방을 만들기 **전에** 서버가 비율 합이 1.0을 넘으면 미리 거절해요.
(7장에서 배운 입장 조절이에요.)

**2겹 - 방 안에서 막기:**
방 안에서 프로그램이 실제로 메모리를 달라고 할 **때**,
`libfgpu.so`가 quota를 넘으면 그 자리에서 거절해요.
(4장에서 배운 DENY예요.)

```
  첫 번째 방어막:        두 번째 방어막:
  서버 접수 창구         방 안의 libfgpu.so

  "방 만들기 전에         "방 안에서 실제로
   비율 합이 너무 커!"     메모리 달라고 할 때
         |                 내 몫을 넘어!"
         v                       |
       거절                      v
    (방 자체를             거절 (DENY)
     안 만들어요)
```

이렇게 두 겹으로 막아서 더 안전해요.

---

## 📦 한 줄 정리

사람 -> 웹 화면 -> 서버 -> Docker -> 컨테이너(libfgpu.so 포함) -> GPU.
이 흐름이 **GpuCluster의 전체 설계도**예요.

## ❓ 스스로 물어보기

**Q1. "방 만들어 줘!"라고 시키는 부품은 누구예요?**
👉 백엔드 서버예요. 신청을 받으면 Docker에게 방을 만들라고 시켜요.

**Q2. quota 검사는 어디서 해요?**
👉 방 안의 `libfgpu.so`가 해요. 프로그램이 GPU에 주문할 때 가로채서 검사해요.

**Q3. 두 겹 방어막이 뭐예요?**
👉 서버에서 미리 막기(비율 합 검사) + 방 안에서 막기(libfgpu.so quota 검사)예요. 막는 때가 달라요.
