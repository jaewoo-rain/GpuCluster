- 작성일: 2026-06-11
- 상태: **설계 전용** (코드/스크립트 미작성). 마일스톤 단위로 진행.
- 타깃 하드웨어: **단일 NVIDIA RTX 4070 (12282 MiB)** — 12GB 기준으로 모든 예산 계산.
- v1 대비 3대 변경: (1) 실행 매체를 batch 컨테이너+셸 자동화 중심 → **jupyter 세션 2개 중심**으로 전환. (2) 워크로드를 test_compute.py 프록시 → **실제 소형 LLM 추론(Qwen2-0.5B bf16)**으로 대체. (3) 측정을 docker logs 파싱 → **노트북 내 파이썬(time, pynvml, torch.cuda.memory_allocated)**으로 내재화.

## 1. 연구 질문 & 가설

### 핵심 연구 질문

> **"GPU 하나를 fractional로 나눠 두 LLM 추론 워크로드를 동시에 돌리는 것이, 순차로 독점 실행하는 것보다 언제·얼마나 이득인가?"**

### 이득의 조작적 정의

| 지표 | 정의 | 판정 기준 |
|---|---|---|
| **speedup** | `makespan_seq / makespan_conc` (makespan은 epoch 타임스탬프 기준, 섹션 5 참고) | **≥ 1.2 이면 이득** |
| **E_share** | `(wall_solo_A + wall_solo_B) / makespan_conc` | >1 이득, <1 손해 |
| **contention_penalty** | `1 − (tokens_conc_A + tokens_conc_B) / (tokens_solo_A + tokens_solo_B)` | 낮을수록 경합 적음 |

- makespan_seq = solo_A.wall_s + solo_B.wall_s (두 단독 실행 합산)
- makespan_conc = max(t_end_A, t_end_B) − min(t_start_A, t_start_B) (epoch 기준 동시 완료)
- speedup과 tokens/sec는 서로 다른 이야기를 할 수 있다(SM 경합이 심하면 tokens/sec가 낮아도 makespan speedup은 1.x 가능). **둘 다 보고**한다.

### 가설 (실제 LLM 맥락으로 재서술)

- **H1 (컴퓨트 바운드 두 개 동시 — PAIR-CC)**: 짧은 프롬프트 대량 배치(batch=8~16, max_new_tokens=50~100)는 SM 상시 포화. 두 세션 동시 공유 시 tokens/sec가 단독 대비 ~0.4~0.5로 떨어지고, makespan speedup은 1.0 근처(이득 없음 또는 음수). → **"공유가 손해/무의미한 지점"**. solo gpu_util_avg_pct > 80% 이면 컴퓨트 바운드 분류.
- **H2 (메모리/idle-heavy + 컴퓨트 바운드 — PAIR-MC)**: 긴 컨텍스트(batch=1, max_new_tokens=512~1024, prompt_len=512) 세션은 KV cache 크지만 SM 간헐 사용. 컴퓨트 바운드 세션이 빈틈을 메워 speedup > 1.2. solo gpu_util_avg_pct < 50% 이면 메모리/idle 바운드 분류. → **"공유가 이득인 지점"**.
- **H3 (oversubscription, 합 > 1.0)**: B 세션 POST 시 HTTP 409(AdmissionDenied) 발생 또는 force=true 우회 후 hook/driver OOM(torch.cuda.OutOfMemoryError). → **"공유가 명백히 손해인 지점 + admission의 보호 가치"**. jupyter 세션에서 컨테이너 spawn이 막히는 것(batch와 다름) 관측.
- **H4 (3-layer 직교성)**: admission 합=1.0(통과) 이라도 CUDA context 오버헤드(~700 MiB × 2세션 = ~1.4 GB)가 quota 밖이므로, 모델 가중치+KV cache+context 합이 12 GB를 초과하면 물리 OOM 가능. → **"admission 통과 ≠ 물리 안전"**. admission.py의 `_TOL=1e-9` 허용 오차로 합=1.0 경계 정확 통과도 검증.

## 2. 메트릭 정의 (노트북 내 파이썬 측정으로 갱신)

### per-session CSV 스키마

각 추론 노트북(A, B 각각)이 `/workspace/session_result.csv`에 자기 기록 저장. 컬럼:

```
scenario, job, ratio, t_start_epoch, t_end_epoch, wall_s,
prompt_tokens, generated_tokens, tokens_per_sec,
peak_mem_alloc_mib, peak_mem_reserved_mib,
gpu_util_avg_pct, gpu_util_samples,
iter_count, notes
```

| 컬럼 | 정의 | 출처 |
|---|---|---|
| `t_start_epoch` / `t_end_epoch` | `time.time()` (공통 epoch 시계, 배리어 직후 ~ generate 완료) | 노트북 파이썬 |
| `wall_s` | `t_end_epoch − t_start_epoch` | 파생 |
| `prompt_tokens` | `input_ids.shape[-1]` | transformers tokenizer |
| `generated_tokens` | `output.shape[-1] − input_ids.shape[-1]` | transformers |
| `tokens_per_sec` | `generated_tokens / wall_s` (분모 0 보호) | 파생 |
| `peak_mem_alloc_mib` | `torch.cuda.max_memory_allocated() / 1024**2` | torch.cuda |
| `peak_mem_reserved_mib` | `torch.cuda.max_memory_reserved() / 1024**2` | torch.cuda |
| `gpu_util_avg_pct` | pynvml daemon 스레드 0.5s 폴링 평균, N samples | pynvml (보조) |
| `gpu_util_samples` | 샘플 수 | pynvml 스레드 |
| `iter_count` | generate 호출 횟수(반복 실험 합산 가능) | 노트북 |

**per-process 분해 한계 명시**: 컨테이너 안 pynvml의 `nvmlDeviceGetComputeRunningProcesses`는 호스트 PID namespace PID를 반환한다(컨테이너 내부 PID 아님). `/proc/self/status`의 NSpid 매핑은 취약하다. 따라서 **주 메모리 지표는 `torch.cuda.max_memory_allocated()`** 사용(PID namespace 문제 없음, fGPU hook `g_used`와 직접 비교 가능). pynvml GPU 전체 util은 두 컨테이너 합산의 보조 proxy로만 사용.

### 분석 노트북이 합치는 runs.csv 스키마

두 세션 결과를 호스트에서 glob(`data/sessions/*/session_result.csv`)으로 수집, 분석 노트북이 합산:

```
scenario, job_a, job_b, ratio_a, ratio_b, throttle,
t_start_a, t_start_b, t_end_a, t_end_b,
makespan_conc_s, makespan_seq_s,
tokens_per_sec_a, tokens_per_sec_b,
tokens_per_sec_solo_a, tokens_per_sec_solo_b,
peak_mem_alloc_mib_a, peak_mem_alloc_mib_b,
gpu_util_avg_pct_conc,
speedup, e_share, contention_penalty, notes
```

### 이득 판정 기준

- **PASS(이득) = speedup ≥ 1.2** (보수적, 이론 최대 2.0에서 spawn 오버헤드·SM 경합 감속 갭 고려).
- **FAIL(speedup < 1.0)은 나쁜 결과가 아니라 H1을 입증하는 반례 데이터** — SM 경합이 공유 이득을 상쇄.
- 두 speedup 정의 모두 보고: `t_start/t_end` epoch 기준(headline) + 별도 주석으로 solo wall_s 합산.

## 3. 시나리오 매트릭스 ("A POST → B POST" jupyter 흐름)

### 전제: jupyter 세션의 quota 점유 특성

**jupyter 세션은 batch와 달리 사용자가 명시적으로 stop/delete하기 전까지 `status=running`을 유지**한다. admission.py의 `sum_used_ratio()`는 `status in {"created", "running"}`인 세션만 합산하므로, jupyter 세션은 종료되지 않는 한 지속적으로 ratio를 점유한다.

따라서:
- **solo 측정**: 각 세션 단독 실행 후 반드시 `DELETE /sessions/<id>` (purge_workspace=false로 노트북 보존)하고 나서 다음 solo 시작. VRAM 해제를 확인한 뒤 측정.
- **순차(seq_AB)**: A 노트북 generate 셀 완료 → 운영자가 A 세션 명시 stop/delete → 그 이후 B POST. 자동화보다 **수동 확인 필수(jupyter 특성)**. 이를 섹션 맨 앞에 전제로 명시.
- **동시(conc)**: A POST(통과) → B POST(admission이 A의 ratio 실측 후 합산, 합 ≤ 1.0이면 force 없이 201) → 두 노트북에서 배리어 동기화 후 동시 generate.

### 시나리오 표

| # | 이름 | ratio A | ratio B | admission | 워크로드 페어 |
|---|---|---|---|---|---|
| 1 | `solo_A` | RATIO_A | — | 통과 | PAIR-CC or PAIR-MC |
| 2 | `solo_B` | — | RATIO_B | 통과 | PAIR-CC or PAIR-MC |
| 3 | `seq_AB` | RATIO_A | RATIO_B | 통과 (A delete 후 B) | 동일 |
| 4 | `conc_50_50` | 0.5 | 0.5 | **합=1.0 통과** | PAIR-CC |
| 5 | `conc_40_60` | 0.4 | 0.6 | **합=1.0 통과** | PAIR-MC |
| 6 | `conc_70_30` | 0.7 | 0.3 | **합=1.0 경계 통과 (`_TOL=1e-9`)** | PAIR-CC or MC |
| 7 | `conc_oversub` | 0.7 | 0.6 (합=1.3) | **B POST → HTTP 409** | — |
| 8 | `conc_oversub_force` | 0.7 | 0.6 (합=1.3) | force=true 우회 | OOM 시연 |

### admission 관계 및 동작 흐름 상세

**통과 셀 (4, 5, 6) 공통 흐름**:
1. A 세션 `POST /sessions {"ratio": 0.5, "mode": "jupyter"}` → 201, `GET /sessions/admission` 응답: `{"by_gpu": {"all": {"ratio_used": 0.5, "ratio_available": 0.5, ...}}}`.
2. B 세션 `POST /sessions {"ratio": 0.5, "mode": "jupyter"}` → 201 (SessionManager의 `_create_lock`으로 원자적 check-then-insert, 합=1.0 통과).
3. 두 노트북에서 배리어 동기화(섹션 5 참고) 후 동시 generate 진입.
4. 노트북 generate 완료 → 각 `/workspace/session_result.csv` 기록.
5. 실험 종료 후 A, B 세션 각각 DELETE.

**합=1.0 조합 3개 (0.5/0.5, 0.4/0.6, 0.7/0.3)는 전부 force 없이 통과** — admission이 아무것도 막지 않는데 성능은 조합마다 다름. 이는 **admission이 성능과 직교함**을 보여주는 증거다. `GET /sessions/admission`이 `ratio_used: 1.0`을 반환하는 상태에서 노트북이 generate 중인 것을 확인.

**0.7/0.3 경계 케이스**: admission.py의 `if used + requested_ratio > 1.0 + _TOL` 조건에 의해 `0.7 + 0.3 = 1.0 ≤ 1.0 + 1e-9` → 통과. 이 한 줄을 별도 보고.

**oversub 셀 (7) 흐름 — jupyter에서 컨테이너 spawn 차단**:
1. A 세션 POST(0.7, jupyter) → 201.
2. B 세션 POST(0.6, jupyter, force=false) → **HTTP 409**, body:
   ```json
   {"error": "admission_denied", "requested_ratio": 0.6,
    "currently_used": 0.7, "available": 0.3, "gpu_index": null, "active_sessions": 1}
   ```
   batch 컨테이너와 달리 jupyter 컨테이너 자체가 spawn 안 됨(spawn 이전 차단). 이 응답을 `oversub_denied.json`으로 박제.
3. A 세션 DELETE 후 재시도하면 통과 — admission이 동적으로 해제됨을 확인.

**oversub_force 셀 (8) 흐름 — hook DENY/driver OOM을 노트북에서 관측**:
1. A 세션 POST(0.7, jupyter) → 201.
2. B 세션 POST(0.6, jupyter, force=true) → 201 (오버서브스크립션 명시적 허용). `GET /sessions/admission`: `ratio_used: 1.3` (oversubscribed 표시).
3. B 노트북에서 대규모 할당 시도 → `try/except torch.cuda.OutOfMemoryError`로 OOM 기록.
4. `docker logs fgpu-<sid_b>` 로 `[fgpu] DENY` stderr 라인 수집. 타임스탬프와 PID로 어느 layer가 거부했는지 판별 (hook quota DENY vs driver OOM).

**admission 직교성 정량화**: `GET /sessions/admission`의 `ratio_used`(예약, 예 0.5)와 노트북의 `torch.cuda.max_memory_allocated() / quota_bytes`(실사용률, 예 0.30)를 함께 측정 → "예약 100%인데 실사용 30%" 한 줄 보고. quota_bytes는 `FGPU_RATIO × GPU total memory`.

### OOM 시연 셀 (H4 전용)

합=1.0(예 0.5/0.5, 각 quota 6144 MiB)인데 `BATCH`, `SEQ_LEN`, `MAX_NEW_TOKENS`를 노트북 상단 변수로 높여 KV cache를 키우면, 모델 가중치(~1 GB) + KV cache(Qwen2-0.5B KV ≈ 24 KiB/token, batch=32 seq=8192 → ~6 GB) + CUDA context(~700 MiB × 2) 합이 12 GB 초과 → "admission 통과인데 물리 OOM" 관측.

## 4. 워크로드 설계 (실제 LLM 추론)

### 4.1 모델 선택: Qwen2-0.5B bf16

- 가중치 크기: bf16 기준 ~1 GB. 12 GB에서 두 모델 동시 적재 예산: 가중치 1 GB × 2 + CUDA context ~700 MiB × 2 + KV cache ≈ 합 ~4 GB (짧은 KV 기준). quota 6 GB(합)의 약 2/3 → "admission 직교성(예약 100%, 실사용 ~33%)" 그림에 적합.
- **GPT-2/DistilGPT-2는 너무 작아 실험 드라마 없음. TinyLlama-1.1B(~2.2 GB bf16)는 대안이나 12 GB 예산 더 빡빡.** 권장: Qwen2-0.5B.
- 모델 아키텍처: 24 layers, KV head 8개, hidden 896 → `cuBLAS matmul + attention + element-wise`로 다수의 `cudaLaunchKernel` 호출 발생(Stage 7 hook 카운터 자연 활성화).

### 4.2 컴퓨트 바운드 vs 메모리/idle 바운드 두 프로파일

| 프로파일 | batch_size | max_new_tokens | prompt_len | do_sample | 예상 특성 |
|---|---|---|---|---|---|
| **PAIR-CC** (컴퓨트 바운드, H1용) | 8~16 | 50~100 | 32 | True | SM 상시 포화, KV cache 작음, gpu_util>80% |
| **PAIR-MC** (메모리/idle 바운드, H2용) | 1 | 512~1024 | 512 | False | KV cache 큼, SM 간헐, gpu_util<50% |

노트북 상단 변수로 분리해 시나리오별 override:
```python
# 노트북 상단 — 실험 전 변경
SCENARIO  = "conc_40_60"     # 식별자
PROFILE   = "PAIR-MC"        # "PAIR-CC" | "PAIR-MC"
RATIO     = 0.4              # 이 세션의 FGPU_RATIO (세션 spawn 시 주입됨)
BATCH     = 1                # PAIR-MC 기본
MAX_NEW   = 512
PROMPT_LEN = 512
REPEAT    = 1                # 반복 횟수
```

### 4.3 HF 캐시 마운트 (baking 금지)

- **호스트 경로**: `<repo>/data/hf_cache/` (gitignore 추가).
- **컨테이너 마운트**: `-v <repo>/data/hf_cache:/root/.cache/huggingface:ro` + `HF_HOME=/root/.cache/huggingface` 환경변수.
- **이미지 baking 금지** — 모델 가중치(~1 GB)를 이미지에 굽지 않는다. 첫 실행 race 회피: 실험 전 별도 `docker run --rm ... python3 -c "from transformers import AutoModel; AutoModel.from_pretrained('Qwen/Qwen2-0.5B')"` 로 사전 다운로드.
- **구현 위치**: `docker_manager.py`의 `create_container()`에서 jupyter 모드 마운트(현재 workspace만 마운트) 옆에 HF 캐시 마운트 추가 필요 → docker-runtime-engineer에게 위임. `config.py`에 `FGPU_HF_CACHE_DIR` 환경변수 추가 필요.

### 4.4 PYTORCH_NO_CUDA_MEMORY_CACHING=1 의 영향과 한계

- 현재 `runtime-image-pytorch/Dockerfile`에 `ENV PYTORCH_NO_CUDA_MEMORY_CACHING=1` 기본값 고정.
- caching off이면 KV cache의 매 attention 스텝이 cudaMalloc/Free → generate() **5~30x 속도 저하** (실배포 조건과 다름).
- **이것은 측정 왜곡이 아니라 다른 동작 조건**이다. 모든 런(solo/conc)이 동일 조건이면 speedup/contention의 상대 비교는 유효 — speedup을 headline 지표로 쓰는 이유.
- 논문에 "caching off 조건에서의 측정임"을 명시적 한계로 기술.
- caching on 별도 셀로 절대 tokens/sec 주석 추가(hook 효과 없어 참고 목적만).

### 4.5 안전 ALLOC 예산 (12 GB, quota=ratio×12282 MiB)

| 조합 | quota A (MiB) | quota B (MiB) | 예산 합 (MiB) | CUDA context 2개 (~1400) 포함 |
|---|---|---|---|---|
| 0.5/0.5 | 6141 | 6141 | 12282 | 물리 초과 위험 (H4 시연용) |
| 0.4/0.6 | 4913 | 7369 | 12282 | 동일 |
| 0.7/0.3 | 8597 | 3685 | 12282 | 동일 |

**안전 측정용 권장**: 두 노트북의 실제 할당(가중치+KV cache) 합이 **9000 MiB 이하**가 되도록 BATCH/SEQ_LEN 조정. H4 OOM 셀에서만 의도적으로 초과.

### 4.6 메모리 ratio vs 컴퓨트 ratio 독립 유지

- `FGPU_RATIO`(메모리 quota, 현재 `config.py` → `docker_manager.py`에서 `FGPU_RATIO` env로 주입): admission 합산 대상.
- `FGPU_COMPUTE_RATIO`(duty-cycle throttle, Stage 12, `SessionCreate.compute_ratio` 필드): `FGPU_THROTTLE_ENABLE=1`과 함께 주입, admission 합산 대상 아님.
- **메인 실험**: mem ratio만 조정, throttle OFF(경합 순수 측정 주).
- **throttle ON 보조 셀**: 동일 ratio에 `compute_ratio=0.5/0.5` 추가 → 각 tokens/sec가 solo 대비 ~0.5배 수렴하는지(공정성 검증).

## 5. 측정·자동화 설계 (노트북 내 파이썬 + 동기화)

### 5.1 노트북 측정 도구

```python
import time
import threading
import pynvml  # nvidia-ml-py 패키지, Dockerfile에 추가 필요

# 1) 공통 시계: time.time() (epoch) → t_start_epoch, t_end_epoch
# 2) pynvml GPU 전체 util 폴링 — daemon 스레드 (generate가 GIL 잡음)
stop_event = threading.Event()
util_samples = []

def _poll_gpu(interval=0.5):
    pynvml.nvmlInit()
    handle = pynvml.nvmlDeviceGetHandleByIndex(0)
    while not stop_event.is_set():
        util = pynvml.nvmlDeviceGetUtilizationRates(handle).gpu
        util_samples.append(util)
        time.sleep(interval)

poll_thread = threading.Thread(target=_poll_gpu, daemon=True)
poll_thread.start()

# ... generate 실행 ...

stop_event.set()
poll_thread.join(timeout=2)
gpu_util_avg = sum(util_samples) / len(util_samples) if util_samples else 0

# 3) torch.cuda 메모리 지표 (주 메모리 지표, PID namespace 문제 없음)
torch.cuda.reset_peak_memory_stats()
# ... generate ...
peak_alloc = torch.cuda.max_memory_allocated() / 1024**2   # MiB
peak_reserved = torch.cuda.max_memory_reserved() / 1024**2  # MiB
```

**주의**: pynvml 0.5s 해상도는 짧은 generate(수 초)에 샘플 수 부족. 따라서 gpu_util_avg_pct는 보조 지표로만 사용하고, 주 메모리 지표는 torch.cuda 사용.

### 5.2 동시 시작 배리어 — 핵심

Stage 10 워크스페이스 bind-mount는 세션별 분리 경로(`data/sessions/<id>/`)라 두 노트북이 직접 공유 파일을 쓸 수 없다. 두 가지 방법:

**방법 A (권장): 공유 배리어 디렉토리 추가 마운트**

- 호스트 `<repo>/data/barrier/` 를 두 컨테이너 모두 `/barrier` 로 추가 마운트.
- `docker_manager.py`에 `barrier_host_dir` 파라미터 추가 필요 (docker-runtime-engineer에게 위임).
- `config.py`에 `FGPU_BARRIER_DIR` 환경변수 추가.
- 노트북 배리어 의사코드:

```python
import os, time, pathlib

SID = os.environ.get("FGPU_SESSION_ID", "unknown")  # 세션 생성 시 env 주입 필요
BARRIER_DIR = pathlib.Path("/barrier")
READY_FLAG = BARRIER_DIR / f"ready_{SID}.flag"
OTHER_FLAGS = ...  # 미리 알려진 상대 SID 목록 또는 COUNT 기반

# 모델 로드 완료 직후 배리어 진입
READY_FLAG.touch()
deadline = time.time() + 60  # 60s timeout
while time.time() < deadline:
    flags = list(BARRIER_DIR.glob("ready_*.flag"))
    if len(flags) >= 2:  # A, B 모두 ready
        break
    time.sleep(0.05)
else:
    raise TimeoutError("배리어 타임아웃 — 상대 세션이 준비 안 됨")

# 배리어 통과 → 동시에 t_start_epoch 찍고 generate 진입
t_start_epoch = time.time()
```

**방법 B (대안, 배리어 마운트 구현 전 임시)**: 노트북 셀에 `time.sleep(time_until := time.time() // 60 * 60 + 60)` 패턴으로 epoch 1분 경계에 맞춰 동기화. 오차 1~5초 허용("skew < 5s" 명시). 수동 동시 Run 셀 실행(양쪽 노트북에서 동시에 Shift+Enter)도 대안 — 오차 수 초.

## 8. 리스크 · 의도된 한계 (실제 LLM + 노트북 맥락)

| 한계 | 실험 결과 해석에 주는 영향 |
|---|---|
| **SM 격리 없음 (cooperative wall-clock throttle만)** | 컴퓨트×컴퓨트 동시 = SM 자유경쟁. sleep 중에도 다른 세션이 SM을 안 쓰면 GPU idle (work-conserving 아님). |
| **PYTORCH_NO_CUDA_MEMORY_CACHING=1 필수** | KV cache 매 스텝 cudaMalloc/Free → generate() 5~30x 저하. 실배포 조건과 다름. 동일 조건 비교라 speedup 상대값은 유효하나 절대 tokens/sec는 실배포 대비 과소평가. |
| **단일 RTX 4070 12 GB, MIG 없음** | 두 LLM + CUDA context ~1.4 GB. multi-GPU aggregation 범위 밖. |
| **CUDA context 오버헤드 미회계** | admission 합=1.0 이라도 물리 OOM 가능 (H4). context ~700 MiB/프로세스는 quota 밖. |
| **per-process 분해 한계** | pynvml PID namespace 불일치 → torch.cuda.memory_allocated() 주력, pynvml 전체 util은 두 세션 합산 보조. 개별 세션 util 분리 불가. |
| **배리어 동기화 skew** | 방법 B(수동/시각 동기화) 사용 시 최대 수 초 skew → makespan_conc 계산에 오차. 방법 A(공유 마운트 배리어) 사용 시 <100ms. |
| **nanosleep 정밀도 (throttle ON 시)** | Linux 최소 ~50μs → 1ms 미만 window에서 throttle 비율 부정확. idle+throttle ON 조합에서 ±0.05 ratio 오차. |
| **cooperative threat model** | 정적 링크 바이너리는 hook 우회 → quota 미적용. transformers/PyTorch는 동적 링크이므로 hook 적용. |
| **idle 세션 VRAM 미회수** | jupyter 세션이 generate 완료 후에도 모델이 VRAM 점유. 시나리오 간 반드시 DELETE로 해제. |
| **수동 노트북 운영의 공정성 한계** | "동시에 Shift+Enter"는 수 초 skew 가능. 자동화는 배리어 마운트로만 해소. |
| **launch count ≠ device time** | cudaLaunchKernel 카운터가 있어도 무거운 커널 1회 vs 가벼운 1000회 구분 불가. tokens/sec를 device-time proxy로 사용. |

**"공유가 이득" 성립 조건**: 적어도 한쪽이 SM을 간헐적으로만 쓰는(메모리 바운드/idle-heavy, PAIR-MC) 워크로드일 때. **불성립 조건**: 둘 다 SM 풀로드 컴퓨트 바운드(PAIR-CC) — 이땐 순차 독점이 동등하거나 낫다. 이 정직한 경계 자체가 핵심 발견.
