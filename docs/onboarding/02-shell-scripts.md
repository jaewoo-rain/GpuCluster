# 2장. 쉘 스크립트와 터미널 — .sh 파일이 도대체 뭔가요

> 📘 **이 장을 읽고 나면**
>
> - 터미널과 쉘(bash)이 뭔지, "명령어로 컴퓨터와 대화한다"는 게 무슨 뜻인지 알게 됩니다.
> - `.sh` 파일이 "명령어를 순서대로 적어둔 대본"이라는 걸 이해하고, 왜 쓰는지 알게 됩니다.
> - `#!/usr/bin/env bash`, `$VAR`, `"${VAR:-기본값}"`, `if`, `for`, `|`, `&&`, `export`, `set -euo pipefail` 같은 기본 문법을 읽을 수 있게 됩니다.
> - `./scripts/build_hook.sh`를 한 줄씩 뜯어보며 `gcc` 명령이 뭘 하는지 이해하게 됩니다.
> - `FGPU_RATIO=0.4 ./...` 처럼 환경변수로 설정을 주입하는 이 프로젝트의 핵심 관례를 알게 됩니다.

---

1장에서 `gcc`로 `.c`를 `.so`로 컴파일한다고 배웠죠. 그런데 그 `gcc` 명령은 옵션이 잔뜩 붙어서 매번 손으로 치기 힘듭니다. 이 프로젝트는 그런 명령들을 `scripts/` 폴더의 `.sh` 파일에 미리 적어두고 실행합니다. 이 챕터는 터미널이 처음인 분을 위해 **완전 바닥부터** 시작합니다.

---

## 1. 터미널과 쉘 — 명령어로 컴퓨터와 대화하기

### 왜 필요한가

우리가 평소 컴퓨터를 쓸 때는 마우스로 아이콘을 클릭하죠(GUI). 그런데 개발자들, 특히 서버를 다루는 사람들은 **글자로 명령을 입력해서** 컴퓨터를 조작합니다. 왜냐하면 서버(예: 이 프로젝트의 Ubuntu GPU 서버)에는 화면과 마우스가 없는 경우가 많고, 명령어는 **똑같이 반복하거나 자동화하기 쉽기** 때문입니다.

### 일상 비유

- **GUI(마우스)** = 식당에서 메뉴판 사진을 손가락으로 가리키며 주문.
- **터미널(명령어)** = 종업원에게 말로 "김치찌개 하나, 밥 곱빼기로"라고 주문.

말로 하면 처음엔 어색해도, 똑같은 주문을 매번 정확히 반복하거나 종이에 적어 남에게 시키기 훨씬 편하죠.

### 용어 정리

- **터미널(terminal)**: 명령어를 입력하는 **검은 창** 자체. (대화창)
- **쉘(shell)**: 그 명령어를 **해석하고 실행하는 프로그램**. (통역사)
- **bash**: 가장 널리 쓰이는 쉘의 한 종류. 리눅스의 기본 통역사라고 보면 됩니다.

즉, "터미널이라는 창"에 명령을 치면 "bash라는 통역사"가 그걸 알아듣고 컴퓨터에게 시킵니다.

### 아주 작은 예시

터미널에 이렇게 치고 엔터를 누르면:

```bash
ls
```

`ls`는 "list"의 약자로, **현재 폴더의 파일 목록을 보여줘**라는 명령입니다. 몇 가지 더:

```bash
pwd          # 지금 내가 어느 폴더에 있는지 보여줘 (print working directory)
cd scripts   # scripts 폴더로 이동해라 (change directory)
cat 파일명    # 파일 내용을 화면에 출력해라
```

### 한 줄 요약

> 터미널은 명령을 입력하는 창, 쉘(bash)은 그 명령을 해석·실행하는 통역사이며, 개발과 서버 작업의 기본 도구입니다.

## 2. .sh 파일 — 명령어를 순서대로 적어둔 대본

### 왜 필요한가

이 프로젝트에서 `libfgpu.so`를 만들려면 다음 같은 일을 순서대로 해야 합니다: "빌드 폴더 만들기 → gcc로 컴파일 → 결과 파일 확인". 매번 이 세 명령을 손으로 치는 건 귀찮고, 실수하기 쉽죠. 그래서 **명령들을 파일 하나에 순서대로 적어두고, 그 파일 하나만 실행**하면 알아서 다 되게 만듭니다. 그 파일이 `.sh` 파일입니다.

### 일상 비유

`.sh`는 **연극 대본**입니다. "1막: 무대 세팅, 2막: 배우 등장, 3막: 조명 켜기"를 미리 적어두면, 매번 즉흥으로 하지 않고 대본만 읽으면 똑같은 공연이 재현되죠. `.sh` 파일도 컴퓨터가 읽는 대본이에요. 요리 레시피 카드라고 생각해도 좋습니다.

### 왜 쓰나 (핵심 이유)

1. **반복 자동화**: 여러 명령을 한 번에 실행.
2. **재현성**: 누가 실행해도 똑같은 결과. (팀원 A가 만든 걸 팀원 B가 그대로 재현)
3. **문서 역할**: "이 프로젝트를 어떻게 빌드하는지"가 스크립트에 그대로 기록됩니다.

### 이 프로젝트 실제 코드에서

`scripts/` 폴더를 열어보면 `.sh` 파일이 잔뜩 있습니다:

- [build_hook.sh](../../scripts/build_hook.sh) — `libfgpu.so`를 빌드하는 대본
- [run_test.sh](../../scripts/run_test.sh) — 테스트 프로그램을 빌드하고 후킹을 걸어 실행하는 대본

각각 "손으로 치면 여러 줄인 명령"을 파일 하나로 묶어둔 겁니다.

### 한 줄 요약

> `.sh` 파일은 명령어를 순서대로 적어둔 "대본"으로, 반복 작업을 자동화하고 누가 해도 똑같이 재현되게 해줍니다.

## 3. 기본 문법 — .sh 파일을 읽는 데 필요한 최소한

이제 `.sh` 안에서 자주 보이는 문법들을 하나씩 봅니다. 전부 `build_hook.sh`와 `run_test.sh`에 실제로 나오는 것들이에요.

### 3-1. 셔뱅 `#!/usr/bin/env bash`

파일 맨 첫 줄에 있는 `#!`로 시작하는 이 줄을 **셔뱅(shebang)**이라고 합니다. "이 대본은 **bash라는 통역사로 읽어라**"라고 컴퓨터에게 알려주는 표시예요.

- [build_hook.sh:1](../../scripts/build_hook.sh#L1) — `#!/usr/bin/env bash`

`/usr/bin/env bash`는 "시스템 어딘가에 있는 bash를 알아서 찾아 써라"라는 이식성 좋은 방식입니다. (참고: 일반 줄에서 `#`는 "이 뒤는 주석, 무시해"라는 뜻이지만, 맨 첫 줄의 `#!`만은 특별히 셔뱅으로 취급됩니다.)

### 3-2. 변수 `$VAR`

값을 담아두는 상자입니다. **만들 때는 `이름=값`**(등호 양옆에 공백 없기!), **쓸 때는 `$이름`**으로 꺼냅니다.

```bash
name="철수"          # 만들기 (= 양옆에 띄어쓰기 금지!)
echo "안녕 $name"    # 쓰기 → "안녕 철수" 출력
```

`echo`는 "화면에 글자 출력해라"는 명령입니다(파이썬의 `print`).

- [build_hook.sh:11](../../scripts/build_hook.sh#L11) — `BUILD_DIR="${SRC_DIR}/build"` — 변수를 만들고
- [build_hook.sh:13](../../scripts/build_hook.sh#L13) — `mkdir -p "${BUILD_DIR}"` — `$BUILD_DIR`로 꺼내 씀

> `${VAR}`처럼 중괄호로 감싸는 건 `$VAR`와 같지만, 뒤에 다른 글자가 붙을 때 경계를 명확히 하려는 습관입니다.

### 3-3. 기본값 `"${VAR:-기본값}"`

**"이 변수가 설정돼 있으면 그 값을, 없으면 기본값을 써라"**라는 아주 유용한 문법입니다.

```bash
echo "${GREETING:-안녕하세요}"
# GREETING 이 설정돼 있으면 그 값, 아니면 "안녕하세요" 출력
```

이 프로젝트에서 매우 자주 나옵니다. "사용자가 설정을 주면 그걸 쓰고, 안 주면 안전한 기본값을 쓰는" 패턴이에요:

- [build_hook.sh:9](../../scripts/build_hook.sh#L9) — `CUDA_HOME="${CUDA_HOME:-/usr/local/cuda}"` — CUDA 위치를 안 정했으면 `/usr/local/cuda`를 기본으로
- [run_test.sh:30](../../scripts/run_test.sh#L30) — `FGPU_RATIO=${FGPU_RATIO:-0.4}` — 비율을 안 주면 기본 `0.4`

### 3-4. `if` 조건문

"만약 ~라면 ~해라". 이 프로젝트에서는 주로 "파일이 있는지 확인" 같은 데 씁니다.

```bash
if [[ ! -f "libfgpu.so" ]]; then
    echo "파일이 없어요!"
fi
```

- `[[ ... ]]` = 조건을 검사하는 부분.
- `-f "파일"` = "그 파일이 존재하나?"
- `!` = "아니다"(부정). 그래서 `! -f`는 "파일이 없으면".
- `then` ~ `fi` 사이가 조건이 참일 때 실행되는 부분. (`fi`는 `if`를 거꾸로 쓴 것 = 블록 끝)

**실제 코드에서**:

- [run_test.sh:18-21](../../scripts/run_test.sh#L18) — `if [[ ! -f "${HOOK_SO}" ]]; then ... exit 1; fi` — `.so` 파일이 없으면 "먼저 build_hook.sh를 돌려라"라는 에러를 내고 종료(`exit 1`). `exit 1`은 "실패로 끝냄"(1장에서 배운 에러코드! 0=성공, 0아님=실패).

### 3-5. `for` 반복문

"목록의 각 항목마다 반복해라". 이 프로젝트의 빌드 스크립트엔 많지 않지만 문법은 알아두세요.

```bash
for name in 철수 영희 민수; do
    echo "안녕 $name"
done
# "안녕 철수", "안녕 영희", "안녕 민수" 를 차례로 출력
```

`do` ~ `done` 사이가 반복되는 부분입니다.

### 3-6. 파이프 `|` — 명령을 관으로 잇기

한 명령의 **출력**을 다음 명령의 **입력**으로 흘려보내는 관(pipe)입니다.

```bash
ls | grep test
# ls 결과(파일 목록)를 grep 에 흘려보내 "test" 가 든 줄만 걸러냄
```

물이 관을 타고 흐르듯 데이터가 왼쪽 명령에서 오른쪽 명령으로 흐릅니다.

### 3-7. `&&` — 앞이 성공해야 뒤를 실행

`A && B`는 "A가 **성공하면** B를 실행"입니다. A가 실패하면 B는 건너뜁니다.

```bash
mkdir build && cd build
# build 폴더 만들기에 성공하면 그 안으로 이동. 실패하면 이동 안 함(안전).
```

### 3-8. 환경변수와 `export` / `LD_PRELOAD`

**환경변수**는 "프로그램 실행 환경 전체가 공유하는 변수"입니다. 일반 변수는 그 스크립트 안에서만 살지만, 환경변수는 그 스크립트가 **실행하는 다른 프로그램에게도 전달**됩니다.

- `export VAR=값` → "이 변수를 환경변수로 승격해서 자식 프로그램에게도 물려줘라."
- 또는 명령 **바로 앞에** `VAR=값 명령` 형태로 쓰면 "그 한 번의 실행에만" 환경변수를 붙일 수 있습니다.

**비유**: 일반 변수는 내 책상 위 메모지(나만 봄), 환경변수는 사무실 공지판(내가 부른 부하 직원들도 다 봄).

1장에서 배운 `LD_PRELOAD`가 바로 환경변수입니다! "이 프로그램 실행할 때 우리 `.so`를 먼저 끼워 넣어라"라고 리눅스에게 전달하는 거죠:

- [run_test.sh:31-34](../../scripts/run_test.sh#L31) — 아래처럼 명령 앞에 환경변수 여러 개를 붙여 실행합니다:

```bash
LD_PRELOAD="${HOOK_SO}" \
FGPU_RATIO="${FGPU_RATIO:-0.4}" \
    "${TEST_BIN}"
```

이 세 줄은 사실 **한 줄**입니다(끝의 `\`는 "다음 줄에 이어짐" 표시). 뜻은: "`LD_PRELOAD`와 `FGPU_RATIO` 환경변수를 붙인 채로 테스트 프로그램(`$TEST_BIN`)을 실행해라." 그러면 그 프로그램 안의 CUDA 호출이 우리 `libfgpu.so`에 가로채이고, 후킹 코드는 `getenv("FGPU_RATIO")`로 `0.4`를 읽어 quota를 계산합니다. (1장에서 본 [fgpu_hook.c:319-320](../../hook/src/fgpu_hook.c#L319)이 바로 이 환경변수를 읽는 곳!)

### 초보가 흔히 하는 오해/함정

- **변수 대입에 공백** → `name = "철수"`는 에러입니다. `name="철수"`처럼 **등호 양옆에 공백 없이** 써야 해요.
- **`$` 빼먹기** → 값을 꺼낼 때는 `$name`, 만들 때는 `name`. 헷갈리지 마세요.
- **환경변수와 일반 변수 혼동** → `export` 안 한 변수는 자식 프로그램에게 안 넘어갑니다. `LD_PRELOAD`가 안 먹으면 이걸 의심하세요(명령 앞 접두사 방식은 자동으로 넘어감).

### 한 줄 요약

> `.sh`는 셔뱅으로 시작하고, `$VAR`로 변수를 쓰며, `"${VAR:-기본값}"`으로 기본값을 주고, `if`/`for`로 흐름을 제어하고, 환경변수(`LD_PRELOAD`, `FGPU_RATIO`)로 실행 설정을 주입합니다.

## 4. `set -euo pipefail` — 스크립트의 안전벨트

### 왜 필요한가

기본 상태의 쉘 스크립트는 **중간 명령이 실패해도 그냥 다음 줄로 넘어갑니다.** 빌드 도중 컴파일이 실패했는데도 "성공했다"며 끝나버리면, 뭐가 잘못됐는지 찾기 힘들죠. 이걸 막는 안전장치가 `set -euo pipefail`입니다. 이 프로젝트의 거의 모든 스크립트 맨 앞에 있어요.

### 일상 비유

자동차의 **안전벨트 + 에어백 + 후방 센서**를 한 번에 켜는 스위치입니다. 사고(에러)가 나면 즉시 멈춰서 더 큰 사고를 막아줍니다.

### 각 글자의 뜻

- **`-e`** (error): 명령 하나라도 **실패하면 즉시 스크립트를 멈춰라.** (조용히 다음 줄로 안 넘어감)
- **`-u`** (unset): **정의 안 된 변수를 쓰면 에러를 내라.** (오타로 `$BUILD_DR` 같은 걸 쓰면 잡아줌)
- **`-o pipefail`**: **파이프(`|`) 중간 명령이 실패하면 전체를 실패로 쳐라.** (기본은 맨 끝 명령 결과만 봄)

즉 "실패를 조용히 넘기지 말고, 오타도 잡고, 파이프 중간 실패도 놓치지 마라"라는 종합 안전장치입니다.

### 이 프로젝트 실제 코드에서

- [build_hook.sh:7](../../scripts/build_hook.sh#L7) — `set -euo pipefail`
- [run_test.sh:10](../../scripts/run_test.sh#L10) — `set -euo pipefail`

### 초보가 흔히 하는 오해/함정

- **"에러가 나도 계속 돌겠지"** → `-e`가 켜져 있으면 첫 실패에서 바로 멈춥니다. 그게 의도예요(빨리 실패 = 빨리 발견).
- **`-u` 때문에 스크립트가 죽음** → 변수 이름 오타이거나 미설정 변수를 쓴 겁니다. 오히려 버그를 미리 잡아준 것. 기본값 문법 `"${VAR:-}"`으로 "없어도 됨"을 표시할 수 있어요.

### 한 줄 요약

> `set -euo pipefail`은 "실패 시 즉시 정지 + 미정의 변수 금지 + 파이프 실패 감지"라는 스크립트 안전벨트입니다.

## 5. 스크립트 실행 방법 — chmod +x 와 ./

### 왜 필요한가

`.sh` 파일을 만들었다고 바로 실행되진 않습니다. 리눅스는 보안상 **"이 파일은 실행해도 되는 파일이야"라는 허가(실행 권한)**가 있어야 실행합니다. 그 허가를 주는 명령이 `chmod +x`입니다.

### 일상 비유

새로 산 가전제품의 **"안전 잠금"**을 푸는 것과 같습니다. 잠금을 풀어야(`chmod +x`) 버튼을 눌러 작동(`./스크립트`)시킬 수 있어요.

### 실제 실행 절차

```bash
chmod +x scripts/*.sh        # scripts 폴더의 모든 .sh 에 실행 권한 부여 (한 번만)
./scripts/build_hook.sh      # 실행! (앞의 ./ 는 "지금 이 폴더의" 라는 뜻)
```

- `chmod +x` = "실행 권한(x = executable)을 추가(+)해라."
- `*.sh` = "이름이 `.sh`로 끝나는 모든 파일"(`*`은 아무 글자나 매칭).
- `./` = "현재 폴더에 있는"이라는 명시. (안 붙이면 시스템이 못 찾을 수 있음)

이 프로젝트의 CLAUDE.md 안내도 정확히 이 순서로 시작합니다: `chmod +x scripts/*.sh` 후 `./scripts/build_hook.sh`.

### 한 줄 요약

> `chmod +x`로 실행 권한을 준 뒤 `./scripts/build_hook.sh`처럼 앞에 `./`를 붙여 실행합니다.

## 6. build_hook.sh 완전 해부 — 한 줄씩 초보 눈높이로

이제 실제 스크립트 [build_hook.sh](../../scripts/build_hook.sh) 전체를 처음부터 끝까지 따라가 봅시다. 지금까지 배운 문법이 다 나옵니다.

### 6-1. 준비 부분

```bash
#!/usr/bin/env bash          # (1) bash 통역사로 읽어라 (셔뱅)
set -euo pipefail            # (2) 안전벨트 켜기
```

- [build_hook.sh:1](../../scripts/build_hook.sh#L1) — 셔뱅
- [build_hook.sh:7](../../scripts/build_hook.sh#L7) — 안전장치

```bash
CUDA_HOME="${CUDA_HOME:-/usr/local/cuda}"          # (3)
SRC_DIR="$(cd "$(dirname "$0")/.." && pwd)"        # (4)
BUILD_DIR="${SRC_DIR}/build"                        # (5)
```

- [build_hook.sh:9](../../scripts/build_hook.sh#L9) — **(3)** CUDA가 설치된 위치. 사용자가 안 정했으면 `/usr/local/cuda` 기본값. (기본값 문법!)
- [build_hook.sh:10](../../scripts/build_hook.sh#L10) — **(4)** 이게 조금 어렵습니다. **"이 스크립트 파일이 있는 폴더의 부모 폴더(프로젝트 루트)를 알아내"**는 코드예요.
  - `$0` = 실행된 스크립트 파일 경로.
  - `dirname "$0"` = 그 파일이 든 폴더(`scripts/`).
  - `/..` = 한 단계 위(프로젝트 루트).
  - `cd ... && pwd` = 거기로 이동한 뒤 그 절대 경로를 출력.
  - `$( ... )` = 안쪽 명령의 **출력 결과를 값으로 가져오기**(명령 치환). 그 결과를 `SRC_DIR`에 저장.
  - 결론: 어디서 실행하든 프로젝트 루트를 정확히 찾아내는 관용구입니다. (통째로 외워도 OK)
- [build_hook.sh:11](../../scripts/build_hook.sh#L11) — **(5)** 빌드 결과를 넣을 `build/` 폴더 경로.

```bash
mkdir -p "${BUILD_DIR}"     # (6) build 폴더 만들기 (이미 있으면 그냥 넘어감)
```

- [build_hook.sh:13](../../scripts/build_hook.sh#L13) — **(6)** `mkdir`은 폴더 생성, `-p`는 "이미 있어도 에러 내지 말고, 중간 경로도 필요하면 만들어"라는 옵션.

### 6-2. 핵심 — gcc 컴파일 명령

이제 1장에서 배운 컴파일이 실제로 일어나는 부분입니다:

```bash
gcc -O2 -fPIC -shared -Wall -Wextra \
    -I"${CUDA_HOME}/include" \
    -o "${BUILD_DIR}/libfgpu.so" \
    "${SRC_DIR}/hook/src/fgpu_hook.c" \
    -L"${CUDA_HOME}/lib64" -lcudart -ldl -lpthread
```

- [build_hook.sh:15-19](../../scripts/build_hook.sh#L15) — 이게 프로젝트의 심장을 만드는 명령입니다. 줄 끝의 `\`는 "다음 줄에 이어짐" 표시라, 사실 **한 줄**이에요. 옵션을 하나씩 보면:

| 옵션 | 뜻 |
|------|-----|
| `gcc` | C 컴파일러(번역가) 호출 |
| `-O2` | 최적화 레벨 2로 번역해라 (더 빠른 결과물) |
| `-fPIC` | "위치 독립 코드"로 만들어라 — **`.so` 라이브러리를 만들 때 필수** |
| `-shared` | 실행파일 말고 **공유 라이브러리(`.so`)로** 만들어라 |
| `-Wall -Wextra` | 의심스러운 코드에 대해 경고를 많이 보여줘라 (버그 예방) |
| `-I".../include"` | 헤더(메뉴판, `.h`)를 이 폴더에서 찾아라 (1장의 `#include`가 여기서 해결됨) |
| `-o ".../libfgpu.so"` | 결과물 이름을 `libfgpu.so`로 해라 (`-o` = output) |
| `".../fgpu_hook.c"` | **번역할 원본 소스** 파일 |
| `-L".../lib64"` | 이어붙일 라이브러리를 이 폴더에서 찾아라 |
| `-lcudart` | CUDA 런타임 라이브러리와 링크(이어붙이기) |
| `-ldl` | `dlsym`이 든 라이브러리와 링크 (진짜 CUDA 함수 주소 찾기용) |
| `-lpthread` | 멀티스레드 잠금(`pthread_mutex`)이 든 라이브러리와 링크 |

이 한 줄이 바로 **`fgpu_hook.c` → `libfgpu.so`** 변환을 수행합니다. 1장에서 본 `#include <cuda_runtime_api.h>`가 `-I` 덕분에 찾아지고, `dlsym`([fgpu_hook.c:307](../../hook/src/fgpu_hook.c#L307))이 `-ldl` 덕분에 연결되고, `pthread_mutex`([fgpu_hook.c:127](../../hook/src/fgpu_hook.c#L127))가 `-lpthread` 덕분에 연결됩니다. **스크립트의 옵션과 C 코드가 서로 맞물려 있는 게 보이시죠?**

### 6-3. 마무리 — 확인

```bash
echo "[build] wrote ${BUILD_DIR}/libfgpu.so"    # (7) 완료 메시지 출력
ls -lh "${BUILD_DIR}/libfgpu.so"                # (8) 만들어진 파일 정보 보기
```

- [build_hook.sh:21](../../scripts/build_hook.sh#L21) — **(7)** "다 만들었어요" 안내.
- [build_hook.sh:22](../../scripts/build_hook.sh#L22) — **(8)** `ls -lh`로 결과 파일의 크기·날짜를 사람이 읽기 좋게(`-h` = human-readable) 보여줌.

### 정리

이 스크립트 하나가 하는 일: **"CUDA 위치 파악 → build 폴더 준비 → gcc로 hook 컴파일 → 확인."** 손으로 치면 헷갈릴 긴 명령을, `./scripts/build_hook.sh` 한 번으로 끝내는 겁니다. 이게 `.sh`의 가치예요.

## 7. 환경변수로 설정 주입하기 — 이 프로젝트 전체의 핵심 관례

### 왜 중요한가

이 프로젝트를 관통하는 **가장 중요한 사용 패턴**입니다. 코드를 고치지 않고, **실행할 때 환경변수로 동작을 바꾸는** 방식이에요. 예를 들어 "GPU의 40%만 써라" vs "60%를 써라"를 코드 수정 없이 명령 한 줄로 전환합니다:

```bash
FGPU_RATIO=0.4 ./scripts/run_test.sh      # GPU 40% quota 로 실행
FGPU_RATIO=0.6 ./scripts/run_test.sh      # GPU 60% quota 로 실행
```

명령 **바로 앞**에 `FGPU_RATIO=0.4`를 붙였죠? 이건 "이번 한 번의 실행에만 이 환경변수를 붙여라"라는 뜻입니다(3-8에서 배운 접두사 방식).

### 어떻게 연결되나 (전체 그림)

1. 사용자가 `FGPU_RATIO=0.4 ./scripts/run_test.sh` 실행.
2. 스크립트가 `LD_PRELOAD`와 `FGPU_RATIO`를 붙여 테스트 프로그램을 실행 — [run_test.sh:31-34](../../scripts/run_test.sh#L31).
3. `LD_PRELOAD` 덕분에 우리 `libfgpu.so`가 먼저 로드되어 CUDA 호출을 가로챔.
4. 후킹 코드가 `getenv("FGPU_RATIO")`로 `0.4`를 읽음 — [fgpu_hook.c:319-320](../../hook/src/fgpu_hook.c#L319).
5. GPU 전체 메모리 × 0.4 를 quota로 계산해 그 이상은 거부.

**즉 명령줄의 환경변수 하나가 → 스크립트 → LD_PRELOAD → C 코드의 `getenv`까지 한 줄로 이어집니다.** 이 흐름을 이해하면 프로젝트의 절반을 이해한 겁니다.

이 프로젝트엔 이런 환경변수가 많습니다: `FGPU_RATIO`, `FGPU_QUOTA_BYTES`, `FGPU_LAUNCH_LOG_EVERY`, `FGPU_COMPUTE_RATIO` 등. 전부 "코드 안 고치고 실행 시점에 동작을 조절하는" 손잡이입니다.

### 초보가 흔히 하는 오해/함정

- **`FGPU_RATIO = 0.4` 처럼 공백** → 안 됩니다. `FGPU_RATIO=0.4`로 붙여 쓰세요.
- **환경변수를 명령 뒤에 붙임** → `./run_test.sh FGPU_RATIO=0.4`는 틀립니다. 환경변수는 명령 **앞**에.
- **한 번 export하면 영원할 거라 착각** → 접두사 방식(`VAR=값 명령`)은 그 한 번만 유효합니다. 여러 명령에 계속 쓰려면 `export FGPU_RATIO=0.4`.

### 한 줄 요약

> `FGPU_RATIO=0.4 ./스크립트`처럼 환경변수로 설정을 주입하는 게 이 프로젝트의 핵심 관례이며, 그 값은 결국 C 코드의 `getenv`까지 흘러갑니다.

## 8. 왜 이 .sh 파일들은 지금 내 Windows PC에서 안 돌까요?

지금 개발 PC는 Windows(현재 작업 폴더도 `C:\Users\...`)입니다. 그런데 이 프로젝트의 `.sh` 스크립트들은 전부 **Ubuntu GPU 서버**에서 돌리라고 만들어졌어요. 왜일까요?

이유는 여러 겹입니다. 첫째, `.sh`는 리눅스의 bash 통역사를 위한 대본이라 Windows의 기본 명령창(cmd/PowerShell)은 그대로 이해하지 못합니다. 둘째, 이 스크립트들이 부르는 `gcc`, `nvcc`(CUDA 컴파일러), `/usr/local/cuda` 경로, `LD_PRELOAD` 메커니즘은 **모두 리눅스 + NVIDIA GPU 환경**의 것입니다. 셋째, 핵심 트릭인 `LD_PRELOAD` 심볼 가로채기 자체가 리눅스 동적 링커의 기능이라 Windows에는 없습니다(Windows는 `.dll`과 다른 방식). 그래서 이 프로젝트는 실제 실행·빌드·테스트를 **RTX 4060이 달린 Ubuntu 서버**에서 하고, Windows PC는 주로 코드를 읽고 편집하는 용도로 씁니다. 빌드/실행 안내가 CLAUDE.md에 "on Ubuntu GPU server"라고 명시된 것도 이 때문이에요.

> 한 줄 요약: `.sh` + `gcc`/`nvcc` + `LD_PRELOAD` + NVIDIA GPU는 리눅스 환경 전용이라, 실제 구동은 Ubuntu GPU 서버에서 하고 Windows에서는 코드 열람/편집만 합니다.
